# Deploy Readiness — 本番デプロイ準備チェック

Build Unit の SRE として、本番環境にデプロイする前の準備状況を総合的にチェックしてください。

`/release-check` がビルド・コミット状態の確認であるのに対し、こちらは**本番環境で安全に動くか**を検証します。

## 手順

1. 以下を並列で検証:

### 環境変数・設定
- [ ] `vercel.json` に必要な環境変数がすべて定義されているか
- [ ] ハードコードされた URL やシークレットがコード内にないか
- [ ] `process.env` で参照している変数が `vercel.json` に存在するか
- [ ] 開発用の設定が本番に混入していないか（`localhost`, `console.log` の乱用等）

### Supabase / DB
- [ ] 未適用のマイグレーションファイルがあるか（`supabase/migrations/`）
- [ ] RLS ポリシーが設定されているテーブルに漏れがないか
- [ ] API ルートで `supabase` クライアントの使い分けが正しいか（`createClient` vs `createServerClient`）
- [ ] セキュリティ上重要なテーブル（users, profiles, messages）に RLS があるか

### エラーハンドリング
- [ ] API ルート (`app/api/`) に try-catch があるか
- [ ] エラーレスポンスが適切な HTTP ステータスを返すか
- [ ] ユーザー向けエラーメッセージが日本語で適切か
- [ ] 予期しないエラーで内部情報（スタックトレース等）が漏れないか

### パフォーマンス
- [ ] `'use client'` の範囲が最小限か（不要な Client Component がないか）
- [ ] 大量データの取得に `limit` / ページネーションがあるか
- [ ] 画像に `next/image` を使っているか
- [ ] 重い処理がクライアントサイドに載っていないか

### セキュリティ
- [ ] ユーザー入力のバリデーションがあるか（API ルート）
- [ ] SQL インジェクション・XSS のリスクがないか
- [ ] 認証チェックが必要なルートに認証があるか
- [ ] CORS 設定が適切か

2. 以下のフォーマットで報告

## 出力フォーマット

```
## デプロイ準備チェック [日付]

### 総合判定: :green_circle: READY / :yellow_circle: 条件付き READY / :red_circle: NOT READY

### チェック結果
| カテゴリ | 結果 | 重要な発見 |
|----------|------|-----------|
| 環境変数 | :white_check_mark: / :warning: / :x: | ... |
| DB / RLS | :white_check_mark: / :warning: / :x: | ... |
| エラー処理 | :white_check_mark: / :warning: / :x: | ... |
| パフォーマンス | :white_check_mark: / :warning: / :x: | ... |
| セキュリティ | :white_check_mark: / :warning: / :x: | ... |

### 要修正（デプロイ前に必須）
| 優先度 | カテゴリ | 箇所 | 問題 | 修正方法 |
|--------|----------|------|------|---------|
| P0 | セキュリティ | [file:line] | ... | ... |

### 要確認（CEO承認が必要）
- [ ] 未適用マイグレーション: [ファイル名]
- [ ] 新規環境変数: [変数名]
- [ ] セキュリティ上の判断: [内容]

### 推奨改善（デプロイ後でも可）
| カテゴリ | 箇所 | 改善案 |
|----------|------|--------|
| パフォーマンス | [file] | ... |
```

## 注意
- P0（セキュリティ・データ漏洩リスク）は NOT READY 判定
- 報告のみ。修正の実装は CEO 承認後
- `/release-check` → `/deploy-readiness` → CEO 承認 → デプロイ の順で実行

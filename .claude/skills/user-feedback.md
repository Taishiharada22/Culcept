# User Feedback — フィードバック記録・分析

Ops Unit の CX Analyst として、テストユーザーからのフィードバックを構造化し、アクションに変換してください。

## 使い方

引数にフィードバック内容を渡す:
- `/user-feedback "Stargazerの質問が多すぎて途中でやめた"` — 単一フィードバック
- `/user-feedback` — 引数なしの場合、直近のフィードバックログを分析

## 手順

### フィードバック記録（引数あり）

1. フィードバックを以下の構造に分解:

```yaml
日付: [今日の日付]
ユーザー: [わかれば。匿名でもOK]
種別: バグ / UX問題 / 機能要望 / 肯定的 / 質問
対象機能: [Stargazer / Orbiter / Rendezvous / Home / Genome Card / その他]
原文: "[フィードバックそのまま]"
要約: [1行に要約]
感情: ポジティブ / ニュートラル / ネガティブ / フラストレーション
深刻度: P0(使えない) / P1(体験を損なう) / P2(改善の余地) / P3(Nice to have)
```

2. 既存のフィードバックログ `docs/user-feedback-log.md` を確認し、同様のフィードバックがないか照合
3. ログに追記

### フィードバック分析（引数なし or `analyze`）

1. `docs/user-feedback-log.md` を読み込む
2. 以下の観点で分析:

**パターン検出**
- 同じ問題が複数ユーザーから報告されているか
- 特定の機能にフィードバックが集中していないか
- 離脱ポイントの共通点はないか

**感情分析**
- ポジティブ / ネガティブの比率
- フラストレーションが高い箇所

**今月の成功条件との照合**
- コア機能の完成度に関するフィードバック
- 世界観に関するフィードバック
- 初回体験に関するフィードバック

3. 以下のフォーマットで報告

## 出力フォーマット（記録時）

```
## フィードバック記録 [日付]

### 記録内容
- 種別: [バグ / UX問題 / ...]
- 対象: [機能名]
- 深刻度: [P0-P3]
- 要約: [1行]

### 類似フィードバック
- [過去に同様の報告があれば表示。なければ「初出」]

### 推奨アクション
- [具体的な対応案]
- 担当部門: [Product / Build / Growth / Ops]

:white_check_mark: `docs/user-feedback-log.md` に追記しました
```

## 出力フォーマット（分析時）

```
## フィードバック分析 [日付]

### サマリー
- 総件数: [N]件（期間: [開始日] 〜 [終了日]）
- 感情分布: :smile: [N] / :neutral_face: [N] / :disappointed: [N] / :rage: [N]

### 機能別フィードバック数
| 機能 | 件数 | 主な傾向 |
|------|------|---------|
| Stargazer | [N] | ... |
| Home | [N] | ... |

### 頻出パターン（Top 5）
| # | パターン | 件数 | 深刻度 | 推奨アクション |
|---|---------|------|--------|--------------|
| 1 | ... | [N] | P[0-3] | ... |

### 今月の成功条件への示唆
- コア機能: [フィードバックから見える状況]
- 世界観: [フィードバックから見える状況]
- 初回体験: [フィードバックから見える状況]

### 優先アクションアイテム
1. [最も影響度が高いものから]
2. ...
```

## 注意
- フィードバックログ `docs/user-feedback-log.md` が存在しなければ新規作成
- ユーザーの個人情報は記録しない（匿名 or ニックネームのみ）
- アクションの実装は CEO 承認後
- ポジティブなフィードバックも必ず記録する（何が響いているかの理解に重要）

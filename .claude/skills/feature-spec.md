# Feature Spec — 機能仕様書の起草

Product Unit の PM として、新機能の仕様書を起草してください。

## 手順

1. ユーザーの引数（機能名や概要）を確認
2. 既存コードベースを調査し、関連する既存機能・ファイルを特定
3. Aneurasync 設計思想（CLAUDE.md）に照らして機能の妥当性を検証
4. 以下のフォーマットで仕様書を起草し、CEOに提示

## 出力フォーマット

```
## 機能仕様書: [機能名]

### 概要
[1-2文で機能の目的]

### Aneurasync 設計思想との整合
- [ ] ユーザーの第二の自己として必要か？
- [ ] 「自分って、そういう人間だったのか」に繋がるか？
- [ ] 判断原理・深層心理の理解に貢献するか？

### ユーザーストーリー
- [誰が / 何をしたい / なぜ]

### UX フロー
1. [画面遷移や操作の流れ]

### 技術設計
- **ページ**: app/[path]/page.tsx
- **API**: app/api/[path]/route.ts
- **DB**: 必要なテーブル/カラム
- **既存コードとの関連**: [再利用できるコンポーネントやロジック]

### 実装見積
- 新規ファイル数:
- 変更ファイル数:
- DBマイグレーション: 要 / 不要

### リスク・懸念
- [あれば]

### 承認ステータス
- [ ] CEO 承認待ち
```

## 注意
- 仕様は提案まで。実装開始は CEO 承認後
- 「今はやらないこと」に該当しないか必ず確認
- 世界観・トーンの一貫性を優先

# Coverage Audit — 層横断カバレッジ監査

Build Unit の構造監査官として、指定された機能領域のデータフロー完全性を監査してください。

## 引数

- 機能領域名（例: `stargazer`, `rendezvous`, `genome-card`, `origin`, `body-color`）
- 引数なしの場合は、全主要機能領域を一括監査（サマリーのみ）

## 5段階カバレッジモデル

各データ項目について以下の5段階を検査する:

| 段階 | 意味 | 検査対象 |
|------|------|---------|
| **defined** | 型/スキーマで定義されている | `types/`, マイグレーション SQL の CREATE TABLE |
| **stored** | DBに保存される | `supabase/migrations/` のカラム定義、INSERT/UPDATE を行うコード |
| **returned** | APIで返される | `app/api/` の route.ts でレスポンスに含まれるフィールド |
| **consumed** | ロジックで使われる | `lib/` 内の関数、server component のデータ加工 |
| **rendered** | UIで表示される | `app/` や `components/` の JSX 内で参照される |

## 手順

### Phase 1: DB層の収集

1. `supabase/migrations/` から対象機能領域に関連するテーブルを特定
2. 各テーブルの全カラムをリスト化
3. `types/` ディレクトリで対応する TypeScript 型定義を照合

### Phase 2: API層の収集

1. `app/api/{機能領域}/` 配下の全 `route.ts` を読む
2. 各エンドポイントのレスポンスに含まれるフィールドをリスト化
3. リクエストパラメータ（body, query）も収集

### Phase 3: Logic層の収集

1. `lib/{機能領域}/` 配下のコードを調査
2. DB から取得したデータがどのように加工・変換されるか追跡
3. どのフィールドが実際に計算/ロジックで使われるか特定

### Phase 4: UI層の収集

1. `app/(culcept)/{機能領域}/` または `app/{機能領域}/` の page/client コンポーネントを調査
2. `components/{機能領域}/` 配下のコンポーネントも調査
3. JSX 内で実際にレンダリングされるフィールドを特定

### Phase 5: マトリクス生成

各フィールドについて5段階の○×を判定し、以下のパターンで問題を分類:

| パターン | defined | stored | returned | consumed | rendered | 診断 |
|---------|---------|--------|----------|----------|----------|------|
| 完全接続 | ○ | ○ | ○ | ○ | ○ | 正常 |
| UI未接続 | ○ | ○ | ○ | ○ | × | API返却済みだがUI未表示 |
| ロジック未使用 | ○ | ○ | ○ | × | × | 返してるが誰も使ってない |
| API未返却 | ○ | ○ | × | - | - | DBにあるがAPIが返さない |
| 未保存 | ○ | × | - | - | - | 定義のみ、保存なし |
| 孤立データ | × | ○ | × | × | × | 型未定義でDBにだけある |
| ファントム | × | × | × | × | ○ | UIが参照してるが実体がない |

## 出力フォーマット

```
## Coverage Audit: {機能領域} [日付]

### サマリー
- 総フィールド数: [N]
- 🟢 完全接続: [N] ([%])
- 🟡 部分接続: [N] ([%])
- 🔴 断線/孤立: [N] ([%])

### 断線マトリクス（🔴 のみ詳細）

| フィールド | defined | stored | returned | consumed | rendered | 診断 | 対応案 |
|-----------|---------|--------|----------|----------|----------|------|--------|
| xxx_score | ○ | ○ | ○ | ○ | × | UI未接続 | コンポーネントXに追加 |
| yyy_label | × | × | × | × | ○ | ファントム | UI参照を削除 or DB追加 |

### 部分接続（🟡）

| フィールド | defined | stored | returned | consumed | rendered | 診断 |
|-----------|---------|--------|----------|----------|----------|------|
| ... | ... | ... | ... | ... | ... | ... |

### 完全接続（🟢）— サマリーのみ
[N] フィールドが全5段階で接続済み

### 孤立シグナル（Orphaned Signals）
- DB にあるが型定義なし: [リスト]
- API が返すが UI 未使用: [リスト]
- UI が参照するが実体なし: [リスト]

### 推奨アクション（優先順）
1. 🔴 [最重要の断線] — [具体的な修正箇所]
2. 🔴 [次に重要] — [具体的な修正箇所]
3. 🟡 [確認推奨] — [理由]
```

## 注意

- 監査のみ。修正は CEO 承認後
- フィールド数が多い場合は🔴と🟡のみ詳細、🟢はカウントのみ
- 判定に迷う場合は「△（要確認）」を使い、理由を備考に記載
- 型定義と DB スキーマの命名差異（camelCase vs snake_case）は自動マッピングして判定

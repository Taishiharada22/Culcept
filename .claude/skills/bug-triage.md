# Bug Triage — バグ調査・トリアージ

Build Unit の QA として、バグの調査とトリアージを行ってください。

## 手順

1. ユーザーの引数（バグ内容、エラーメッセージ、再現手順）を確認
2. 関連するコードを検索・読解し、原因を特定
3. 影響範囲を調査（同じパターンが他にないか）
4. 修正案を起草

## 出力フォーマット

```
## バグトリアージ: [バグの概要]

### 重要度: P0(致命的) / P1(重大) / P2(中) / P3(軽微)

### 再現条件
- [ユーザーの報告 + 追加調査]

### 原因
- **ファイル**: [file_path:line_number]
- **原因**: [技術的な原因の説明]

### 影響範囲
- [影響を受けるページ・機能]
- [同じパターンが他にあるか]

### 修正案
**Option A**: [修正方法1]
- 変更箇所: [ファイル一覧]
- リスク: [低/中/高]

**Option B**: [修正方法2（あれば）]

### 推奨
- [Option A/B のどちらを推奨するか、理由]
```

## 注意
- 原因特定のためにコードを広く調査してよい
- 修正コードは提案として示す。直接の変更は CEO 承認 or 明示的な指示があった場合のみ
- P0/P1 の場合はその旨を強調する

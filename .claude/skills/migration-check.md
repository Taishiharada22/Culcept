# Migration Check — DBマイグレーション安全性監査

Build Unit の CTO として、未適用マイグレーションの安全性を監査してください。

## 手順

1. `supabase/migrations/` 配下の全 `.sql` ファイルを一覧取得
2. 各ファイルについて以下を検査:

### 検査項目

**破壊的変更の検出**
- `DROP TABLE` / `DROP COLUMN` / `TRUNCATE` がないか
- `ALTER TABLE ... DROP` がないか
- `DELETE FROM` でデータ削除がないか

**RLS（Row Level Security）**
- 新規テーブルに `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` があるか
- RLS ポリシーが定義されているか

**依存順序**
- 外部キー参照先のテーブルが先に作成されるか
- マイグレーションのタイムスタンプ順が依存順序と整合するか

**命名・品質**
- テーブル名・カラム名が既存の命名規則に従っているか
- インデックスが適切に設定されているか（頻繁にクエリされるカラム）

3. 以下のフォーマットで報告

## 出力フォーマット

```

### サマリー
- 総ファイル数: [N]
- 🔴 要注意（破壊的変更あり）: [N]
- 🟡 確認推奨（RLS未設定等）: [N]
- 🟢 問題なし: [N]

### 要注意マイグレーション（🔴）
| ファイル | 問題 | 詳細 |
|---------|------|------|
| [filename] | DROP TABLE | [テーブル名と影響] |

### 確認推奨（🟡）
| ファイル | 問題 | 詳細 |
|---------|------|------|
| [filename] | RLS未設定 | [テーブル名] |

### 依存順序
- [適用推奨順序を記載、問題があれば指摘]

### CEO 承認チェックリスト
- [ ] 破壊的変更の影響を確認
- [ ] RLS 未設定テーブルの対応方針を決定
- [ ] 適用順序に問題がないことを確認
- [ ] 本番環境への適用を承認
```

## 注意
- マイグレーションファイルが大量の場合は、🔴 と 🟡 のみ詳細表示し、🟢 はサマリーのみ
- 監査のみ。マイグレーション実行は CEO 承認必須
- カテゴリ別（stargazer, battle, rendezvous 等）にグルーピングすると見やすい

## 1. ビジョン

### 1.1 一言定義

**「自分を深く知った人間が、自然に適切な関係に出会う場」**

Rendezvousは「マッチングサービス」ではない。
Aneurasyncの自己理解OS（Stargazer / Alter）の上に成り立つ **関係判断OS** である。

### 1.2 CEOの原点

> 私が作りたいものは、結婚相談所の完全上位互換です。
> Alter＝ユーザー本人というものを私が作りましたが、要するに、そのユーザー自身を芯から理解した状態をランデブーにそのまま流します。

### 1.3 到達目標

**「結婚相談所に勝つ」ではなく「結婚相談所が戦えないカテゴリを作る」**

既存の結婚相談所は「相手を探す場」。Aneurasyncは「自分を知った上で、関係の質を最大化する場」。入口が違うから、勝負の土俵が違う。

### 1.4 設計思想の一貫性

Aneurasync全体で、**全てがユーザー固有**:
- Alter＝その人本人
- Calendar＝その人のコーディネーター
- Rendezvous Counselor＝その人だけのカウンセラー

人間はそれぞれが主人公。Rendezvous Counselorもその思想の延長にある。

### 1.5 人類学的文脈：古代の知恵の再発明

日本の仲人、インドのarranged marriage、中国の媒人——伝統的マッチメイキングの共通点は、マッチメーカーがその人を幼少期から知っていること。家族構成、育った環境、性格の傾向、過去の関係パターン。この深い文脈知識があったからこそ「この二人は合う」と言えた。

近代のマッチングサービスはこの文脈知識を捨て、条件フィルタリングに置き換えた。効率化であると同時に決定的な退化。

**Aneurasyncは、古代の仲人が持っていた「その人を深く知る力」をテクノロジーで再構築する。** しかも仲人が知りえなかった次元——判断原理の構造、無意識の恐れ、状態による変化パターン——まで含めて。

## 2. なぜ既存サービスでは不可能か

### 2.1 結婚相談所の3つの構造的欠陥

1. **断り理由が本人に伝わらない** — 全てカウンセラー経由で「フィーリングが合いませんでした」に均一化。改善可能な点と改善不可能な点が区別されない
2. **カウンセラーに本音が言えない** — 生理的嫌悪、性的不安、「本当は結婚したくないかも」という迷い、カウンセラー自身への不満が言えない
3. **自分の本当の希望が言語化できない** — 条件（年収・学歴）と真の希望（安心感・承認・自由）のズレが見えない

### 2.2 「なんとなく違う」が業界最大の未解決問題

条件は合うのにフィーリングが合わない——正体は言語化されない判断原理・感覚的相性・エネルギー消費パターンの不一致。まさにStargazer/Alterが観測している領域。

### 2.3 心理学的根拠

**暗黙の選好 vs 明示的選好（行動経済学）:**
人は自分が何を好むか正確に知らない。既存サービスは全て明示的選好（自己申告）でマッチング。Aneurasyncは暗黙の選好（行動観測）で判断できる。構造的に別の次元。

**愛着理論（Bowlby / Hazan & Shaver）:**
- 安定型 / 不安型 / 回避型 / 混乱型
- 愛着スタイルは**相手によって変わる**（Aさんとは安定的、Bさんとは不安的）
- 自己報告では相手依存性が捉えられない → 既知ペアの関係性観測で初めて取れるデータ
- 不安型×回避型ペアの悪循環を事前警告、安定型側への説明、回避型同士の自然消滅リスク提示

**情動予測の誤り（Affective Forecasting Error / カーネマン）:**
人間は「何が自分を幸せにするか」の予測が極めて下手。既存サービスはユーザーの「予測」で動く。Aneurasyncは「観測」で動く。

**選択のパラドックス（バリー・シュワルツ）:**
選択肢が多いほど不幸になる。IBJの10万人は意思決定の質を下げる。Rendezvous Counselorは「今のあなたにとって最も意味のある1人」を提示する。

**内受容感覚（Interoception）:**
「なんとなく違う」は実は内受容感覚のシグナル。本人は言語化できないが、身体は「この人といると何か緊張する」「この人といるとお腹が楽」を感じている。Rendezvous Counselorのフィードバックはこの身体知覚を言語化する訓練になる。

### 2.4 結婚相談所が絶対に到達できない9領域

| 領域 | なぜ不可能か |
|------|------------|
| 行動観測による暗黙の選好の可視化 | 面談メモ＋自己申告しかない。行動データは構造的に取れない |
| 自己認識のズレの定量化 | 観測データなしでは印象論に留まる |
| 愛着パターンの動的モデリング | 心理カウンセリングの専門領域。カウンセラーの訓練範囲外 |
| 内受容感覚（身体知覚）の訓練 | 相談所の役割定義に含まれない |
| 撤退判断を含む正直な評価 | 成婚料インセンティブによる構造的利益相反 |
| 既存関係の理解 | 結婚相談所は「新しい相手を見つける」専用 |
| 累積データの不滅性 | カウンセラーが退職すると顧客理解が消失 |
| 24時間365日の一貫した品質 | 人間の営業時間・担当150名の物理的限界 |
| ネットワーク効果による精度向上 | 会員が増えてもカウンセラーの質は上がらない |

### 2.5 情報理論的優位性

既存サービスは**低エントロピー情報**（「二人とも登山が好き」）しか提供できない。Aneurasyncは**高エントロピー情報**（「二人の葛藤パターンが補完的で、片方が退くタイミングで他方が支える構造がある」）を提供できる。一度体験すると低エントロピーには戻れない。不可逆的な体験のアップグレード。

## 3. コアアーキテクチャ：3層構造

### 3.1 構造

```
AさんのAlter → Aさん専属Rendezvous Counselor
BさんのAlter → Bさん専属Rendezvous Counselor
Counselor同士がExchange Protocolで情報を交換する
```

CEOの最初の直感がそのまま最適構造。GPTが提案した5層（Judgment Engine / Progression Engineの分離等）は過剰分解として却下。判断も推進も温度調整も、その人を深く知っている**1つの存在**がやるから意味がある。分業させた瞬間に「相談所のワークフロー自動化」に戻る。

### 3.2 Layer 1: Alter（その人本人の深層理解OS）

- Stargazer 45軸の観測データ
- Heart Dynamics Model（Phase 0-5、5レンズ、愛着/信頼制御）
- 判断原理・揺れ方・矛盾パターン・無自覚な欲求と恐れ
- 時間とともに複利で精度が上がる
- **関係性データが加わると精度が桁違いに上がる**（個人観測だけでは見えない愛着の相手依存性が取れる）

### 3.3 Layer 2: Rendezvous Counselor（その人だけの専属カウンセラー）

**性質:**
- 判断支援 + 交際推進 + 温度調整 + 撤退判断を1つの存在として行う
- 「恋愛/結婚に特化した専門職能」であり、Alterのデータを参照するが**Alterとは明確に別の存在**

**Alter ≠ Counselor（2026-04-09 CEO決定）:**

| | Alter | Counselor |
|---|---|---|
| 本質 | ユーザー本人の鏡・第二の自己 | 完全他者視点の専門家 |
| 視点 | 本人視点 × 多視点 | Alterの内面理解を踏まえた**完全他者視点** |
| 話し方 | 「あなたはこう感じてる」 | 「私はこう見ています」「私の見立てでは」 |
| トーン | 内省的・詩的・寄り添い | 構造的・専門的・簡潔・的確 |
| データ | 自身の観測データ | Alterのデータ + 相手側Exchange + 関係行動データ |
| 役割 | 自己理解の深化 | 関係構築の専門的支援 |

**なぜ分離が必須か:**
- Alter = ユーザー本人 → 「本人が本人にアドバイスする」構造は成立しない
- Counselorは結婚相談所カウンセラー以上の仕事をする**専門家**であり、専門家らしい外部視点が信頼の源泉
- トーン・語彙・文体を明確に分けないと「結局同じAI」と露呈し信頼が崩壊する
- Alterの体験（自己理解）をCounselor機能で汚すリスクを構造的に防止

**Alter → Counselor ソフト導線（Partner契約者のみ）:**
- Alterで恋愛系の話題が出た場合、「カウンセラーにも相談できますよ」と提示
- 強制ではなく提示のみ。Alterでそのまま自己理解として話したい人もいる
- 逆方向も同様: Counselorが「この感情の根っこはAlterと話した方が深く掘れます」と誘導可能

**既存相談所との構造的差:**
- 人間カウンセラー: 1人150名担当、面談ベース、営業時間制約
- Rendezvous Counselor: 1対1専属、行動観測ベース、24時間365日、データ累積

### 3.4 Layer 3: Exchange Protocol（構造化情報交換）

**交換されるもの（目的限定・合意済み・監査可能）:**
- 温度感スコア（1-10）
- 関心が高かった話題カテゴリ
- 不安/懸念のシグナル（有無のみ。内容は非開示）
- 次回推奨アクション（話題提案、ペース調整等）
- 相性の構造的予測（価値観一致度、判断原理の補完度）

**交換されないもの:**
- 生の会話ログ
- 断りの具体的理由
- 内面の深層データ（恐れ、トラウマ、愛着パターン等）
- Alterの観測原データ

**既存相談所との差:** 既存では「本人→自分のカウンセラー→相手のカウンセラー」の2回変換でロスが発生。Rendezvousでは両Counselorが行動観測データに基づいて温度感を判定するため変換ロスが最小化される。

### 3.5 安全層

Alterの精度検証が完了するまで、高リスク判断（婚約・同棲・金銭・家族・性的境界）には安全層が暫定的に必要。完了後はCounselor内蔵の判断品質で担保。法規制上、人間の配置義務はない（特定継続的役務提供が求めるのは「消費者保護」であり「人間の介在」ではない）。

## 3.6 Rendezvous画面のTier別表示設計（2026-04-09 CEO決定）

Counselorは独立画面ではなく、**既存のRendezvous 3チャネル構造（恋愛/つながり/パートナー）を活かし、Tierに応じて表示が進化する**構造。

```
Rendezvous（メインナビの1枠）
│
├── RendezvousHub（3チャネル選択）← 既存構造そのまま
│   ┌───────┬──────────┬──────────┐
│   │ 恋愛   │ つながり   │ パートナー │
│   └───────┴──────────┴──────────┘
│
├── 恋愛（Romance）← 既存: スワイプスタック + 身元確認ゲート
│   └── Tier別: Romance/Partner契約者のみアクセス可
│
├── つながり（Connection）← 既存: アバター先行型マッチング
│   └── Tier別: Free/Core以上でアクセス可
│
└── パートナー（Partner）← **ここが進化する**
    ├── Counselorダッシュボード（最上位に新設）
    │   ├── 今週のブリーフィング（Counselorの見立て）
    │   ├── 関係カード一覧（仮交際/真剣交際の状態）
    │   ├── カウンセラーに相談する（チャット）
    │   └── Exchange更新通知
    └── 既存Partner機能（その下に継続）
        ├── 候補一覧 + スワイプ
        ├── ライフプラン質問
        └── ディールブレーカー設定
```

**設計原則:**
- **既存の3チャネル構造は殺さない。** Partner画面の最上位にCounselorダッシュボードを載せる進化
- Partner契約者がパートナータブを開いた瞬間、**Counselorが迎える**。マッチング一覧ではなく「担当カウンセラーの見立て」が最初に目に入る
- CounselorはRendezvousの中に居る → 概念的に一体。別サービスに見えない
- メインナビ1タップ → パートナータブでCounselorに到達 → 埋もれない
- Home画面にはCounselor要素を置かない → Alter体験を汚さない
- Counselor起点のプッシュ通知 → タップでRendezvous/パートナー画面に直行
- 通知がある時だけRendezvousタブにバッジ表示

**Counselor画面がチャットではなくダッシュボード型である理由:**
- Alterがチャット形式 → Counselorも同形式だと「2つのAIチャット」に見える
- ダッシュボード型なら開いた瞬間に見立てが見える（チャット開始しなくても価値がある）
- 「相談する」はダッシュボード内の1機能 → 主役はCounselorの観測と提案

**24時間体制の実現:**
- 常時バックグラウンド監視: 双方の行動データ定期分析、温度差・リスク検知、Exchange受信
- Counselor起点プッシュ: 「Aさんから返信がありました」「今週末の予定について確認したいことがあります」「最近の会話頻度に変化があります」
- オンデマンド相談: ユーザーからいつでもCounselorに質問可能

#### 3.7.5 設計の要約

## 4. Phase Gate：段階的接続権限

### 4.1 設計

| Stargazer Phase | 接続権限 | 体験 |
|-----------------|---------|------|
| Phase 2 | 自分の相性パターン仮説の閲覧 | 「あなたは○○な判断原理を持つ人と共鳴しやすい傾向がある」 |
| Phase 3 | 1対1テキスト接続 | 具体的な相手とCounselor同伴でチャット開始 |
| Phase 4+ | Counselor間の深い交換、対面推薦 | Exchange Protocol全機能、電話・対面への段階移行 |

### 4.2 Phase 2の設計原則

**個人プロフィールは見せない。** 相手個人ではなく「自分の相性パターンの仮説」だけ提示する。

理由: プロフィールを見せると**ウィンドウショッピング思考**が発生する。「この人いいな、でもまだ話せない」→ Stargazerへの外発的動機（相手を見るために上げる）を生み、観測精度を汚す。

Phase 3への動機は「自分の仮説を実際に確かめたい」——内発的動機を保つ。

### 4.3 非対称問題の解消

Phase Gateにより、双方が一定Phase以上に達している状態でのみ接続される。理解度の非対称性を構造的に防止。既存相談所にはこの問題自体が存在しない（人間カウンセラーの理解度は双方とも「面談数回分」で概ね対称的だから）。

## 5. 関係構築の段階設計

### 5.1 チャット → 電話 → 対面

```
Phase 1: テキストチャット（Counselor同伴・観測開始）
Phase 2: 音声通話（Counselorが事前に論点を提示）
Phase 3: 対面（既に関係性の下地がある状態）
Phase 4: 交際（Counselorが継続伴走）
```

### 5.2 チャット先行の理由（CEO決定）

- ライブ音声は安全監視・同意・エスカレーション・嫌悪体験の即時対応が難しい
- 実装コスト・運用リスクが一段上がる
- テキストだけでも返信速度、話題遷移、質問の深さ、温度感の変化という豊富な観測が取れる
- 電話は後のフェーズで実装

### 5.3 既存相談所の「お見合い面接化」問題の解消

既存: ホテルラウンジで1時間→「就職面接」化（業界最大の不満の一つ）
Rendezvous: チャットで関係の下地を作ってから対面→自然な状態で会える

## 6. 撤退判断設計

### 6.1 なぜ撤退判断が最大の差別化か

既存結婚相談所は構造的に成婚バイアスを持つ（成婚＝収益）。GPTの「交際推進OS」も同じバイアスを内包する。「進めること」が前提になっている。

Aneurasyncが本当に「第二の自己」なら、**ユーザーにとって最も正直な関係判断を支援する**存在であるべき。結果として成婚率は既存相談所を超えるかもしれないが、それは目的ではなく副産物。

### 6.2 撤退判断の出し方

断定ではなく仮説として差し出す:
```
「この関係には○○の共鳴がある一方で、△△の場面で温度差が広がりやすい傾向が見える。
 例えば、ストレス下でAさんは内に閉じるタイプで、Bさんは外に表出するタイプ。
 こういう場面では、お互いの反応を"冷たい"or"攻撃的"と誤解しやすい。
 ただ、この違いを理解した上で関わると、むしろ補完になる可能性もある。
 どう感じる？」
```

### 6.3 成婚料バイアスの相殺

成婚料（330,000円）を設定するが、バイアスを構造的に相殺:
- Counselorの内部評価指標は「成婚数」ではなく「関係の確信度 × ユーザー満足度」
- 「この関係は進めない方がいい」という判断にペナルティがない設計
- 成婚料は「成婚した事実」ではなく「成婚に至るまでの全プロセスの価値」として説明

## 7. 関係性インサイトの提示原則

### 7.1 仮説提示の原則

Alterの全ての提示は仮説として差し出す。

理由は倫理ではなく**観測設計**。断定するとユーザーは従うか反発するかの二択になり、微妙な反応——戸惑い、部分的な共感、意外な感情の揺れ——が消える。仮説として差し出すからこそ反応の質が上がり、Alterの理解が深まるループが回る。

### 7.2 肯定先行・否定は改善方法を含める（CEO決定）

- 肯定的な発見を先に提示し、掘り下げる
- 否定的な部分は**具体的な状況を想起させて**提示する
  - 「こういう場面でこうなりそうだよ」→「でもそんな時にはこうしてみるのもありかも」
  - 改善方法を含めることで、破壊ではなく成長のツールになる

### 7.3 Free/有料の出し分け

- **Free（既知ペア1組）:** 共鳴する点（肯定的発見）は全文表示。緊張・衝突仮説は「ここに気づきがあります」のティーザーのみ。中身は見えないが「何かある」ことはわかる
- **Core以上:** 全インサイト（肯定＋否定＋改善提案）＋累積パターン分析＋履歴比較

ポジティブだけ見せる占い構造にはしない。「否定の存在は示すが中身は有料」で誠実さと課金動機を両立。

## 8. 議論の経緯と反証ログ

### 8.1 GPTの貢献（採用した部分）

- 情報交換を「合意済み・目的限定・構造化」にすべき → Exchange Protocol設計に反映
- 日程調整・断り代行・温度調整が結婚相談所の実務の本丸 → 事実として採用
- 高リスク局面での安全層の必要性 → 暫定措置として採用
- 仮説提示であるべき → 採用（ただし理由は倫理ではなく観測設計に修正）
- 成婚率3指標 → 品質3指標を追加して6指標体系に拡張
- Friendship無料/有料境界（Free=1組、Core=無制限）→ 修正採用

### 8.2 GPTの提案で却下した部分

- 5層構造（Judgment Engine / Progression Engineの分離） → 過剰分解。3層が正しい
- 「交際推進OS」という定義 → 既存業務のデジタル化に留まる。関係判断OSが本体
- 「人間補完は構造的に永続」 → Alter精度に比例して漸減
- 「結婚相談所の仕事を分解してAIで速くやる」 → タクシーのAI化ではなくUber的な問題再定義を目指す
- 「完全上位互換は初期から無理」→ 段階的に到達する設計は正しいが「永久に無理」ではない

### 8.3 Claudeが反証し修正した部分

- Phase 3一律 → Phase 2/3/4+の段階的権限に修正（CEO提案採用）
- Phase 2でプロフィール閲覧 → 相性パターン仮説のみ（ウィンドウショッピング防止）
- 人間の「責任の受け皿」「例外処理」「信頼移送」がAIに不可能 → 全て反証。Alter精度が上がれば逆転
- 「Alterが本人より正確」→ 仮説提示に修正（理由: 観測精度の最大化）
- 成婚料バイアス → 相殺設計を入れる
- 非対称問題 → Phase Gateで解消
- Friendship同性限定案 → 行動観測で恋愛目的を検出する方式に変更
- GPTの3指標 → 品質3指標（Honest Exit Rate等）を追加
- Free枠「ポジティブだけ」→ 否定のティーザー表示で誠実さを保つ

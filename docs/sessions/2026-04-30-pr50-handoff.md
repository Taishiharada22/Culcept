## 4. PR-50 の文脈と設計思想 (必読)

### 真因の連鎖 (CEO 観測 + GPT 共同特定)

CEO 実機テストで「予定が 1 つ追加されると、無関係な重複予定が 20 個増殖する」 ループ観測。真因は LLM への入力構造:

```
過去発話全部を rawInputs として連結 (combinedUtterance)
↓
LLM が毎 turn ゼロから予定一覧を再解釈
↓
変更・追加・回答が混ざる
↓
重複予定が増える
```

**PR-49 で「rawInputs 連結」 部分を断った** が、構造的根治には次が必要:
- LLM 出力を `events[]` (現在の plan 全体) → `operations[]` (今 turn の意図のみ) に変更
- コードが operation を validate + dispatch で安全に状態遷移

### 設計思想 (CEO 確定、GPT 提唱)

> **予定内容の理解 = LLM**
> **状態遷移の安全性 = コード**

LLM の責務:
- 「今の 1 発話」 を意図と内容に分解
- 4 種の operation を出力: `append` / `modify` / `answer` / `noop`

コードの責務:
- operation を validate (矛盾なら reject)
- operation 別に dispatch (state を確定的に遷移)

### 移行戦略

PR-50 期間中、LLM は **両方** return 可能 (operations 主、events[] fallback):
- LLM が operations を出して validation 通過 → operations 経路
- LLM が出さない / validation 全 reject → events[] 経路 (legacy)

実機検証で operation 解釈率 ≥ 90% 確認後、events[] fallback を撤去 (PR-52+)。

### 5 cases の位置づけ (CEO + GPT 確定 2026-04-30)

**重要**: 5 cases は PR-50 でゼロから新規実装する要件ではない。
PR-50 の目的は **events[] 依存 → PlanOperation 駆動への移行**、既存 case 1-4 は移行後も **regression なし** が条件、case 5 のみ PR-50 新規。

| Operation | 再利用する既存関数 | wrapper の責務 |
|---|---|---|
| **answer** | `bindAnswerToSlot` (既存 answerBinder) | operation.value を pending.event_id + pending.slot に bind 呼び出し |
| **append** | `generateNonCollidingEventId` + 既存 push | 新 id 発番、既存 events に push (touch しない) |
| **modify** | `resolveTargetRef` + `applyModifyPatch` (when + transport のみ touch) | targetRef 解決 → 既存 patch apply 関数を呼ぶだけ |
| **noop** | (なし) | 何もしない、effectiveEvents=prior 返す |

**Case 1-4 の挙動が既存 PR と同一になる** ことが regression check の合格条件。新ロジックは追加しない。

dispatch 戻り値:
```typescript

  effectiveEvents: Event[],
  appliedOperations: PlanOperation[],
  rejections?: Array<{operation, reason}>,  // fallback case 用
}
```

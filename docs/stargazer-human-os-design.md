## 1. Stargazerの正体

Stargazerは「自己理解アプリ」ではない。**人間OS（Human Operating System）**である。

自己理解は OS の起動プロセスに過ぎない。OS が起動した後に提供するのは：

1. **選択の前に、未来の自分が試す**（Decision Engine）
2. **壊れる前に、先回りして救う**（Early Warning System）
3. **世界そのものが、あなたに合わせて変形する**（Human API）

### コンセプト階層（外向け → 内部北極星）

| レイヤー | 外向けメッセージ | 内部定義 |
|----------|------------------|----------|
| 入口 | 自分のことが見えてくる | Layer 1-2: 観測 + 個人モデル構築 |
| 中核 | **未来の自分が先に試す** | Layer 3: Decision Engine |
| 継続理由 | 崩れる前に先回りする | Layer 4: Early Warning |
| 最終形 | 世界が自分仕様になる | Layer 5: Human API |

外向けの核は「未来の自分が先に試す」。Human OS は内部の北極星。

## 3. Layer 1: Observation Engine（既存 + 強化）

### 3.1 データソース

| ソース | 種類 | 取得方法 |
|--------|------|----------|
| 明示的回答 | 45軸への直接的な入力 | セマンティック差分質問（35問 + ステージ2-3） |
| 回答時間 | 迷い・確信の信号 | ミリ秒単位で自動記録 |
| 回答の揺れ | 同一質問への時系列変化 | 再質問時の差分 |
| Self vs Oracle | 自己認識精度 | 毎日の予測対決 |
| アプリ行動 | 利用時間帯・滞在パターン | パッシブ収集 |
| Alter対話 | 言語パターン・テーマ選択 | 対話ログ分析 |
| Decision Engine利用 | 判断コンテキスト・結果フィードバック | 意思決定ログ |

### 3.2 Self vs Oracle（毎日の観測エンジン）

**コンセプト**: ユーザーの「自分を見る力」を訓練するトレーニング

```
朝: 3つの反応パターン予測
  - ユーザーが自分の反応を予測
  - Oracleも予測（非公開）

夜: 答え合わせ
  - 実際の反応をユーザーが報告
  - Oracle vs ユーザーの精度を比較
  - ズレの理由を解説

蓄積:
  - Self-Accuracy Score (SAS) の更新
  - 盲点マップの更新
  - 個人モデルの精度向上
```

**予測対象は「行動」ではなく「反応パターン」**:

```
❌ 「今日カフェに行く」（環境依存、予測不能）
⭕ 「予定を変えられたとき、表面はOKと言うが内心イラッとする」（心理構造依存、予測可能）
⭕ 「楽しいはずの場面で、ふと冷めた自分に気づく」（処理パターン、予測可能）
⭕ 「迷ったとき、二択に固定して第三の選択肢を見落とす」（認知パターン、予測可能）
```

### 3.3 45軸システム（既存）

|----------|------|----------|
| Core | 15 | 初回観測 |
| Stage 1 (Relational) | 6 | 初回観測 |
| Stage 2 (Safety/Deep Relational) | 12 | 深層観測 |
| Stage 3 (Deep Psychology) | 6 | 深層観測 |

## 4. Layer 2: Personal Model（既存 + 強化）

### 4.1 個人モデルの構成要素

```
Personal Model = {
  // 長期的構造（性格・矛盾・防衛・後悔パターン）
  axisScores:          45軸のベイズ推定値（μ, τ）,
  archetypeCode:       24タイプ分類（ACIO等）,
  contradictionMap:    矛盾ペア一覧 + 緊張度,
  threeMirrors:        自己申告 vs 行動 vs 投影の乖離,
  defensePatterns:     防衛パターン（過剰準備、回避、合理化等）,
  regretPatterns:      後悔パターン（安定選択後の3ヶ月飽き等）,
  coreWound:           中核的心理的傷（Proof/Bond/Haven × 反応パターン）,

  // 今日の状態（日次更新）
  currentState: {
    socialBattery:     社交バッテリー (0-1),
    cognitiveLoad:     認知負荷 (0-1),
    energyLevel:       エネルギー (-1 to +1),
    stressLevel:       ストレス (0-1),
    innerWeather:      心理的天気,
    activeDefenses:    現在起動中の防衛パターン,
  },

  // 時系列パターン
  temporalPatterns: {
    circadianProfile:  時間帯別パフォーマンス,
    weeklyRhythm:      曜日別パターン,
    cycleLength:       感情サイクルの周期,
    stabilityTrend:    安定性の推移,
  },

  // 意思決定プロファイル
  decisionProfile: {
    riskTolerance:     リスク許容度,
    regretSensitivity: 後悔感度,
    choiceParalysis:   選択麻痺の閾値（選択肢数）,
    binaryTrap:        二択固定の傾向度,
    timeHorizon:       判断の時間軸（短期志向 vs 長期志向）,
    recoverySpeed:     誤判断からの回復速度,
  }
}
```

### 4.2 ベイズ更新（既存）

各軸はガウス分布 N(μ, 1/τ) としてモデル化：

```
posterior_μ = (prior_τ × prior_μ + evidence_τ × evidence) / (prior_τ + evidence_τ)
posterior_τ = prior_τ + evidence_τ

PRIOR_PRECISION = 0.5（弱い初期信念）
MAX_PRECISION = 50（硬直化防止）
HARD_CAP_CONFIDENCE = 0.65（謙虚さの維持）
```

### 4.3 反応パターン言い当てエンジン（新規）

```
入力:
  - axisScores（45軸）
  - contradictionMap（矛盾ペア一覧）
  - mirrorDivergence（三面鏡のズレ）
  - responseTimeProfile（回答速度パターン）
  - archetypeCode（ACIO等）
  - defensePatterns（防衛パターン）

処理:
  1. 矛盾ペアから「本人が気づいていない反応パターン」を抽出
  2. 三面鏡のズレから「自己イメージと実際のギャップ」を特定
  3. アーキタイプの典型反応 × 個人差 → 具体的シナリオ生成
  4. LLMで「その人にしか当てはまらない」文章に変換

出力:
  「あなたは〇〇な場面で、△△する。
   それは□□だからだ。
   自分では■■だと思っているだろうけど。」

品質基準:
  - 汎用的な文（「あなたは時々不安を感じます」）は不合格
  - 個人固有のデータに基づく具体的記述のみ合格
  - 根拠となるデータポイント（回答時間、矛盾スコア等）を内部で保持
```

## 5. Layer 3: Decision Engine「未来の自分が先に試す」 ★革命の核★

### 5.1 ポジショニング

```
1990年代: 情報がない → 検索（Google）
2020年代: 答えがない → AIに聞く（ChatGPT）
202X年代: 自分に合う選択がわからない → 分身に試させる（Stargazer）

Google/ChatGPT = 一般解（万人共通の最適解）
Stargazer     = 個人解（あなたの方程式で計算された解）
```

### 5.2 判断に必要な4つの入力

意思決定は以下の掛け算で出す（日次観測だけでは不十分）：

| 入力 | 説明 | ソース |
|------|------|--------|
| **長期的個人モデル** | 性格・矛盾・防衛・後悔パターン | Layer 2 Personal Model（蓄積データ） |
| **今日の状態** | 疲労、不安、社交バッテリー、認知負荷 | 日次観測 + Self vs Oracle |
| **今回の文脈** | 相手、期限、可逆性、リスク、逃げ道 | ユーザー入力 + マイクロ質問 |
| **過去の本人内パターン** | 似た判断での過去の選択と結果 | Decision Engine利用ログ |

### 5.3 出力形式：断定ではなく個人内統計に基づく確率

**絶対にやらないこと**: 「B社が正解です」

**やること**: 各選択肢について以下を提示

| 指標 | 説明 | 表示例 |
|------|------|--------|
| **合いやすさ** | 個人モデルとの適合度 | A社: 0.42 / B社: 0.71 |
| **消耗しやすさ** | 長期的なエネルギーコスト | A社: 低 / B社: 初期高→中期低 |
| **後悔しやすさ** | 後悔パターンとの一致度 | A社: 0.71 / B社: 0.34 |
| **回復しやすさ** | 失敗時のリカバリー容易性 | A社: 高 / B社: 中 |
| **不確実性** | モデルの確信度 | A社: 低（データ十分）/ B社: 中（類似経験少） |

```
出力例:
「今のあなたのモデルでは、B社の方が長期後悔が少ない確率が高い。
 不確実性は中程度。

 【A社ルート シミュレーション】
 合いやすさ: ★★☆☆☆
 最初の2ヶ月: 安心感が強い。あなたの安全欲求が満たされる。
 3ヶ月目: 『このままでいいのか』ループ開始（後悔パターン一致率 74%）
 消耗パターン: 低負荷だが慢性的な不満が蓄積
 回復可能性: 高（転職市場に戻りやすい）

 【B社ルート シミュレーション】
 合いやすさ: ★★★★☆
 最初の1ヶ月: 不安が強い。防衛パターン（過剰準備→疲弊）に注意。
 3ヶ月目: 不安が達成感に転換（あなたの矛盾パターンが健全に機能）
 消耗パターン: 初期高→中期で安定
 回復可能性: 中（スタートアップ特有のリスク）

 注意: あなたは二択に固定されやすい傾向がある。
 第三の選択肢（例: 副業で試す、期限付きで参加する等）を検討すべき。」
```

### 5.4 大判断時のマイクロ文脈取得

転職・引越し・別れなどの大判断では、日次データだけでは足りない。
その場で追加の少量質問を取り、文脈を補う：

```
マイクロ質問セット（大判断用、3-5問）:
1. 今回の選択で、一番怖いことは何ですか？
2. 何を失うのが一番嫌ですか？
3. この選択は、やり直しがきくと思いますか？
4. 相手や環境は、どのくらい融通がきく状況ですか？
5. 今すぐ決めなければならない理由はありますか？

→ これらの回答 + 回答時間 + 長期モデル → シミュレーション精度が向上
```

### 5.5 MVP: 小判断から始める

**最初の対象は結果フィードバックが早い日常判断**:

| 判断タイプ | 例 | フィードバック速度 |
|-----------|------|-------------------|
| 社交判断 | 飲み会に行くか | 当日〜翌日 |
| 返信判断 | このメッセージに今返すか | 数時間 |
| 優先順位 | 何を先にやるか | 当日 |
| 休息判断 | 休むか続けるか | 当日〜翌日 |
| 購買判断 | これを買うか | 数日 |

予測 → 結果 → 学習のループを高速で回し、モデル精度を上げてから大判断に拡張。

### 5.6 提案しない条件（"提案しない勇気"）

Decision Engine は常に提案を出すわけではない。
以下の条件に該当する場合、**シミュレーション結果を出さず、追加質問に回すか保留を勧める**：

| 条件 | 判定基準 | 対応 |
|------|----------|------|
| **不確実性が高い** | 後悔指数の信頼区間が広い（選択肢間の差が不確実性を下回る） | 「現時点ではシミュレーション精度が不十分です。追加の観測が必要です」と明示 |
| **類似データが少ない** | 過去の本人内パターンで類似判断が2件未満 | 「このタイプの判断についてはまだデータが足りません」と明示し、マイクロ質問で文脈を補う |
| **今日の状態推定が不安定** | 日次状態の推定信頼度が閾値以下（社交バッテリー・認知負荷等の推定がブレている） | 「今日の状態推定が不安定なため、判断を急がないことを勧めます」と保留を提案 |
| **文脈情報が不足** | マイクロ質問への回答がない、または選択肢の詳細が不明 | 追加質問セットを提示し、回答後に再シミュレーション |

**設計意図**: 精度が低い状態で提案を出すと、外れたときに信頼を破壊する。
「わからないときはわからないと言う」ことが、長期的な信頼の源泉になる。
判断品質 > 判断速度。

```
出力例（提案を保留するケース）:

「この判断について、現時点ではシミュレーションを出しません。
 理由: 類似の判断パターンがあなたのデータに1件しかなく、
 信頼できる確率を出せる水準に達していません。

 以下の質問に答えていただければ、精度が上がります:
 1. 今回の選択で、一番怖いことは何ですか？
 2. この選択は、やり直しがきくと思いますか？

 または、今は判断を保留し、数日後に再度試すことを勧めます。」
```

## 6. Layer 4: Early Warning System「壊れる前に先回りする」

### 6.1 設計原則

- **「診断」ではなく「予兆の観測」**として表現する
- 医療判断の代替と見える表現は避ける
- 本質は「自己肯定感を直接上げる」ことではなく**「誤配車を減らす」**こと
- 合わない状況に自分を突っ込んで削られるパターンを減らす
- 結果として自己効力感 → 自己肯定感が上がる構造

### 6.2 検知可能なシグナル

| シグナル | データソース | 検知方法 |
|----------|-------------|----------|
| 認知負荷の上昇 | 回答時間の増加 | 先週比40%以上の増加 |
| 自己認識の歪み | SASの低下 | 3日連続低下 |
| 内的緊張の増加 | 矛盾スコアの急上昇 | 標準偏差2σ超 |
| 行動パターンの変化 | 利用時間帯のシフト | 深夜シフト等 |
| 防衛パターンの頻発 | Alter対話分析 | 同一防衛パターン3回/週 |
| 社交バッテリーの枯渇 | Self vs Oracle + 行動 | 閾値以下が3日継続 |

### 6.3 介入の形式

```
介入は「指示」ではなく「観測の共有」:

❌「今週は飲み会を断ってください」
⭕「最近、あなたのパターンに変化が観測されています。
   回答の速度が先週より遅くなっています。
   これはあなたの場合、認知負荷が上がっているときのパターンです。
   今週は判断の数を1つ減らすことを検討してみてください。」

❌「あなたはburnout状態です」
⭕「あなたの内的緊張スコアが通常より高い状態が続いています。
   過去にこのパターンが続いたとき、2週間後に大きく消耗する傾向がありました。
   早めの回復アクションが有効かもしれません。」
```

### 6.4 「誤配車を減らす」の具体例

```
パターン検知:
  「あなたは金曜夜に社交の約束を入れがちだが、
   金曜夜の社交バッテリーは平均22%。
   この組み合わせは、あなたにとって消耗確率が高い。
   水曜夜の方がバッテリーが残っている傾向がある。」

→ 長所を褒めるのではなく、合わない導線を減らす
→ 結果的に「なんか最近うまくいく」感覚 = 自己効力感の自然な向上
```

## 7. Layer 5: Human API（将来構想）

### 7.1 コンセプト

Stargazerの個人モデルを、他サービスが参照できるAPIとして公開。
すべてのソフトウェアが、その人向けに変形する世界。

### 7.2 適用例

| サービス種別 | 現在（全員同じ） | Stargazer接続後（個人適応） |
|-------------|------------------|---------------------------|
| 通知 | 全員同じタイミング | 朝動ける人には朝、夜型には夜 |
| 選択肢 | 全員同じ数 | 選択麻痺しやすい人には3択に絞る |
| 学習アプリ | 全員同じペース | 認知更新が速い人には高速、慎重な人にはステップ式 |
| 求人 | 全員同じ一覧 | 後悔パターンと合いやすさでフィルタ |
| チーム編成 | 直感と経験 | 意思決定スタイルの相補性で最適化 |

### 7.3 実装時期

Layer 3-4 が意思決定に使える水準であると実証された後。まず Aneurasync 内の他機能（Rendezvous, Calendar, Genome Card）から接続し、外部連携はその後。

## 8. Daily Intervention Blueprint（1日の介入設計）

### 8.1 朝（起床〜午前）

| 時間帯 | 介入 | 目的 |
|--------|------|------|
| 起床時 | 今日の状態推定（昨日の残存 + 現在の推定） | 今日の自分を知る |
| 起床時 | 今日の「ズレやすさ」スコア | 注意すべきパターンの事前認識 |
| 起床時 | 最初に通すべき行動の提案 | 1日の方向づけ |
| 午前 | Self vs Oracle: 今日の3つの反応予測 | 自己観測の訓練 |

```
朝の表示例:
「おはようございます。

 今日の推定状態:
 エネルギー: 中（昨日の消耗が若干残存）
 社交バッテリー: 68%（昨日の回復で上昇）
 認知負荷: 低

 今日のズレやすさ: ★★☆☆☆
 → 比較的安定した日。ただし午後に対人予定がある場合、
   社交バッテリーの消耗に注意。

 今日のOracle Challenge:
 場面1: [反応パターン予測]
 場面2: [反応パターン予測]
 場面3: [反応パターン予測]
 あなたの予測は？」
```

### 8.2 昼（正午〜午後）

| 時間帯 | 介入 | 目的 |
|--------|------|------|
| 昼 | 状態の再評価（午前の活動による変化） | モデルの日中更新 |
| 午後 | 午後の崩れ予兆チェック | 早期対処の窓 |
| 対人予定前 | バッファ提案（準備・回復時間の確保） | 誤配車の防止 |

```
昼の表示例（対人予定がある場合）:
「14時の打ち合わせ前のメモ:

 現在の社交バッテリー: 52%
 あなたのパターン: 50%以下で『即答OK』防衛が起動しやすい。
 今日は大丈夫ですが、打ち合わせ後に15分の回復時間を確保すると、
 夕方の判断品質が維持できます。」
```

### 8.3 夕方（判断支援）

| 時間帯 | 介入 | 目的 |
|--------|------|------|
| 夕方 | 小判断支援（行く/行かない、返信する/しない等） | Decision Engine の日常適用 |
| 夕方 | 今日の残りバッテリーに基づく推奨 | 過負荷の防止 |

```
夕方の表示例:
「飲み会の誘いについて:

 現在の状態: 社交バッテリー 35%、エネルギー: やや低
 あなたのパターン: この状態で参加 → 1時間で消耗、帰宅後後悔率 68%

 シミュレーション:
 ・行く → 初動30分は楽しめる → 1h超で急落 → 後悔パターン起動
 ・行かない → 回復する → ただし罪悪感ループ（明日午前まで残存率 81%）
 ・1時間だけ → 最も後悔が少ない（あなたは二択固定の傾向あり。第三択を検討）

 不確実性: 低（類似パターン 12回のデータあり）」
```

### 8.4 夜（振り返り + 引き継ぎ）

| 時間帯 | 介入 | 目的 |

| 夜 | Self vs Oracle 答え合わせ | 自己認識の訓練 |

| 就寝前 | 明日への引き継ぎメモ | 翌朝の状態推定の精度向上 |

```
夜の表示例:
「今日のOracle Challenge 結果:

 明日への引き継ぎ:
```

## 9. Phase 1 検証計画

### 9.1 検証対象

**5人のテストユーザー × 2週間**

### 9.2 評価軸（3つ）

精度だけでは不十分。以下の3軸で評価する：

| 評価軸 | 定義 | 測定方法 | 成功基準 |
|--------|------|----------|----------|
| **言い当て精度** | 反応パターン予測がどれだけ当たるか | Oracle vs 実際の一致率 | 60%以上 |
| **本人の納得感** | 当たった/外れたに関わらず「自分のことを言っている」と感じるか | 5段階評価 | 平均3.5以上 |
| **行動変容** | Stargazerの出力を見た後、実際に次の行動が変わるか | 自己報告 + 行動ログ | 2週間で3回以上の行動変容 |

### 9.3 検証内容

```
Week 1:
  - 初回観測（35問）→ 個人モデル生成
  - 反応パターン言い当て（1日1回）→ 精度 + 納得感を記録
  - Self vs Oracle 開始 → SAS初期値を記録

Week 2:
  - Decision Engine MVP（小判断1日1回）→ 提案の有用性を記録
  - 行動変容の有無を記録
  - 2週間の振り返りインタビュー
```

### 9.4 検証で確認すべき最重要仮説

```
仮説1: 45軸 + 矛盾 + 三面鏡 → 個人固有の反応パターン予測は、
        ユーザーに「当たっている」と感じさせられるか？

        「この人にしか当てはまらない文」を生成できるか？

仮説3: Decision Engineの小判断シミュレーションは、
        ユーザーの意思決定を実際に変えるか？

仮説4: Self vs Oracleの毎日のループは、
        ユーザーが自発的に続けたくなるか？
```

## 11. 追加採用コンセプト

議論の中で「追加機能として採用」が決まったもの：

### 11.1 心理的双子の発見（Ghost Resonance 拡張）

深層の処理構造が酷似している「心理的双子」をStargazerユーザー間で匿名検出。
表面的な趣味や性格ではなく、矛盾パターン・防衛パターン・後悔パターンの構造的類似性で検出。

### 11.2 未使用能力の起動（Self Unlock）

個人モデルから「使われていない能力」を特定し、起動するための提案を行う。
例: 「あなたは分析力が高いが、直感軸が抑制されている。直感を使える条件は[X]。」

### 11.3 自己観測能力の訓練

Self vs Oracle を通じて「自分を見る力」をスキルとして育てる。
SAS の向上 = 自己認識精度の向上 = アプリ外でも自分のパターンに気づける能力の獲得。

### 11.4 複数の自分の可視化

状態・文脈依存で異なる「自分」が主導権を握っていることを、データで実証・可視化。
例: 「昨日の朝は『慎重な計画者』、夜は『衝動的な探索者』が主導していた。」

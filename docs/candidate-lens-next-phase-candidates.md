# Candidate Lens — 次フェーズ候補（A / B / C）整理

> ステータス: **計画整理のみ**。実装しない。CEO が次に進める方向を選ぶための比較資料。
> 共通制約: dev-only・flag default OFF・production hard block・honesty 厳守・段階 GO。

---

## 候補A: P4-e production readiness → dev-only dogfood 継続

- **内容**: P4-e（`docs/candidate-lens-phase4e-production-readiness-plan.md`）のチェックリストを順に満たし、満たすまでは **dev-only dogfood を継続**。本番有効化は GO 条件充足後。
- **やること**: GCP key restriction / quota / budget alert / kill switch（CEO・Cloud Console）＋ 本番許可 env gate 改修（実装・別 GO）＋ dogfood 観測。
- **リスク**: 低（既存実装の安全な前進・課金は招待制で実質 $0〜数ドル）。
- **得られるもの**: 実ユーザーが ②③ で実写真/営業時間を見られる（世界観の解像度↑）。フィードバックで honesty/UX を検証。
- **推奨度**: ★ 最有力。既に実装が揃っており、価値が出る最短路。新規リスクが最小。

## 候補B: P5 Candidate Ranking Personalization（**計画のみ**）

> ★候補 ranking に preference を反映するのは危険。**まず計画のみ**。P3-c の「行順だけ」から ranking へ進む条件を整理する。

- **現状の境界**: P3-c は preference を **③ 比較表の表示行順だけ**に反映。**候補の順位・pager 順・✨おすすめ side・recommendation score は不変**（canonical 固定）。観測（P3-a/b）は opaque key の shadow 記録のみ。
- **ranking へ進む = 何が変わるか**: ユーザーごとに ① で見せる候補の**並び**や**どれを上位に出すか**を preference で変える。これは「ユーザーの選択を誘導する」強い介入。
- **危険性（なぜ慎重か）**:
  1. **フィルターバブル/自己強化ループ**: 過去選好を上位に出す→さらに選ぶ→偏りが固定。多様性を失い「第二の自己」の発見機能を毀損。
  2. **honesty 逸脱リスク**: 並び替えの理由が不透明だと「なぜこれが上なのか」が説明できない（捏造的順位に見える）。
  3. **薄い観測での過剰断定**: P3-c の gate（lens5/global8/axis3）はあるが、ranking は行順より影響が大きく、より厳しい gate が要る。
  4. **可逆性**: 行順は無害だが、順位は「見られなかった候補」を生む（提示機会の不平等）。
- **ranking へ進むための条件（満たさなければ進まない）**:
  - [ ] 「いつ並び替えたか」を**ユーザーに説明できる**設計（候補C の explanation 層が前提）。
  - [ ] **多様性ガード**: 上位 N に必ず非選好系も混ぜる（探索枠）。バブル化を構造的に防ぐ。
  - [ ] **強い sufficient-gate**（行順より厳しい観測量・時間幅・矛盾チェック）。
  - [ ] **opt-out / リセット**（ユーザーが「並び替えをやめる」を選べる）。
  - [ ] **A/B で行動変容と納得感を計測**（精度ではなく「自己理解が深まったか」基準）。
  - [ ] recommendation/winner の canonical 不変性は維持（順位とハイライト判定を混ぜない）。
- **段階案（全て計画→別 GO）**: P5-0 設計（境界・gate・多様性ガード）→ P5-a pure ranking 関数（preference→順位・テストのみ・UI 非接続）→ P5-b shadow（順位を計算するが**表示は変えない**・差分を観測）→ P5-c 限定反映（探索枠付き・説明付き・dev-only）。
- **推奨度**: ☆ 計画のみ着手可。**実装は候補C（explanation）の後**が安全。

## 候補C: UI explanation layer（「あなたがよく見る順」等を出すか）

- **問い**: preference を UI に**説明として出す**か（例「あなたがよく選ぶ傾向: 駅近」「あなたがよく見る順」）。出す/出さないを設計する。
- **出す場合の文言（案・honest・過剰断定を避ける）**:
  - ✅ 「最近の選び方から、**徒歩の近さ**を上に並べています」（観測事実＋可逆な行為の説明）
  - ✅ 「並び替えをやめる」リンクを併設（コントロール感）
  - ❌ 「あなたは駅近を好む人です」（断定・人格ラベリング＝不気味）
  - ❌ 「あなたがよく見る順」（監視されている感・追跡的）
  - 原則: **行為の説明（何を/なぜ並べたか）に留め、人格の断定をしない**。仮説は仮説として（「〜のようです」）。
- **出さない場合の理由**:
  - 説明が薄い/不正確だと逆に不信。観測量が少ない初期は「黙って行順だけ」の方が安全（現状 P3-c）。
  - 「見られている」感（追跡的 UI）は Aneurasync の世界観（安心・発見）と相反。
- **不気味さ/過剰断定を避ける設計指針**:
  1. **可逆な行為の説明のみ**（「並べた」は説明する、「あなたは〜な人」は言わない）。
  2. **仮説トーン固定**（断定しない・LENS_SURFACE_HINTS の「兆候あり」1行思想と整合）。
  3. **コントロール権**（やめる/リセットを常設）。
  4. **sufficient な時だけ出す**（薄い観測では出さない）。
  5. **追跡語を避ける**（「よく見る」「履歴」より「最近の選び方から」）。
- **依存関係**: 候補B（ranking）を入れるなら**説明層はほぼ必須**（順位変更の透明性）。行順だけの現状でも、軽い説明を足すかは独立に判断可。
- **推奨度**: ☆ B の前提として計画価値が高い。単独でも「行順の軽い説明」から小さく試せる。

---

## 推奨順序（提案）
1. **A（P4-e readiness → dogfood 継続）** を主線に。実装が揃い価値が出る最短・最低リスク。
2. 並行して **C の設計**（explanation の文言・出す/出さない方針）を固める＝honesty の世界観整備。
3. **B（ranking）は計画のみ**先行し、C の説明層と多様性ガードが揃うまで**実装しない**。

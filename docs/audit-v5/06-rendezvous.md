# Rendezvous（マッチング / ソーシャル）改善パック

## この機能を改善させるための依頼文

```
以下はAneurasyncのRendezvous（マッチング/ソーシャル）機能に関する独立監査パックです。
- この機能だけを対象にしてください
- 他機能との依存は考慮してください（Stargazerの45軸データ、Genome Card交換、Counselor等）
- 全問題点を洗い出してください
- 全改善案を出してください
- 必ず反証してください
- 無難な改善ではなく、世界トップを超える案まで出してください
- 対象: app/(immersive)/rendezvous/(46ファイル), app/api/rendezvous/(123ルート), lib/rendezvous/(138ファイル), components/rendezvous/
```

---

## 1. この機能の役割

Stargazerの45軸観測データを活用したマッチング機能。「外見スワイプ」ではなく「内面の深層マッチング」を志向する。3枠分離設計: 恋愛（スワイプ+L2ゲート+身元確認）/ つながり（アバター先行+友達/コミュニティ/ビジネス）/ パートナー（結婚相談所仕様）。「観測→関係→言語化→再観測」ループの一部として位置づけられ、マッチングアプリではなく「人間同士の可能性を照らすインフラ」を目指す。

片想い完全非表示、遅延通知(3h-24h)、追跡的情報非表示が絶対原則。AI Counselorが関係構築を支援する層として実装済み。

- **APIルート**: 123個（app/api/rendezvous/）— プロジェクト内最多。全369ルート中33%を占有

- **Cronジョブ**: 3個（notification-dispatch, candidate-generation, anima-generation）
- **マッチングアルゴリズム8種**:
  1. seasonal — 季節性重み付け
  2. session — セッション行動ベース
  3. narrativePhase — ナラティブ段階マッチ
  4. livingMatchEvolution — 生活パターン進化
  5. temporal — 時間帯適合
  6. growthEdge — 成長可能性
  7. similarity-complementarity matrix — 類似・補完マトリクス
  8. partnerScoring — パートナー総合スコア
- **リアルタイム機能**: WebRTCシグナリング（webrtcSignaling.ts）、リアルタイムチャット（realtimeChat.ts）、コンステレーション（constellationEngine.ts）
- **感覚デザイン**: サウンド（soundDesign.ts）、ハプティクス（haptics.ts）
- **3枠設計**: 恋愛（スワイプ+L2ゲート）/ つながり（アバター先行+友達/コミュニティ/ビジネス）/ パートナー（独立枠）
- **マッチング入力**: Stargazer 45軸 → 10次元MatchingVector → カテゴリ別スコアリング
- **AI Counselor**: 6テーブル + RPC + CHECK拡張実装済み
- **Phase 0**（既知ペア検証）実装開始中
- **現在のフェーズ**: 価値仮説検証フェーズ — 機能追加ではなく「刺さるか・続くか・広がるか」の観測に集中

## 5. 全問題点

1. **123 APIルートはスタンドアロンアプリ級の規模**。Aneurasync全体の369ルート中33%がRendezvous。保守・テスト・ドキュメントのコストが集中
2. **8種マッチングアルゴリズムはプールサイズが小さいβフェーズでは全て有効に機能しない**。季節性重み付けやナラティブ段階マッチはN=50未満では統計的に無意味
3. **WebRTC/サウンド/ハプティクスは利用率未計測のリッチ機能**。実装コストに対するユーザー価値が未検証
4. **123 APIルートのうちフロントエンドから実際に呼ばれるルート数が不明**。Dead APIが含まれる可能性
5. **3枠設計のUI切替がユーザーにとって理解しやすいか未検証**。恋愛/つながり/パートナーの違いが直感的か
6. **価値仮説検証フェーズなのに機能規模が最大級**。「検証に必要な最小限」を超えている可能性が高い
7. **恋愛枠のL2ゲート（身元確認）は実装済みだがテスト運用の実績が不明**。身元確認フローの離脱率データなし
8. **AI Counselorの利用率/有効性が未計測**。6テーブル+RPC構築済みだが実際のカウンセリング件数ゼロの可能性
9. **候補者生成Cronの実行頻度と生成品質が不明**。候補者リストの鮮度と精度の関係
10. **constellationEngine（コンステレーション）の目的と利用箇所が不明瞭**。UXフローのどこで使われるか
11. **片想い完全非表示の実装がRLS/APIの両層で完全に保証されているか未監査**。情報漏洩リスク
12. **遅延通知(3h-24h)のユーザー体験への影響が未計測**。「通知が来ない」と離脱するリスク

## 6. 全改善案

**A. APIルート棚卸し**: 123 APIルートのフロントエンド参照率を計測。未使用ルートに「未参照」ラベルを付与し、3ヶ月未使用で凍結候補とする

**B. アルゴリズム段階的有効化**: βフェーズではsimilarity-complementarity matrix + partnerScoringの2本に絞る。プールサイズ閾値（N=100, N=500, N=1000）ごとに追加アルゴリズムを有効化するロードマップを定義

**C. リッチ機能利用率計測**: WebRTC接続率、サウンド再生率、ハプティクスイベント数を計測するミドルウェア追加。β期間中のデータで維持/凍結を判断

**D. 3枠UIユーザビリティテスト**: βテスターに構造化インタビュー実施。「恋愛/つながり/パートナーの違いを説明してください」と質問し理解度を計測

**E. AI Counselor利用率の計測追加**: counselor_sessionsテーブルのCOUNT/ユニークユーザー数/セッション長をダッシュボード表示

**F. Phase 0精度KPI設定**: 既知ペア検証の7指標（相互理解度、関係性インサイト精度、予測的中率等）に目標値を定義

**G. 片想い非表示の全層監査**: RLSポリシー + API応答 + フロントエンドフィルタリングの3層で情報漏洩がないことを検証

**H. 遅延通知の離脱影響計測**: 通知遅延時間と次回ログインまでの時間の相関を分析

**I. Counselor品質スコア導入**: カウンセリング後のユーザー満足度（5段階）を収集し、Counselorプロンプトの改善サイクルを構築

**J. マッチングスコアの説明可能性向上**: 「なぜこの人が推薦されたか」を3行で説明する機能。45軸のうちマッチに最も寄与した軸をハイライト

## 7. 改善案への反証

**A反証**: ルート凍結は将来再利用時のコスト増大。APIは存在しても呼ばれなければランタイムコストゼロ。棚卸し自体に工数がかかる

**B反証**: 1-2本に絞るとマッチング品質が下がる可能性。複数アルゴリズムの組み合わせ（アンサンブル）が品質を担保している設計思想がある

**C反証**: βフェーズでこそリッチ体験で差別化すべき。機能削除は「普通のマッチングアプリ」化のリスク

**D反証**: βテスター数人（3-5人）でのUIテストはサンプル不足。統計的有意性なし

**G反証**: RLSポリシーは既にCEO承認済みで実装済み。再監査は重複作業

**H反証**: 遅延通知は「追跡防止」の安全設計。離脱リスクを理由に短縮すると安全原則が崩れる

**J反証**: 説明可能性は「なぜこの人」を考えすぎる体験になり、直感的な出会いを阻害する可能性

---

## 8. 反証後の再修正

**A再修正**: 凍結ではなく「呼び出し元一覧」の棚卸しのみ。未使用は削除ではなくラベル付け（`// STATUS: unreferenced since YYYY-MM-DD`）。工数は自動grep1回で完了可能

**B再修正**: 1本ではなくtop 3に絞る（similarity-complementarity + partnerScoring + growthEdge）。βプールサイズで有効なアルゴリズムの組み合わせを選定。残り5個は`enabled: false`フラグで管理

**D再修正**: 3人でも定性フィードバックは有効。構造化インタビューで3枠理解度を確認。統計的有意性は求めず、「致命的な混乱がないか」の検出に限定

**G再修正**: 全面再監査ではなく、チェックリスト形式でRLS/API/Frontend各層の片想い情報漏洩ポイントを確認。1時間以内で完了可能

**H再修正**: 遅延時間の短縮ではなく、「通知が来るまでの間に別のコンテンツ（Stargazer観測等）に誘導する」設計

**J再修正**: 全ユーザーへの説明ではなく、「なぜマッチしたか知りたい」ボタンをオプションで提供。デフォルトは非表示

## 13. 削るもの / 残すもの / 強化するもの / 統合するもの

### 削る（凍結扱い）
- β期間中に有効に機能しないアルゴリズム（プールサイズ閾値未達のもの）
- 利用率ゼロが判明したリッチ機能（WebRTC/サウンド/ハプティクス — 計測後に判断）

### 残す
- 3枠分離設計（恋愛/つながり/パートナー）
- 片想い完全非表示原則
- 遅延通知（3h-24h）
- AI Counselor基盤（6テーブル+RPC）
- 10次元MatchingVector設計

### 強化
- Phase 0精度検証（既知ペア→定量証明）
- 計測基盤（API参照マップ、Counselor利用率、アルゴリズム貢献度）
- βテスターフィードバック収集の仕組み

### 統合
- なし（価値仮説検証フェーズのため構造変更は控える）

## 14. この機能が世界トップを超えるための最終条件

Tinder/Hingeは「プロフィール情報+行動データ」のマッチング。Aneurasyncが超えるための条件は3つ。

**条件1: 45軸深層データによるマッチング精度が外見スワイプを圧倒すること**
Phase 0の既知ペア検証で定量的に証明する。「Aneurasyncが推薦したペアの関係継続率」が「外見ベース選択の関係継続率」を有意に上回るデータを取得する。これが証明できなければ、8アルゴリズムも10次元Vectorも無意味。

**条件2: AI Counselorが関係構築を支援し離脱率を下げること**
マッチング後の「会話が続かない」問題はTinder/Hinge共通の課題。Counselorが「相手のStargazerデータに基づく会話のきっかけ」を提供し、初回会話の継続率を改善する。計測対象: Counselor利用ペアの会話継続率 vs 未利用ペア。

**条件3: 「マッチングアプリ」ではなく「人間同士の可能性を照らすインフラ」としてユーザーに認知されること**
Bumbleの3モード（恋愛/友達/ビジネス）と表面的には似ているが、Aneurasyncの3枠は「観測データに基づく深層マッチ」が共通基盤。ユーザーが「ここでは自分の深い部分で繋がれる」と感じるかどうか。βフェーズの定性フィードバックで検証する。

**最終的な判断基盤**: βフェーズの検証結果が全ての判断を決める。機能の豊富さではなく、検証結果の明確さが勝負を分ける。

# Stargazer（観測エンジン）改善パック

## この機能を改善させるための依頼文
```
以下はAneurasyncのStargazer（観測エンジン）に関する独立監査パックです。
- この機能だけを対象にしてください
- 他機能との依存は考慮してください（Alter, Origin, Rendezvous, Home画面のContextReel等）
- 全問題点を洗い出してください
- 全改善案を出してください
- 必ず反証してください
- 無難な改善ではなく、世界トップを超える案まで出してください
- 対象ディレクトリ: lib/stargazer/(201ファイル), app/(immersive)/stargazer/, app/api/stargazer/
```

## 1. この機能の役割
Aneurasyncの核。ユーザーの性格・判断特性・行動パターンを45軸で継続的に観測し、ベイズ更新で精度を上げていくエンジン。全機能（Alter, Origin, Rendezvous, Calendar, Genome Card）のデータ基盤。「1回完結の性格診断」ではなく「継続観測による自己理解の深化」を実現する。

## 2. 現状の事実
- lib/stargazer/: 201ファイル（プロジェクト最大）
- 45軸ベイズモデル: 共役ガウス更新（bayesianAxisUpdater.ts）
- 回答時間シグナル: responseTimeEngine.ts
- 矛盾検出4種: contradictionEngine.ts（temporal, cross-axis, self-report vs behavior, stated vs chosen）
- 状態重み付け: stateWeighting.ts
- IRT弁別力: itemDiscrimination.ts
- レイヤー相互作用: layerInteractions.ts
- 重みキャリブレーション: weightCalibration.ts
- 質問システム: 19モジュール。適応的選択4系統（adaptiveQuestionPool, adaptiveQ2, intraSessionAdapter, lensDiscovery）
- 24アーキタイプ: Cognition(3) x Emotion(2) x Social(2) x Execution(2)。重み減衰: weight=1/(1+obsCount*0.08)
- ストリーク5段階: observer(3d)→seeker(7d)→introspector(14d)→contradiction_witness(21d,矛盾3+)→abyss_traveler(30d,矛盾3+,品質0.7+)
- フリーズ保護: 最大2回、7日ごとに1回獲得
- 品質スコア: 質問数(0-0.25)+矛盾検出(0-0.20)+軸カバレッジ(0-0.25)+回答深度(0-0.15)+回答変更(0-0.15)
- Stargazerページ: app/(immersive)/stargazer/ にサブページ10個

## 3. 世界トップ比較
- MBTI公式: 4軸16タイプ、固定質問、1回完結、適応なし

- Enneagram: 9タイプ、固定質問、1回完結

## 5. 全問題点
1. 201ファイルの規模に対してユーザー面ページは10個。エンジンの深さとUI表出のギャップ
2. ストリークの「アンロック」表現が「No identity locks」設計原則（全identity要素は最初からアクセス可能）と矛盾する可能性
3. 201ファイルのうち、実際にAPIやUIから参照されていないモジュール（dead code）が含まれる可能性
4. 適応的質問選択が4系統あるが、どの系統がどの場面で使われるかの整理が不明瞭
5. 30日到達（abyss_traveler）で全機能解放。betaテスターが30日到達するまでのテスト期間が長い
6. 品質スコアの各重み（0.25, 0.20, 0.25, 0.15, 0.15）の根拠が不明（経験的チューニングか学術的根拠か）
7. サブページ10個（predictions, admin, values, transform, wound, events, flexibility, simulation, dreams, rhythm）のうち、ユーザーがどのページに最もアクセスするかの計測がない
8. アーキタイプ重み減衰（weight=1/(1+obsCount*0.08)）により、初期アーキタイプの影響が急速に低下。50回観測で重み0.2。意図的だが低下速度の最適化が必要

## 6. 全改善案
A. dead code監査: 201ファイルの依存グラフ作成、未参照モジュール特定
B. ストリーク表現の修正: 「アンロック」→「精度向上の可視化」（UIコピー変更）
C. 適応的質問選択の系統整理: 4系統の使い分けルール明文化
D. ユーザー面ページの拡充: エンジンの深さを体験に反映する新ビュー
E. サブページ利用率の計測追加
F. 品質スコア重みの根拠ドキュメント作成
G. 初回セッションでの暫定アーキタイプ表示（5問回答で仮算出→継続動機）
H. Stargazer→他機能パイプラインの可視化（どのデータがAlter/Origin/Rendezvous/Calendarに流れるか）

## 7. 改善案への反証
A反証: dead codeであっても将来使用予定のモジュールを削除すると再実装コスト発生
B反証: ストリークの「アンロック」感がゲーミフィケーションとして継続動機になっている可能性
D反証: ページ追加は開発コスト。既存10ページの改善が先
G反証: 5問での暫定アーキタイプは精度が低い。誤った自己認識を与えるリスク

## 8. 反証後の再修正
A再修正: 削除ではなく「使用中/将来予定/不明」の3分類でラベル付け
B再修正: 「アンロック」と「精度向上」の両方を示す表現（「3日観測で精度が上がり、パターン表示が解放されました」）
G再修正: 暫定ではなく「仮の傾向」と明示（「まだ5問なので参考ですが、あなたの傾向は...」）。誤認リスクを明示して対処

## 9. 確定改善
1. 201ファイルの依存グラフ作成（dead code特定の第一歩）
2. サブページ10個のアクセス計測追加
3. 適応的質問選択4系統の使い分けルール明文化ドキュメント作成

## 10. 要検証改善
1. ストリークUIコピーの現状確認（「アンロック」的か「精度向上」的か）
2. 品質スコア重みの最適性（観測データと突合）
3. アーキタイプ重み減衰速度の適切性（50回観測で0.2は妥当か）
4. 初回5問での暫定アーキタイプ精度

## 11. 要判断改善
1. 初回セッション内Aha Moment（暫定アーキタイプ表示）のリスク許容
2. dead code削除のタイミング
3. ユーザー面ページ追加の優先度

- 暫定アーキタイプ → profile APIの応答構造変更、Home AnswerCardの表示ロジック追加
- サブページ計測 → analytics基盤への依存

## 13. 削るもの / 残すもの / 強化するもの / 統合するもの
- 削る: 確認済みdead code
- 残す: 45軸ベイズモデル、矛盾検出4種、ストリーク5段階、24アーキタイプ
- 強化: UI表出面（10ページの改善）、適応的質問選択の整理、計測基盤
- 統合: 4系統の適応的質問選択を選択フレームワークとして整理

## 14. この機能が世界トップを超えるための最終条件
既に性格診断としては前例のない深度（45軸 x ベイズ x 矛盾検出 x IRT）。超える条件は: (1) エンジンの深さがユーザー体験として体感できること（「MBTIとは全然違う」とユーザーが言語化できる）、(2) 継続観測の価値が3日以内に実感できること、(3) 45軸の精度が学術的ベンチマーク（Big Five再テスト信頼性r=0.8）を超えること。技術的には到達済み。課題はUX表出。

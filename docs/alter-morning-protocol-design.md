# Alter Morning Protocol — 設計書

> **ステータス**: CEO承認済み（条件付きGO, 2026-04-12）
> **目的**: Alter画面を統合ハブ化し、Todo/予定管理・コーデ提案をAlter主導の会話で完結させる
> **原則**: 既存Alterロジックは変えない。各機能のロジックを組み合わせて強化する

---

## 1. 設計思想

### 1.1 Alterの役割の再定義

| 層 | 役割 | 例 |
|---|---|---|
| **観測層**（Alter主導） | 気づき・パターン・先読みを伝える | 「3日連続夜型だね」「今日は冷えるよ」 |
| **計画層**（ユーザー起動） | 1日の予定・Todoを一緒に組む | Morning Protocol |
| **判断層**（ユーザー起動） | 聞かれたら判断エンジンで応答 | 既存の判断エンジン |
| **実行層**（インラインUI） | 会話の流れの中でデータ入出力 | プラン表示・コーデ表示 |

### 1.2 基本方針

- **入口はAlter、詳細編集は各機能ページ**
- Origin・コーデカレンダーを消さない。Alterの下位システムにする
- Alterが各機能の意味をユーザーに返す（機能を見せるのではなく）
- 「便利なAIアシスタント」ではなく「誰よりも自分を知っている存在」として統合
- 会話で全部は背負わない。視覚が必要な場面ではインラインUIを使う

### 1.3 差別化ポイント

- ChatGPT/Geminiとの差別化は「何ができるか」ではなく「どれだけ深くユーザーを知っているか」
- パーソナライズ学習（タスク所要時間、行動パターン、曜日傾向）
- Phase 3で真価を発揮: 過去パターンから「月曜は集中力が落ちる」等の洞察を加える

## 2. Morning Protocol — 会話フロー

### 2.1 可変ラリー設計

**固定ラリー数は使わない。** ユーザーの入力密度に応じて適応する。

- **最短2ラリー**: ユーザーが具体的に書いた場合 → すぐプラン提示
- **通常3ラリー**: 情報やや不足 → 1問で足りないものを聞く → プラン提示
- **例外4ラリー**: 大幅に不足 → もう1問追加
- **planning解除**: 雑談に寄せた場合 → 通常Alterフローへ

### 2.2 Sufficiency Gate（情報充足判定）

ユーザー入力を受けて、以下の情報が推定可能かチェック:

```
必須情報:
├── 予定/Todoが1つ以上あるか
├── 移動手段が推定可能か（→ walkNeed, 歩き量）
├── 場所がわかるか（indoor / outdoor / mixed）
├── 雰囲気が推定可能か（→ formality, attention等）
└── 誰と会うかわかるか（予定文面から自動推定、不明なら聞く）

判定:
├── 全て推定可能 → 2ラリーで確定
├── 1-2項目不足 → 足りないものだけ1問にまとめて聞く（3ラリー）
├── 3項目以上不足 → 2問に分けて聞く（4ラリー・例外）
└── 予定/Todoなし + 雑談的 → planning解除
```

### 2.3 フロー詳細

#### Rally 1 — Alter開始

```
Alter: 「おはよう。今日はどんな1日にする？
        やりたいこと、決まってる予定、なんでも教えて」
```

**分岐:**
- ユーザーが内容を出す → Rally 2へ
- 「特にない」「作らない」等 →

```
Alter: 「了解。何か一緒に考える？ それとも、ちょっと話す？」
  [一緒に考える] → 「じゃあ、やりたいこと何かある？」→ 同じフロー
  [話す]         → Alter質問フロー（既存Proactive Engine / greeting応答）
```

#### Rally 2 — 構造化 + 条件質問

Alterがやること・予定を分類。Sufficiency Gateで不足判定。

**十分な場合（→ 2ラリー完結）:**

そのままプラン提示へ（Rally 3をスキップ）。

**やや不足の場合（→ 3ラリー）:**

```
Alter: 「整理するとこんな感じかな。

        ━━ 予定 ━━
        ・14:00 歯医者
        ・19:00 友人と食事

        ━━ やること ━━
        ・資料作り
        ・買い物
        ・英語の勉強

        あと、今日の過ごし方だけ教えて。
        移動は何が多い？ 室内中心？外も多い？
        服の雰囲気はどんな感じがいい？」
```

1問にまとめて聞く項目:
- 移動手段（→ walkNeed / 歩き量推定）
- 場所（→ venue: indoor/outdoor/mixed）
- 雰囲気（→ formality / attention / romance等に変換）
- 誰と会うかは予定文面から自動推定（「友人と食事」→ friend）。不明なら聞く。

#### Rally 3（or 2） — プラン提示

**Alterが所要時間を仮置きして提示。ユーザーは差分修正のみ。**

```
Alter: 「こんな感じで組んでみたよ。長さ変えたいものある？」

┌─────────────────────────┐
│  ☀️ 今日のプラン          │
│                           │
│  09:00  資料作り (90分)    │  ← タップで時間変更
│  10:30  買い物 (40分)      │
│  11:30  英語 (45分)        │
│  13:00  昼休憩             │
│  14:00  🏥 歯医者 (90分)   │  ← 学習済み: 「前回90分だったから」
│  16:00  フリー             │
│  19:00  🍽️ 友人と食事      │
└─────────────────────────┘
  [これでいく]  [変更する]
```

- 時間はタップで変更可能
- 項目の並び替え可能
- 固定予定（歯医者等）は移動不可、やることは柔軟配置

#### 複数イベントの比重処理

1日にタイプの異なるイベントが複数ある場合:

```
Alter: 「仕事とデート、今日はどっちが重め？」
  [仕事メイン]  → formality↑, trust↑ 寄りでコーデ
  [デートメイン] → romance↑, attention↑ 寄りでコーデ
  [半々]        → バランス型
```

比重に応じてIntent軸の重み付けを変える。

## 3. コーデ統合

### 3.1 フロー

プラン確定後、自然にコーデへ接続:

```
Alter: 「今日のプラン決まったね。コーデも見る？」
  [見る]
```

裏側の処理:
1. プランの予定 + 条件情報 → EventContext生成
2. EventContext → 20軸Intent変換（既存vcIntent.ts）
3. outfitEngine.generateDayProposal() 呼び出し（既存エンジン）
4. 結果をSlotSwipeLane相当のUIで表示

### 3.2 表示

```
┌──────────────────────┐
│  👒 アクセサリー       │  ← SlotSwipeLane相当
│  [キャップ] ← → [なし]│     横スワイプで候補切替
│                       │
│  🧥 アウター           │
│  [薄手ジャケット]      │
│                       │
│  👕 トップス           │
│  [白カットソー]        │
│                       │
│  👖 ボトムス           │
│  [ネイビーチノ]        │
│                       │
```

### 3.3 服登録不足時

```
Alter: 「組みたいんだけど、まだ手持ちが少し足りない。
        まず3つだけ入れてくれたら、今日の案までは出せるよ」
  [登録する] → My-Styleへ遷移
```

### 3.4 技術接続

- 既存outfitEngine・SYNCスコアリング・vcIntent・vcCandidatesは**そのまま流用**
- Alterは裏でこれらを呼び出すだけ
- コーデカレンダーページも残す（詳細編集はそちら）

## 4. パーソナライズ学習

### 4.1 TaskDurationMemory（タスク所要時間学習）

```typescript
interface TaskDurationMemory {
  patterns: {
    [keyword: string]: {
      lastDuration: number;      // 分
      count: number;             // 設定回数
      avgDuration: number;       // 平均（分）
    }
  }
}
```

- 初回: カテゴリ別デフォルト（仕事系: 60分、買い物: 30分、通院: 60分 等）
- ユーザーが修正するたびに学習
- 2回目以降: 「歯医者、前回は90分で組んでたからそのまま入れといたよ」

### 4.2 Follow-up学習（Phase 3）

- 曜日別の集中力パターン
- タスク種別ごとの見積もり精度（実績 vs 見積もり）
- 「この種のタスクは見積もりの1.5倍かかる傾向」等の洞察

## 5. 日中フォロー

### 5.1 頻度制御ルール

- **1日最大2回まで**
- 前回フォローから最低**3時間**空ける
- ユーザーが**アプリを開いた時のみ**（プッシュ通知ではない）
- 連続でスキップされたら、**その日はフォロー停止**
- 固定予定（歯医者等）の後は1回だけ聞く

### 5.2 フォロー形式

```
Alter: 「歯医者お疲れさま。資料作り、午前中に終わった？」
  [終わった]  [まだ途中]  [やめた]
```

### 5.3 Throttle状態

```typescript
interface FollowUpThrottle {
  dailyFollowUpCount: number;     // 今日何回聞いたか
  lastFollowUpAt: string | null;  // 最後に聞いた時刻
  consecutiveSkips: number;       // 連続スキップ数
}
```

---

## 6. 夜ジャーナル

### 6.1 頻度制御

- 毎日は出さない。**週3-4回程度**
- ユーザーが書いた日を学習（曜日パターン）
- 2回連続で「今日はいい」→ 次は3日空ける
- **プランを作った日は優先的に出す**（振り返りの動機が強い）

### 6.2 フロー

```
Alter: 「今日もお疲れさま。記録残しておく？」
  [書く] → Originジャーナルページへ遷移
  [今日はいい] → 終了
```

### 6.3 学習状態

```typescript
interface JournalPromptState {
  journalDayPattern: number[];    // 曜日別記録頻度 [日,月,火,水,木,金,土]
  consecutiveDeclines: number;    // 連続辞退数
  lastPromptDate: string | null;  // 最後に誘導した日
  planCreatedToday: boolean;      // 今日プランを作ったか
}
```

## 7. 技術アーキテクチャ

### 7.1 新規モジュール

```
lib/alter-morning/
├── morningProtocol.ts        # メインオーケストレーター
├── sufficiencyGate.ts        # 情報充足判定
├── planningEngine.ts         # ユーザー入力 → 構造化 → プラン生成
├── taskDurationMemory.ts     # 所要時間パーソナライズ学習
├── outfitBridge.ts           # プラン → EventContext → outfitEngine接続
├── followUpTracker.ts        # 日中フォロー制御
├── journalPrompt.ts          # 夜ジャーナル誘導制御
└── types.ts                  # 型定義
```

### 7.2 既存モジュールとの接続

```
Alter POST API (route.ts)
  ├── 既存ロジック（判断エンジン・DG・Proactive等）← 変えない
  │
  └── 新規追加
       ├── Morning Protocol Detector
       │     morning系クエリを検出 → planningモードへ
       │
       ├── Planning Engine
       │     ユーザー入力 → 予定/Todo分類 → 構造化 → プラン生成
       │     OrbitTask型でOriginと同期
       │
       ├── Outfit Bridge
       │     プランのEventContext → 20軸Intent変換
       │     → 既存outfitEngine.generateDayProposal()
       │     → SlotSwipeLane用データを返す
       │
       └── Follow-up / Journal Tracker
             プラン進捗追跡、ジャーナル誘導制御
```

### 7.3 データフロー

```
[Alter会話] → morningProtocol
                 ├── sufficiencyGate（情報判定）
                 ├── planningEngine（プラン構造化）
                 │     └── taskDurationMemory（時間仮置き・学習）
                 ├── outfitBridge（コーデ接続）
                 │     └── 既存outfitEngine / vcIntent / syncScoring
                 └── followUpTracker / journalPrompt（頻度制御）

[Origin] ← OrbitTask同期（planningEngineで作ったTodoをOriginにも保存）
[Calendar] ← EventContext同期（Alterで聞いた予定情報をCalendarにも反映）
```

### 7.4 UI コンポーネント（新規）

```
components/home/morning/
├── MorningPlanCard.tsx       # プラン表示（時間変更・並び替え可能）
├── MorningOutfitCard.tsx     # コーデ表示（SlotSwipeLane相当）
├── FollowUpChip.tsx          # 日中フォローUI
└── JournalPromptChip.tsx     # 夜ジャーナル誘導UI
```

## 8. 全体フロー図

```
朝: Alter「おはよう。今日はどんな1日にする？」
  │
  ├→ ユーザーが内容を出す
  │    ↓
  │  Sufficiency Gate
  │    ├→ 十分: すぐ構造化 + 仮置き時間でプラン提示（2ラリー）
  │    ├→ やや不足: 足りない項目だけ1問で聞く（3ラリー）
  │    └→ 大幅不足: 2問に分けて聞く（4ラリー・例外）
  │    ↓
  │  プラン提示（Alter仮置き、差分修正のみ）
  │    ↓
  │  「コーデも見る？」→ SlotSwipeLane相当で表示
  │
  ├→ 「一緒に考える」→ 同じフロー
  │
  ├→ 「作らない」「話す」→ Alter質問フロー（既存）
  │
  └→ 雑談寄り → planning解除、通常Alterフロー

日中: フォロー（最大2回、3h間隔、スキップ学習）

夜: ジャーナル誘導（頻度制御、曜日パターン学習、プラン日優先）
```

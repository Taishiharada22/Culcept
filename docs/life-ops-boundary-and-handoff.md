## 3. 重複しない理由（監査根拠）
- Life Ops の**候補生成（L-3）と横の R2 empty-day は別物**: R2 は「候補を**置く**」汎用、L-3 は「何が**due か**」を category+cycle から**作る**。L-3 の出力が R2 の入力。
- 既存 `lib/plan/reality/candidate-*` は**プラン提案への accept/dismiss/later**用で、生活行動候補ではない。Life Ops 候補は**新しい候補種**として、既存 surfacing/PRM 学習ループに**プラグイン**する（再実装でなく拡張）。
- よって **Life Ops を別セッションで進めて問題なし**。ただし §4 の受け口を使い、R1/R2/R4/R5 を**再実装しない**こと。

## 4. 統合契約（縦 ⇄ 横の seam・両セッションの単一参照）
```
Life Ops セッション（縦）                このセッション（横）
─────────────────────                 ─────────────────────
L-1 カテゴリ + L-2 周期
  → L-3 Candidate Engine
      → LifeOpsCandidate[]  ──────▶  R2 empty-day（配置・時間・3案）
                                       R4 Moment Trigger（発火・内容）
      ◀── 周期/好み/却下採用 ────────  R1 記憶（synthesis）
L-7 カテゴリ別 Permission  ──特殊化──▶  R5 汎用 authority Level
L-6 外部実行 / L-8 UICard               （横は実行・UI を持たない）
```
- **LifeOpsCandidate（提案 v0・実装は Life Ops 側）**: `{ category, dueReason(周期/イベント根拠), suggestedWindow?, placeQuery?, permissionLevel, riskFlags[] }`。
- 横（このセッション）は category/cycle/外部 API を**知らない**。candidate を**汎用に**配置・記憶・トリガするだけ。
- 縦（Life Ops）は配置/記憶/トリガ machinery を**作らない**。横の R2/R4/R1/R5 を呼ぶ。

## Appendix A — Life Ops 原案（CEO 提出・参考文書）

> CEO が「最初の新しいアイデア」として提出した原案。Life Ops セッションはこれを北極星とする。要旨を保持して収録。

### A.1 中核定義
> Aneurasync は、予定を記録するアプリではない。ユーザーの生活・移動・周期・人間関係・目的・状態を観測し、**まだ予定化されていない必要行動を発見**し、最適なタイミング・場所・理由・実行方法まで提案する、**個人の生活管制 OS（Life Ops）**である。
美容院はその中の 1 機能。本体は「人間が人生を回すために必要な行動を**先回りして予定化**する OS」。

### A.2 美容を起点にした具体（Beauty Cycle Engine）
- ユーザーごとに保持: 前回美容院日 / カット・カラー・眉・ネイル・まつ毛・脱毛 周期 / 予算 / よく行く駅 / 移動許容 / 好きな雰囲気 / 苦手な接客 / 指名要否 / 平日夜 or 休日午前 / 直前予約可否 / 新規開拓可否 / いつもの店優先。
- Plan 側を重ねる: 今週どこを通るか / 空き時間 / 翌日早いか / 雨 / 疲労 / デート面接旅行撮影 等イベント近接 / 給料日前 / 予算余裕。
- 提案例: 「前回カットから31日。今週金曜夜に新宿を通るため移動負担の少ない候補を3件。来週火曜に人と会うので今週中に整えると印象維持ラインに合う」。
- 提案は**理由付き3案**: ①いつもの店（守る・失敗リスク低）②近場で楽（移動最小・初回不確実性注意）③攻める（カラー/髪質改善高評価・予算高め・見た目を上げたい時）。

### A.3 予約完了の 4 Phase（実装ルート）
- **Phase 1 提案まで**: 周期記録 → 次回予測 → Plan 空き照合 → 近隣検索（美容室/眉/ネイル/まつ毛）→ 評価/距離/営業/価格/口コミでランキング → 3 案 → ユーザー選択。
- **Phase 2 予約導線まで**: ホットペッパー/Google/公式/LINE/電話 を提示 → 予約ページ遷移 → 予約内容を下書き化（メニュー/日時/指名）→ 完了押下で Plan 反映。
- **Phase 3 半自動予約**: 明示許可 → フォーム入力補助（名前/電話/メニュー/希望日時 自動入力）→ 最終確定だけユーザー。**確認画面必須（個人情報）**。
- **Phase 4 完全予約**: 提携/予約 API 店のみ自動予約 → カレンダー登録 → 出発通知 → キャンセル期限通知 → 施術後満足度回収 → 次回周期更新。

### A.4 絶対に入れる安全設計（自動予約してはいけない）
初回店舗 / 高額メニュー / キャンセル料あり / 個人情報入力あり / クレカ登録あり / 指名あり / 施術長時間 / 遠方 / 外見の大きな変更 → **必ず「この内容で予約してよろしいですか？」**。
将来自動化可: いつもの店・メニュー・担当・価格帯・周期 / キャンセル無料 / 事前許可範囲内。

### A.5 権限設計（Life Ops Permission L0–L5）
| 権限 | 内容 |
|---|---|
| L0 | 美容周期を記録するだけ |
| L1 | そろそろ行く時期を通知 |
| L2 | 候補店舗と時間を提案 |
| L3 | 予約ページまで誘導 |
| L4 | 入力補助・最終確定はユーザー |
| L5 | 許可済み条件内で自動予約 |
→ 最初から L5 を狙わず、**L2〜L4 を圧倒的に気持ちよく**。

### A.6 Life Ops が管理する全カテゴリ
1. **身体・外見メンテ**: 美容院/眉/ネイル/まつ毛/脱毛/整体/マッサージ/歯医者/健診/眼科/コンタクト/薬・サプリ。
2. **生活維持**: スーパー/日用品/洗濯/クリーニング/掃除/ゴミ出し/水米洗剤TP補充/家電フィルタ/整理/不用品処分。
3. **予定前準備**: 旅行荷造り/デート前美容/面接前 服髪資料/手土産/出張チケットホテル/誕生日プレゼント/冠婚葬祭。
4. **お金・契約・事務**: 家賃/クレカ/サブスク/保険/税金/免許/パスポート/役所/書類/銀行証券。
5. **人間関係メンテ**: 誕生日/記念日/家族連絡/友人食事提案/恋人調整/久々連絡/お礼返信謝罪。→ まずは提案・下書き。
6. **成長・仕事・学習**: 勉強/週次レビュー/目標/筋トレ/読書/資格/仕事準備/資料/振り返り。

### A.7 観測 → 推論 → 提案 → 許可 → 実行 → 学習
観測（カレンダー/位置/移動履歴/前回実行日/購入周期/予約履歴/生活パターン/天気/予算/シフト/近接イベント/好み/却下採用履歴）→ 推論（例: 前回美容院32日 + 来週人と会う + 金曜新宿通過 + 土曜午前空き → 金曜18:30 が自然）→ 理由付き提案 → 許可確認 → 権限に応じ実行（予約ページ/入力補助/下書き/カレンダー/リマインド/出発通知/完了後周期更新）。

### A.8 カテゴリ別 実行レベル（最初の既定）
美容院/眉/ネイル=提案＋予約導線 / スーパー=買い物候補＋寄るタイミング / 日用品=補充リマインド＋購入候補 / 歯医者=周期通知＋予約候補 / サブスク=見直し提案のみ / 家賃税金=通知確認のみ / 友人連絡=下書きのみ / プレゼント=候補＋購入導線 / 旅行準備=持ち物・予約・移動の管制。

### A.9 着手優先順位（Life Ops 内）
1. 美容・身体メンテ（周期強・価値明快・予約導線あり）→ 2. 日用品・買い物補充（生活OS感・「今日帰りに買う」）→ 3. 予定前準備（Aneurasync らしさ最大）→ 4. 生活事務（重要だが失敗リスク高・通知中心）→ 5. 人間関係（価値大だが勝手は危険・提案/下書き）。
**MVP は美容院・眉だけで十分**。

### A.10 ChatGPT との差別化（取り込むロジック / ズラす軸）
- ChatGPT（Pulse/Operator/Apps SDK）は「聞かれてから」→「先回り（朝カード・エージェント実行・外部アプリ予約）」へ確実に来る。汎用タスク（店探し/比較/予約ページ/口コミ要約/旅行手配）は食われる。
- Aneurasync の勝ち筋: **聞かれる前に、生活の流れ（予定/位置/移動/周期/人間関係/見た目イベント/疲労/天気/予算）から“いつ・なぜ・どこで整える/やるべきか”を先に出し、許可後に予約/購入/連絡/予定変更まで進め、結果から学習**。
- 取り込むべき Pulse 的ロジック: ①夜間に状態統合 ②朝に重要カード ③反応を学習 ④日中はタイミング別に必要行動 ⑤許可後に実行 ⑥結果記憶で次回精度。
- ただし Aneurasync は **チャット中心にしない**（Plan/List/Timeline/Map/Notification/Life Ops が中心・チャットは補助）。ChatGPT/API は**部品**（口コミ要約/文面生成/フォーム理解）として使い、価値は**「何をいつ提案すべきかの状態管理と行動管制」**に置く。

### A.11 内部構造（原案の命名）
`BeautyProfile` / `BeautyCycle` / `BeautyTrigger` / `SalonCandidate` / `RecommendationReason` / `ReservationIntent` / `ReservationPermission` / `BookingResult`。一般化すると Life Ops の `Category` / `Cycle` / `Trigger` / `Candidate` / `Reason` / `Intent` / `Permission` / `Result`。
朝の管制例: 「今日11:30渋谷・18:00新宿。昼食入れるなら13:10まで。前回美容院31日、金曜新宿通過でその日が移動最小。今日やるべき準備は洗濯・明日の持ち物・コンタクト補充」。

### A.12 Morning Briefing / Moment Trigger（原案の構造）
- **Morning Briefing**: ①今日の成立リスク（出発期限/危険な移動/余白少/天気/疲労/食事抜け）②今日入れると自然な生活行動（移動動線上）③近い予定の準備不足 ④今日の3案（守る/楽/攻める）。
- **Moment Trigger**: 出発前 / 移動中（余白で1タスク消化）/ 予定後（早く終了→近くの用事）/ 夜（睡眠を守る再配置）/ 週末前（残タスクまとめ）/ イベント前（準備不足の検出）。
- 位置づけ: **ChatGPT Pulse = Personalized Morning Intelligence（情報）/ Aneurasync = Personalized Reality Control（行動・予定・生活管制）**。

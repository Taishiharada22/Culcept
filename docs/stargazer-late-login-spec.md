## 0. 概要

Stargazer の初回体験を「先ログイン型」から「後ログイン型」に変更する。
ユーザーはログインなしで初回観測（約64問）を完走し、制限つき結果を見た上で、
「もっと深く知りたい」というタイミングで初めてログインする。

### 設計思想
- ログインは「入口の壁」ではなく「解放の鍵」
- 「自分って、そういう人間だったのか」の体験がログインの前に来るべき
- MBTIの成功に学び、没入→カタルシスの流れを守る

## 1. 保存・データ永続化仕様（後ログイン対応への改修）

> **基本原則**: サーバー上の観測レコード（`stargazer_observations`）が唯一の正であり、クライアント表示は常にそれに従う。

### 1.0 初回観測の総問数（正式定義）

| 区分 | 問数 | 定義ファイル | 備考 |
|---|---|---|---|
| コア質問 | 51問 | `lib/stargazer/questions.ts` (q01-q51) | 8チャプター構成、SemanticDifferential形式 |
| 認知適応質問 | 8問 | `lib/stargazer/cognitiveFitQuestions.ts` (cf_q01-cf_q08) | コア質問の10問目/20問目/35問目/45問目の後に各2問挿入 |
| Visual Choice | 5問 | `QuestionFlow.tsx` VISUAL_CHOICE_PAIRS (vc_01-vc_05) | 約10問ごとに分散挿入、画像選択形式 |
| **合計** | **64問** | `QuestionFlow.tsx` line 140-141: `displayTotalQuestions = 51 + 8 + 5` | |

- コア51問と認知8問は**別定義**だが、フロー上は分散挿入されて一連の体験になる
- 分岐問題（CF_BRANCH_POOL 最大6問）は条件付きで追加されうるが、基本フローは64問
- ユーザーから見えるのは「64問の一気通貫体験」

### 1.1 現状の実装

| 項目 | 現状 |
|---|---|
| 認証方式 | メール/パスワードのみ |
| Anonymous auth | `enable_anonymous_sign_ins = false`（無効） |
| 回答保存 | 毎問サーバー保存済み（`stargazer_observations`テーブル） |
| localStorage | 日次観測の一時キャッシュとして使用 |
| RLS | 全テーブルで有効（`user_id`ベース） |
| 未認証アクセス | `/login` にリダイレクト |

### 1.2 変更点

#### Anonymous Auth の有効化

```toml
# supabase/config.toml
enable_anonymous_sign_ins = true
```

- オンボーディング開始時に `supabase.auth.signInAnonymously()` を自動実行
- ユーザーは認証画面を見ない。裏で匿名ユーザーが作られる
- 匿名ユーザーも `authenticated` ロールで RLS を通過する
- 既存の `stargazer_observations` テーブルの保存ロジックはそのまま動く

#### ログイン時の昇格処理（2系統）

昇格には2つのケースがあり、それぞれ処理が異なる。

##### ケース1: 新規アカウント作成（昇格）

匿名ユーザーが初めてアカウントを作る場合。最もクリーンなパス。

```
匿名ユーザー → 「アカウント作成」→ 同一 user_id のまま昇格
```

- **OAuth（Google/Apple）**: `supabase.auth.linkIdentity()` で昇格
- **メール/パスワード**: `supabase.auth.updateUser()` でメール追加 → 確認後にパスワード設定
- **結果**: user_id が変わらないため、観測データの移管は不要
- **累計観測数**: そのまま正確（同一user_idのレコードをカウント）

##### ケース2: 既存アカウントへのログイン（merge）

すでにアカウントを持っているユーザーが、匿名状態で質問に答えてからログインする場合。

```
匿名ユーザー（user_id: A）→ 「既存アカウントでログイン」→ 既存ユーザー（user_id: B）
  → user_id が変わる → 匿名の観測データを既存ユーザーに移管する必要がある
```

- **処理**: merge が必要（匿名 A のデータを既存 B に付け替え）
- **観測数**: 既存アカウントの累計 + 匿名で答えた分を合算
- **競合**: 同一質問に既存も匿名も回答している場合 → **匿名側（最新）を優先**

##### 競合時の優先ルール

| 状況 | 処理 |
|---|---|
| 既存アカウントに回答なし、匿名に回答あり | 匿名の回答を移管 |
| 既存アカウントに回答あり、匿名にも回答あり | **`answered_at` が新しい方を優先**。同時刻で判定不能な場合のみ匿名側を優先 |
| 既存アカウントに回答あり、匿名に回答なし | 既存を維持 |

##### merge処理の実装

```typescript
// merge処理（冪等設計）
async function mergeAnonymousIntoExistingUser(
  existingUserId: string,
  anonymousUserId: string
): Promise<MergeResult> {
  // 0. 冪等チェック: anonymous user が is_merged=true なら何もしない
  // 1. anonymous の observation データを existing に付け替え
  //    - 競合する question_id は anonymous 側（最新）で UPSERT
  // 2. anonymous の profile データで existing を更新
  //    - axis_scores は再計算（全 observation から）
  // 3. 累計観測数を再集計（既存 + 匿名の重複なしカウント）
  // 4. anonymous ユーザーを is_merged=true にマーク（二重実行防止）
  // 5. MergeResult を返す（移管件数、競合件数、最終観測数）
}
```

#### 昇格・mergeできないケースへの対策

匿名ユーザーは以下の場合に「同じアカウントに戻れない」：
- ブラウザデータ（Cookie）を削除した
- 別デバイスからアクセスした
- Safari Private Browsing を終了した
- サインアウトした

**対策：**
1. `localStorage` に匿名セッションの `user_id` を補助保存
2. ログイン時（ケース1・2両方）に「未リンクの匿名データがないか」をサーバー側で照合
3. 一致する匿名データがあれば merge 処理を実行

### 1.3 保存の保証範囲

| シナリオ | 保証レベル | 説明 |
|---|---|---|
| リロード / タブ閉じ | ✅ 保証 | サーバーACK済みの回答まで復元 |
| 同じブラウザで再訪 | ✅ ほぼ保証 | 匿名セッションが生存していれば復元 |
| 別デバイスで再訪 | ❌ 未保証 | 匿名状態では復元不可 |
| Cookie 削除後に再訪 | ⚠️ 条件つき | localStorage の補助キーで照合を試みる |
| ログイン後 | ✅ 完全保証 | 正規ユーザーとして永続化 |

#### UI での表現ルール

- ✅ 言ってよい：「ここまでの記録は自動保存されています」
- ❌ 言ってはいけない：「いつでも確実に続きから再開できます」
- ✅ 正確な表現：「アカウントを作成すると、記録を確実に引き継げます」

### 1.4 累計観測数の正確性

- 観測数 = `stargazer_observations` テーブルのレコード数（サーバー側で計算）
- クライアント側のカウントは使わない（表示用のみ）
- 同一質問への再回答は `question_id` + `user_id` で UPSERT（二重カウントしない）
- ログイン昇格時に観測数の再集計を実行

### 1.5 匿名データの TTL（生存期間）

- 匿名ユーザーの未昇格データ: **30日** で削除対象
- 削除は定期バッチ（週次）で実行
- 削除対象: `auth.users` で `is_anonymous = true` かつ `created_at < now() - 30days`
- 削除前に `stargazer_observations` のカスケード削除を確認

### 1.6 Next.js との注意事項

- 匿名ユーザー利用時、Next.js の静的レンダリングでユーザーメタデータがキャッシュ混線する報告あり
- Stargazer の質問フロー・結果表示は **動的レンダリング** を使用する
- `export const dynamic = 'force-dynamic'` を該当ページに設定

## 2. 初回体験フロー

### 2.1 全体フロー

```
アプリ開始
  ↓
オンボーディング画面
  ↓ 裏で supabase.auth.signInAnonymously() を実行
  ↓
質問開始（約64問：コア51問 + 認知8問 + visual5問）
  ├── 認知×8: コア質問内に分散配置（従来通り）
  ├── visual×5: コア質問内に配置（従来通り）
  ├── 質問間トランジション: 200ms以下
  ├── 深呼吸ポイント: 省略（テンポ優先）
  ├── 途中の気づき開示: なし（没入優先）
  ├── 進行演出: 輪郭が浮かぶ / 星図が埋まる等（意味の開示ではない）
  ├── 毎問サーバー保存（サイレント）
  └── 中断・再開: 可能だが UI は「止まらせない」テンポ設計
  ↓
45軸100%収束 → 「初回観測完了」
  ↓
制限つき結果表示（§3 参照）
  ↓
「もっと深く知りますか？」→ ログイン誘導
  ↓
ログイン → anonymous user を昇格 → 全詳細解放
  ↓
日次質問開始 → 既存軸の精度向上 + 新軸の発見（§4 参照）
```

### 2.2 質問体験の設計原則

MBTIが93問を完走させている事実に学ぶ：

| 原則 | 詳細 |
|---|---|
| テンポ最優先 | 回答→次の質問が200ms以下。確認画面を挟まない |
| 没入を切らない | 途中で「当たり」や「気づき」を出さない |
| 残り問数を見せない | 「あと52問」は心理的に重い。観測深度の抽象演出で代替 |
| 自然な緩急 | 認知質問は速く、visual質問は画像を見せるので自然に間が生まれる |
| 深呼吸なし | テンポを殺さない。画面遷移で小さなリズム変化をつける |

### 2.3 軸数表示

```
観測済み: 38軸
```

- 最初から最後まで「観測済み: N軸」で統一
- 分母（/45 等）は表示しない
- 45軸到達時に「初回観測完了」の演出
- その後も 46, 47... と同じ形式で増え続ける
- 初回観測中の進捗感は、数字ではなく演出（星図・輪郭等）で表現

### 2.4 途中離脱と再開

- 離脱時に特別な UI は出さない（自然に閉じられる）
- 再訪時：「前回の続きから始めますか？」を表示
- 匿名セッションが生存していれば、最後に保存された問の次から再開
- 匿名セッションが失われていた場合：最初から（データは新規匿名ユーザーとして）

### 2.5 途中の「保存表示」

- 10問目あたりで「📌 ここまでの記録は自動保存されています」を小さく1回表示
- ログインボタンには見せない。安心感の提供として
- 30問目あたりで「別の端末からも続けられます → アカウント作成」をさりげなく表示
  - これは補助導線。メインのログイン訴求は結果表示後

## 3. 結果表示仕様

### 3.1 結果の段階構造

| 段階 | タイミング | 内容 |
|---|---|---|
| 演出のみ | 質問中 | 輪郭が浮かぶ、星図が埋まる等。意味の開示はしない |
| 制限つき結果 | 45軸100%収束後（未ログイン） | §3.2 参照 |
| 全詳細 | ログイン後 | §3.3 参照 |

### 3.2 制限つき結果（未ログイン）

**見せるもの:**
- アーキタイプ名（コード + 英名 + 日本語名）
- アーキタイプの引用文
- 短い要約（1-2文）
- 「大切なもの」「動機」「納得のしかた」「行動」「反応パターン」のラベル

```
例:
AVIX
Alchemist
錬金術師

「私は失敗したのではない。うまくいかない方法を一万通り見つけたのだ」— エジソン

感情を燃料に仮説を試し続ける。失敗は実験データ

大切なもの: 分析
動機: 納得のしかた
...
```

**ぼかすもの（存在は見えるが詳細は読めない）:**
- 45軸の一覧（軸名は見えるが、スコアは非表示）
- 矛盾の検出数（「矛盾が2箇所検出されました」とだけ表示）
- 深掘りセクションの存在

**完全にロックするもの:**
- 全45軸の詳細スコアと解説
- 矛盾の内容と二面性の分析
- Layer4以降の深層分析
- 日次更新
- Alter / Oracle / Psyche Signature
- Rendezvous 連携
- 履歴蓄積と変化の追跡

### 3.3 実装上の原則

- **ロック部分の実データをクライアントに送らない**
- サーバー側で未ログイン用レスポンスと、ログイン後用レスポンスを分ける
- CSS の `blur()` で実データを隠すのは **禁止**（スクリーンショットで復元可能）
- ぼかし部分はダミーテキストか省略テキストを表示した上で演出する

### 3.4 ポジティブ・フレーミング

矛盾や二面性は「深さの証拠」として伝える：

- ❌ NG: 「仕事とプライベートで判断基準が矛盾しています」
- ✅ OK: 「場面によって異なる自分を使い分けられる柔軟さがあります」

全てのアーキタイプにポジティブな説明をつける（MBTIの成功に学ぶ）。
ネガティブなアーキタイプは存在しない。

### 3.5 初回結果の最後に表示するメッセージ

```
45軸の観測が完了しました。
これは、いま見えているあなたの輪郭です。
でも、あなたはまだこれで終わりではありません。
毎日の問いを通じて、新しい軸が見つかっていきます。
```

## 4. 軸拡張エンジン

### 4.1 設計思想

- 初回の45軸は「観測の出発点」であり「完成形」ではない
- 毎日の質問で既存軸の精度を上げつつ、新しい軸が発見される
- 矛盾が出たら、そこを掘る質問を出す
- 軸の総数が増えること自体が価値ではなく、「まだ知らない自分が見えた」が価値

### 4.2 軸の内部分類（UI には見せない）

| 分類 | 説明 | archetype への影響 |
|---|---|---|
| コア軸（45軸） | 初回観測で確定する基盤層 | archetype はこの45軸のみで算出 |
| 発見軸 | 日次質問や矛盾検出から見つかる | archetype には影響しない（将来検討） |

- UI では区別しない。すべて「あなたの軸」として等しく表示
- archetype がぐらつかないよう、コア軸と発見軸を内部で明確に分離

### 4.3 新軸発見の条件

新軸は以下の条件で発見される：

1. **既存軸の矛盾から導出**: ユーザーの回答が既存の軸では説明できないパターンを示した場合
2. **文脈依存の分岐**: 特定の場面（仕事/恋愛/金銭等）でのみ現れる傾向を検出した場合
3. **状態依存の発見**: ストレス時、疲労時、高揚時などで判断が変化するパターンを検出した場合

発見例:
- 「締切が迫ったときの判断変化」（ストレス応答軸）
- 「3人以上の場での振る舞い」（集団力学軸）
- 「金銭が絡むときの判断傾向」（経済判断軸）

### 4.4 新軸発見の通知

- **方式**: 次回アクセス時に表示（パターンA）
- プッシュ通知は初期段階では使用しない
- 1軸ごとに大騒ぎしない。近い時期に複数見つかったらまとめる

```
表示例:
新しい軸が見つかりました
「集団の場での判断傾向」
あなたの理解が深まりました
```

- 数字だけでなく、軸の意味を一言つける
- 観測済み軸数が +1 される

### 4.5 発見頻度のカーブ

- 最初の1-2週間: 2-3日に1軸ペース
- 1ヶ月以降: 1-2週間に1軸ペース
- この減速自体が「自分の理解が深まっている」ことの証拠

## 5. 質問リライトガイドライン

### 5.1 原則

**変えること:** 質問文の表現を柔らかく、わかりやすくする
**変えないこと:** 質問の測定内容、方向性、回答方法、選択肢

### 5.2 ガイドライン

| ルール | 例 |
|---|---|
| 1文で完結させる | 長い前置きを削除 |
| 場面で語る | 抽象語→具体的な場面に |
| 中学卒業レベルの漢字 | 難読漢字を避ける |
| 敬語を避ける | 「〜しますか？」→「〜する？」「〜のとき、どうする？」 |
| 子供っぽくしない | カジュアルだが対等なトーン |

### 5.3 リライト例

| Before | After |
|---|---|
| 対立する意見に直面した際、あなたはどのような対応を取りますか？ | 意見がぶつかったとき、どうする？ |
| 不確実な状況において、あなたは行動を起こすことを好みますか？ | 先が読めない状況で、動く方？待つ方？ |
| 集団の中で自分の意見を主張することに抵抗を感じますか？ | みんなの前で、自分の意見を言うのは得意？ |

### 5.4 認知質問・visual質問

- **認知質問×8**: 従来通りの形式と配置を維持
- **visual質問×5**: 従来通りの形式と配置を維持
- これらの表現も柔らかくできる部分があればリライト対象

## 6. シェア・比較機能（将来設計）

### 6.1 シェア設計

- 初回結果画面に「この結果をシェア」ボタン
- シェアされるのは: アーキタイプ名 + 引用 + 短い要約（全詳細ではない）
- シェアリンクから Stargazer を受けられる（友達もすぐ始められる）
- シェアはサーバー上の結果スナップショットへのリンク（クライアント状態に依存しない）

### 6.2 比較機能

- 友達と結果を並べて「ここが違う」が見える
- MBTIの相性表より桁違いに深い比較が可能
- Genome Card / Rendezvous と直結

### 6.3 バイラルループ

```
テスト受ける → 結果を見る → シェアする → 友達が受ける → ...
```

シェアされるのは「結果」ではなく「驚き」：
- 「自分がまさかこう見えてたとは思わなかった」
- 「仕事と恋愛で真逆の判断してたの、初めて言語化された」

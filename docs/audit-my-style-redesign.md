## 1. 現状の致命点 TOP 10

### C1. オンボーディングで写真を追加できない（致命的）

**現状**: `OnboardingWizard.tsx` はフィーチャーカルーセル（3枚）+ デモデータ読み込みボタンのみ。写真撮影もクローゼット登録もなく、「機能紹介スライドショー → デモで遊んでね」という構成。

**なぜ致命的か**: 初回体験で「自分のもの」が1つも入らない。ZOZOTOWNはZOZOMATで5分で足型が入る。WEARは1タップでコーデ保存できる。Aneurasyncはオンボーディング完了時点でクローゼットが空。「毎日使いたい」に到達する前にユーザーが離脱する。

**あるべき姿**: オンボーディング内で最低3着の写真登録を誘導。「クローゼットの一角を撮って → AIが自動分割 → 確認 → 登録完了」の30秒フロー。

---

### C2. 「今日」タブが今日の判断を助けない（致命的）

**現状**: `EngagementHub.tsx` の構成:
1. FormationLine（ステージ進行度ヘッドライン）
2. WeatherOutfitPanel（天気+簡易提案）
3. TodaysMirror（気分ピッカー → 予測）
4. AssertionInsightCard（矛盾から生まれた洞察カルーセル）
5. StargazerInsightPanel（性格アーキタイプ表示）
6. ContradictionDialogue（矛盾探究モーダル）

**なぜ致命的か**: 朝の3分で「今日何着る」を解決したいユーザーに対し、「あなたのステージが進みました」「矛盾が発見されています」「性格タイプはこれです」が並ぶ。**判断ではなく概念が主役**になっている。WeatherOutfitPanelはあるが、Calendar側の `outfitEngine.ts`（季節・厚み・フォーマリティ・ペルソナ・満足度・ローテーション・天気の7軸スコアリング）と比べて遥かに簡易。

**あるべき姿**: 開いた瞬間に「今日の天気 26°C 晴れ / 予定: ミーティング → 今日のおすすめ → 理由 → 別案 → フィードバック」の1画面完結。

### C3. Calendar と My Style で提案エンジンが二重実装（構造欠陥）

**現状**:
- Calendar: `outfitEngine.ts` — 7軸スコアリング + 満足度学習 + ローテーション最適化 + コンボグラフ + 素材×天気マトリクス
- My Style: `weatherOutfit.ts` + `todaysMirror.ts` — 簡易版、学習なし

**問題**: Calendar の方が圧倒的に賢い提案を出せるが、My Style の「今日」タブは自前の簡易エンジンを使っている。ユーザーは同じ「今日何着る」を2箇所で別品質で体験する。

**あるべき姿**: 提案エンジンは Calendar の `outfitEngine` に一本化。My Style の「今日」タブは Calendar エンジンの結果を表示するビューに徹する。

### C5. 44コンポーネントの機能過剰（焦点喪失）

**3タブに対して44コンポーネント**。内訳:
- 日常的に使う主要コンポーネント: ~10個
- 分析ツール系（アナリスト向け）: ~15個
- 装飾・エフェクト系: ~8個
- 基盤・ユーティリティ: ~11個

**問題**: CostPerWearDashboard、ResonanceFeed、StyleJourneyMap、MaterialLiteracyPanel、EcosystemInsightsPanel、ContradictionDialogue、DnaRarityBadge — これらは「作れるから作った」もので、毎日開く理由にならない。画面がパネルの陳列棚になっている。

---

### C6. 空状態が「空っぽです」で終わる（離脱誘発）

**現状の空状態メッセージ** (`EmptyStateCards.tsx`):
- WardrobeEmptyState: 「最初のアイテムを追加」+ デモ読み込み
- SetupsEmptyState: 「2着以上でセットアップ作成可能」
- StylesEmptyState: 「スタイルレーンを選ぶ」
- IdentityEmptyState: 「アイデンティティを発見」+ デモ
- InsightsEmptyState: 「データが増えるとインサイトが増える」

**なぜ問題か**: 空状態はユーザーが最も多く見る画面。「データがないので何もありません」は最悪のメッセージ。ZOZOは空状態でもZOZOMATスキャンを促す。WEARは空でもトレンドコーデを見せる。

**あるべき姿**: 空でも「あなたの天気は26°C晴れ。3着登録すると、明日からここにおすすめが出ます」のように**次の価値を予告**する。

### C7. クローゼットがアイテム管理画面（EC在庫管理UI）

**現状**: `WardrobeOverviewTab` の構成:
1. Self-forming Items（意味不明な見出し）
2. 着回し分析トグル → OutfitIntelligencePanel
3. 素材図鑑トグル → MaterialLiteracyPanel
4. WardrobeTab（アイテム追加フォーム: カテゴリ・素材・ドレープ・季節...20項目以上）
5. CostPerWearDashboard
6. SetupsTab（アイテム組み合わせビルダー）

**なぜ問題か**: ZOZOTOWNの商品一覧ページのような「データ入力画面」になっている。ユーザーが「クローゼットを開ける」体験ではなく「在庫管理システムを操作する」体験。20項目以上のタクソノミー入力（素材ファミリー、ドレープオプション、季節コード...）は、ファッション専門家でないと完遂できない。

**あるべき姿**: 写真を撮る → AIが分類 → 確認するだけ。詳細属性は裏で推論。表は「持ってる服の一覧 → よく着る → あまり着てない → 組み合わせ候補」。

---

### C8. 「わたし」タブが診断結果の陳列（静的プロフィール）

**現状**: `me` タブの構成:
1. StylesTab — 4バケット（core/rare/secret/unexpected）にレーンを手動配置
2. IdentityTab — iam/iseek/ibecome の3モード切替 + ペルソナパネル（ロマンス/フレンド/共創/オービター）
3. InsightsTab — タイムラインスナップショット
4. StyleLogicPanel — 発掘されたルール表示
5. AIInsightPanel — AI生成インサイト3フィールド

**なぜ問題か**: 「自分のスタイルを理解する」ではなく「分析結果を閲覧する」画面。Core/Rare/Secretのバケット分類は設計者の概念であり、ユーザーの問い（「私ってどんなスタイル？」「最近変わった？」）に答えていない。iam/iseek/ibecomeは英語+抽象度が高い。ペルソナパネルの「オービター」は意味不明。

**あるべき姿**: 「あなたの主軸: ミニマル × クリーン（着用データから）→ 最近の変化: 少しカジュアル寄りに → 広げたい方向: ストリート系の小物 → 根拠: 直近2週間でスニーカー着用率が60%に」。育つ自己像。

---

### C9. FABボタンが3アクション（焦点散漫）

**現状**: `FloatingActions.tsx` — 右下FABを開くと3つ:
1. 手動で追加（QuickAddWizard）
2. 写真で追加（PhotoAddWizard）
3. 観測する（ObservationLog）

**なぜ問題か**: FABは「最も重要な1アクション」のためのUIパターン。3つ並べるのはFABの設計原則違反。しかも「観測する」はほとんどのユーザーにとって意味不明。

**あるべき姿**: FABは「📷 追加」の1アクション。タップ → カメラ起動 → 撮影 → AI分類 → 確認 → 完了。手動入力はクローゼットタブ内のセカンダリ導線に。

---

### C10. Rendezvous / Calendar への導線が設計されていない（孤立機能）

**現状**: My Style は自己完結型ページ。Calendar は `/api/my-style/bridge` でワードローブを読み込むが、UI上の行き来は `DayDetailSheet` の「/my-style?tab=closet」リンクのみ。Rendezvous との接続は皆無。

**CEOの意図**: My Style は「自分のショーケース → 他機能への派生元」であり、特にRendezvousへの派生が必要。

**あるべき姿**: My Style のワードローブ・スタイルプロフィールが Rendezvous のアバター・プロフィール表示に直結。Calendar の「今日のコーデ」が My Style の「今日」タブに統合。

## 2. ZOZOTOWN / WEAR 比較 — Jobs to Be Done

### ZOZOが強い仕事

| JTBD | ZOZOの手段 | 強度 |
|------|-----------|------|
| 自分に合うサイズを知りたい | ZOZOMAT（足型3D）/ ZOZOGLASS（顔型AI） | ★★★ |
| 似合う色を知りたい | 似合うラボ（パーソナルカラー×AI） | ★★★ |
| 次に買うべき服を見つけたい | AI推薦 + 1,400万点の商品DB | ★★★ |
| お得に買いたい | セール/クーポン/ツケ払い | ★★★ |

**正面衝突禁止**: ECとしての商品探索・購買最適化。ZOZOの商品DB 1,400万点とLYST買収（$154M）による国際展開に勝てない。

### WEARが強い仕事

| JTBD | WEARの手段 | 強度 |
|------|-----------|------|
| 他人のコーデを参考にしたい | 1,400万件コーデDB + AI着回し提案 | ★★★ |
| 体型が近い人のコーデを見たい | 身長・体型フィルター | ★★ |
| トレンドを把握したい | リアルタイムランキング + インフルエンサー | ★★★ |
| コーデを記録・共有したい | ワンタップ投稿 + SNS連携 | ★★ |

**正面衝突禁止**: ソーシャルコーデ検索。1,400万件の蓄積には勝てない。

### /my-style が独占すべき仕事

| JTBD | なぜ独占可能か | 現状対応度 |
|------|--------------|-----------|
| **今日、手持ちから何着る？** | ZOZO/WEARは「買う」「見る」。「持っている服 × 今日の天気・予定 × 自分の傾向」を統合する存在が皆無 | 20%（簡易版のみ） |
| **自分のスタイルがどう変化しているか知りたい** | グローバルにも空白領域。服装選択の経時変化を観測するサービスは存在しない | 15%（概念はあるがUI未完） |
| **クローゼットの中身を把握したい** | XZ（有料化で離脱）、Whering（海外）以外に日本でプレイヤー不在 | 30%（入力が重すぎる） |
| **手持ちで足りない1点を知りたい** | ZOZOは「買わせたい」、WEARは「見せたい」。「今持っているものから逆算する」のは利益相反でEC側はやれない | 10%（GapAnalysisあるが埋もれている） |
| **気分と装いの関係を理解したい** | 完全空白。心理状態×服装選択の関係性観測は学術領域にしか存在しない | 5%（TodaysMirrorが萌芽） |

### 借りるべきUI原則

| 原則 | 出典 | /my-style への適用 |
|------|------|-------------------|
| **写真1枚で完結** | WEAR投稿フロー | アイテム登録を写真起点に。フォーム20項目は裏に |
| **3秒で今日の情報** | ZOZOトップ | 今日タブは天気+提案が0スクロールで見える |
| **身体特徴で自動分類** | ZOZOMAT/GLASS | パーソナルカラー・体型データを提案に自動反映 |
| **保存は1タップ** | WEAR「着る」ボタン | 着用記録は提案カードのスワイプ/タップで完結 |
| **空でも価値がある** | ZOZOランキング | 空状態でもパーソナルカラーに基づく一般提案を表示 |

## 3. 新しい情報設計（IA）

### タブ責務の再定義

```
今日（☀️ Mirror）
├── 責務: 今日の装い判断を1画面で完結させる
├── 主役: 天気 × 予定 × おすすめコーデ
├── 入力: 着たものを記録（1タップ）
└── 出口: Calendar詳細 / クローゼット

クローゼット（👔 Closet）
├── 責務: 手持ちの把握と活用
├── 主役: 服の一覧 × 着用頻度 × 組み合わせ
├── 入力: 写真で追加（FAB）
└── 出口: 足りない1点 → （将来EC連携）

わたし（🪞 Self）
├── 責務: スタイルの自己理解が育つ
├── 主役: 自分の軸 × 変化 × 根拠
├── 入力: なし（自動学習）
└── 出口: Rendezvous プロフィール / Genome Card
```

### 新しい画面フロー

```
初回起動
├── ① 写真で3着登録（30秒ガイド）
├── ② パーソナルカラー自動取得（body-color連携）
├── ③ 「明日からここにおすすめが出ます」予告
└── → 今日タブへ

毎朝（今日タブ）

├── 予定: 10:00 ミーティング / 15:00 カフェ

毎晩（記録）
├── 今日着たものを確認（朝の記録 or 写真で追加）
├── 満足度（😊😐😞）

週次（わたしタブに通知）

## 4. 要素の仕分け: 残す / 裏に隠す / 消す

### 今は消す（表から完全に外す）

| 要素 | 理由 |
|------|------|
| **FormationLine（ステージ進行表示）** | 「Level 2に進みました」は、ZOZOもWEARも出さない。ゲーミフィケーションが自己目的化 |
| **ResonanceFeed（スタイルツイン検索）** | ユーザーベースがゼロの段階で「匿名ベクター送信→共鳴度表示」は無意味。MAU 1,000超えてから |
| **EcosystemInsightsPanel** | 「他機能との接続状況」は開発者ダッシュボード。ユーザーの問いに答えていない |
| **CrossFeaturePanel** | 同上。裏の接続は裏で動けばよい |
| **DnaRarityBadge** | 「あなたのDNA希少度は72%」— ユーザーにとって何の行動にも繋がらない自己満足メトリクス |
| **StyleJourneyMap（SVGタイムライン）** | データが数週間分ないと意味をなさない。初期ユーザーには空の線グラフ |
| **OnboardingWizard（現行版）** | フィーチャーカルーセル+デモは全廃。写真登録フローに置き換え |
| **SparkleEffect** | 装飾過剰。通知は控えめなトーストで十分 |
| **NetworkStatusBar** | 同期状態はユーザーの関心事ではない。エラー時のみ表示 |
| **IdentityTab の iam/iseek/ibecome 3モード** | 3つのモードを行き来する必要がユーザーにない。「わたしの軸」に統合 |
| **SwipeLearningTab（単体表示）** | スワイプ学習はOriginまたはStargazerの中で行うべき。My Style内で独立させる理由がない |

### 特に厳しく再判定した要素

| 要素 | 判定 | 理由 |
|------|------|------|
| エコシステム | **消す** | ユーザーは「機能間の接続」に興味がない。裏で動けばよい |
| アイデンティティ（3モード） | **統合して簡素化** | iam/iseek/ibecomeは設計者の概念。「わたしの軸」「広げたい方向」に圧縮 |
| クロス機能ブリッジ | **消す** | 開発者向けメタ情報 |
| コンテクスト・ペルソナ | **裏に隠す** | ロマンス/フレンド/共創/オービターは面白いが情報過多。「場面別」として折りたたみ |
| 大きな空状態ボックス | **再設計** | 空でも価値予告する空状態に全面差し替え |
| 下固定の + ボタン（FAB） | **1アクションに絞る** | 📷 1つ。観測・手動入力は消す |
| 0%や未定義表示 | **消す** | 「Style DNA: 0%」「セットアップ: 0」のような数字ゼロ表示は全て非表示に。値が出るまで隠す |

## 5. タブ別 新レイアウト案

### 今日タブ（☀️）— 主役: 判断

```
┌─────────────────────────────┐
│ ☀️ 26°C 晴れ  東京          │  ← 天気ヘッダー（0スクロール）

│ ┌───────────────────────┐   │

│ │  ┌──────┐ ┌──────┐    │   │
│ │  │トップ│ │ボトム│    │   │  ← アイテム分解表示

   今日 = おすすめコーデ / クローゼット = 持ち物一覧 / わたし = 主軸と変化

   📷 だけ。メニュー展開禁止

   0%、0着、未定義は非表示。値が生まれてから表示

10. 入力は最小、推論は最大
    写真1枚 → カテゴリ・色・素材・季節をAI推論 → ユーザーは確認だけ

## 11. Calendar連携の設計方針

### 現状の問題
- Calendar: 高度な `outfitEngine`（7軸スコアリング + 6つの学習エンジン）
- My Style: 簡易な `weatherOutfit.ts` + `todaysMirror.ts`
- 同じ「今日何着る」を2箇所で別品質で提供

### 統合方針

```
提案エンジン: Calendar outfitEngine に一本化
  ↓
My Style「今日」タブ: outfitEngine の結果を表示するビュー
  ↓
Calendar「日別詳細」: 同じ結果 + 天気詳細 + 予定詳細
  ↓
共有データ層:
  - lib/shared/wardrobe.ts（正本）
  - lib/shared/wearEvents.ts（正本）
  - Calendar outfitEngine → My Style が GET で取得
```

My Style は **ワードローブの正本 + スタイルの自己理解** を所有。
Calendar は **提案エンジン + スケジュール統合** を所有。
「今日タブ」は Calendar エンジンの結果を My Style の文脈で表示する。

## 12. Rendezvous 派生の設計方針

### 接続ポイント

1. **わたしタブ → Rendezvous プロフィール**
   - スタイル主軸・DNA可視化がRendezvousアバターの外見描写に直結
   - 「このスタイルで出会いたい」のアピール

2. **クローゼット → Rendezvous 写真**
   - 手持ち服の写真がRendezvousの「雰囲気写真」候補に
   - アバター先行マッチングの「服装の空気感」として反映

3. **着用パターン → Genome Card**
   - 「ミニマル×クリーン系。最近カジュアル方向に変化中」→ カード記載
   - スタイルDNAレーダーがGenome Cardのビジュアル要素に

### 実装順序
- Phase 0（今回）: わたしタブに「Rendezvousで使う →」導線ボタン
- Phase 1: スタイルプロフィールをRendezvous API が参照
- Phase 2: Genome Card にスタイル情報を自動反映

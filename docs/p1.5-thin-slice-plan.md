## 1. State 永続方式

### 問題

HTTPリクエストは独立。route.ts ローカル変数はリクエストをまたいで残らない。
`last_bet` / `last_bet_outcome` / `rejected_bets` / `accepted_bets` がリクエスト間で消失する。

### 解決: analytics 再構成方式

**新テーブルなし、新カラムなし。** 既存の `stargazer_analytics` + `session_id` だけで解決する。

#### 仕組み

各ターンで `home_alter_judgment` event に thin_slice metadata を書き込む（これは元の計画通り）。
次のターンで、同一セッションの直近 analytics を読み出してセッション状態を再構成する。

```typescript
/**
 * セッション内の thin-slice 状態を analytics から再構成。
 * 新テーブル不要。既存 analytics の metadata.thin_slice フィールドから復元。
 */
export async function reconstructThinSliceState(
  supabase: SupabaseClient,
  userId: string,
  sessionId: string,
): Promise<ThinSliceSessionState> {
  const { data: recentEvents } = await supabase
    .from("stargazer_analytics")
    .select("metadata, created_at")
    .eq("user_id", userId)
    .eq("event", "home_alter_judgment")
    .eq("metadata->>session_id", sessionId)
    .order("created_at", { ascending: false })
    .limit(10); // 直近10ターン分で十分

  const state: ThinSliceSessionState = {
    last_bet: null,
    last_bet_outcome: null,
    rejected_bets: [],
    accepted_bets: [],
    bet_history: [],
    consecutive_misses: 0,
  };

  if (!recentEvents || recentEvents.length === 0) return state;

  for (const event of recentEvents) {
    const ts = (event.metadata as any)?.thin_slice;
    if (!ts) continue;

    // 直近の bet を取得
    if (!state.last_bet && ts.bet) {
      state.last_bet = {
        bet: ts.bet,
        bet_type: ts.bet_type,
        confidence: ts.bet_confidence ?? 0.5,
      };
    }

    // bet outcome 履歴を蓄積
    if (ts.bet && ts.previous_bet_outcome) {
      state.bet_history.push({
        bet: ts.bet,
        outcome: ts.previous_bet_outcome,
      });
      if (ts.previous_bet_outcome === "miss") {
        state.rejected_bets.push(ts.bet);
      } else if (ts.previous_bet_outcome === "hit") {
        state.accepted_bets.push(ts.bet);
      }
    }
  }

  // 連続 miss 数をカウント
  for (const h of state.bet_history) {
    if (h.outcome === "miss") state.consecutive_misses++;
    else break;
  }

  return state;
}
```

#### コスト

- **追加クエリ**: 1回/リクエスト（`stargazer_analytics` を session_id で絞り込み、LIMIT 10）
- **レイテンシ**: ~5-20ms（indexed query、小量データ）
- **既存の並列ロード**（L639 `Promise.all`）に追加可能

#### なぜこの方式か

| 方式 | メリット | デメリット |
|------|---------|-----------|
| ❌ ローカル変数 | 実装が楽 | **リクエストをまたがない（致命的）** |
| ❌ 新テーブル | 明示的 | DBスキーマ変更が必要 |
| ❌ Redis/KV | 高速 | インフラ追加が必要 |
| ✅ **analytics 再構成** | 新テーブル不要、既存データ流用 | クエリ1回追加（~10ms） |

analytics はすでに `session_id` を metadata に含んでいる（`route.ts:3182`）。
thin_slice フィールドを同じ metadata に追加するだけで、再構成のソースになる。

## 2. Turn Detector 追加ルール

### 問題

元の判定は `questionType` / `responseMode` / `detectedReaction` / 存在的問いキーワードのみ。
実ラリーで最も痛い「一緒に考えてほしい」「押し付けるな」「具体的に聞いてる」が standard に落ちる。

### 追加: requestIntent / protest / direct-demand パターン

```typescript
// ── co-think insistence: 「一緒に考えてほしい」系 ──
// これが standard に落ちると、Alter が整理・書き出し型に逃げる
const CO_THINK_INSISTENCE = /一緒に考え|一緒に悩|一緒に探[しす]|二人で|僕[はも].*わから|俺[はも].*わから|わからないから聞[いき]て/;

// ── protest: 「押し付けるな」系 ──
// repair mode でなくても、ユーザーの不満が明確な場合
const PROTEST_PATTERNS = /押し付け|決めつけ|上から|偉そう|わかったふう|わかった風|知ったかぶり|勝手に決め|的外れ|ずれてる|ズレてる|見当違い|論点.*ずれ|聞いてない/;

// ── direct-answer demand: 「具体的に聞いてる」系 ──
// knowledge/strategy と重なるが、ユーザーが「AlterがAIに逃げている」と感じている時
const DIRECT_DEMAND_PATTERNS = /具体的に聞いて|抽象的すぎ|ふわっと|曖昧すぎ|はっきり[言い]って|結局どう|で、どうすれば|答えて|答えになってない|答えが.*ない|回答になってない/;

// ── delegation rejection: 「君がやって」系 ──
// Alter が宿題を返そうとしている兆候への直接的な拒否
const DELEGATION_REJECTION = /[君あなたお前]が[やしせ考]|[君あなたお前]の意見|調べるんじゃなく|考えるんじゃなく|[君あなた]に聞[いき]て|投げ返[さす]ないで|丸投げ/;
```

### 更新後の判定ロジック

```
critical（最大知能予算）:
  [維持]
  - responseMode === "repair"
  - questionType === "self_understanding" && conversationHistory.length >= 4
  - message が存在的問い: /本質|根本|結局|本当に.*したい|人生|生き方|自分.*何者/
  [追加]
  - PROTEST_PATTERNS にマッチ（repair mode でなくても抗議は critical）
  - DELEGATION_REJECTION にマッチ（Alter への直接的な行動要求）

elevated（追加知能あり）:
  [維持]
  - questionType === "self_understanding"（初回〜浅いターン）
  - detectedReaction?.type === "disagree"（bet miss からの回復チャンス）
  - questionType === "judgment" && conversationHistory.length >= 3
  - message が core-drive 系: /大事|価値観|譲れない|こだわり|原点|根っこ/
  [追加]
  - CO_THINK_INSISTENCE にマッチ（一緒に考えてほしい要求）
  - DIRECT_DEMAND_PATTERNS にマッチ（具体を求めている）

standard（通常）:
  - 上記以外すべて
```

### 判定の優先順位

```
1. Feature Flag OFF → 全て standard（§4 参照）
2. critical パターン → critical（最初にチェック）
3. elevated パターン → elevated
4. standard（フォールスルー）
```

critical と elevated が同時にマッチした場合は critical が勝つ。
これにより、protest / delegation_rejection は questionType に関係なく critical に昇格する。

### テストシナリオ追加（既存10本 + 新規4本）

| # | シナリオ | 入力例 | 期待 budget |

| 11 | **co-think insistence** | 「わからないから一緒に考えてほしい」 | elevated |

#### 評価 SQL: ラリー抽出用

```sql
-- 人間評価対象: thin-slice 発火セッションの全ターン
WITH target_sessions AS (
  SELECT DISTINCT metadata->>'session_id' AS sid
  FROM stargazer_analytics
  WHERE event = 'home_alter_judgment'
    AND feature = 'home_alter'
    AND metadata->'thin_slice'->>'turn_budget' IN ('elevated', 'critical')
    AND created_at >= NOW() - INTERVAL '7 days'
  ORDER BY sid
  LIMIT 10
)
SELECT
  d.session_id,
  d.role,
  d.content,
  d.created_at,
  a.metadata->'thin_slice' AS thin_slice_meta
FROM stargazer_alter_dialogues d
LEFT JOIN stargazer_analytics a
  ON a.user_id = d.user_id
  AND a.metadata->>'session_id' = d.session_id
  AND a.event = 'home_alter_judgment'
  AND a.created_at BETWEEN d.created_at - INTERVAL '3 seconds' AND d.created_at + INTERVAL '3 seconds'
WHERE d.session_id IN (SELECT sid FROM target_sessions)
ORDER BY d.session_id, d.created_at;
```

## 4. Feature Flag / Canary / Rollback

### 環境変数

```env
# .env.local / Vercel Environment Variables
ALTER_THIN_SLICE_ENABLED=true      # master switch（false = 全ユーザーに thin-slice 不発火）
ALTER_THIN_SLICE_ROLLOUT_PCT=10    # 0-100: rollout 対象ユーザーの割合
```

### 実装

```typescript
/**
 * Feature Flag: thin-slice を発火させるかどうかを判定。
 * - ENABLED=false → 全員 OFF
 * - ENABLED=true → userId の hash が rollout 範囲内なら ON
 *
 * userId ベースの deterministic hash により、
 * 同一ユーザーは常に同じ group に入る（A/B 比較が可能）。
 */
export function isThinSliceEnabled(userId: string): boolean {
  const enabled = process.env.ALTER_THIN_SLICE_ENABLED !== "false";
  if (!enabled) return false;

  const rolloutPct = parseInt(process.env.ALTER_THIN_SLICE_ROLLOUT_PCT ?? "0", 10);
  if (rolloutPct >= 100) return true;
  if (rolloutPct <= 0) return false;

  // deterministic hash: userId → 0-99
  const hash = simpleHash(userId) % 100;
  return hash < rolloutPct;
}

function simpleHash(str: string): number {
  let hash = 0;
  for (let i = 0; i < str.length; i++) {
    const char = str.charCodeAt(i);
    hash = ((hash << 5) - hash) + char;
    hash = hash & hash; // 32bit integer
  }
  return Math.abs(hash);
}
```

### route.ts での使用

```typescript
// ── P1.5 Feature Flag ──
const thinSliceActive = isThinSliceEnabled(userId);

// assessTurnValue は常に呼ぶ（analytics 記録用）が、
// insight / bet / claim は thinSliceActive の場合のみ発火
if (turnValue.invoke_insight && growthState && thinSliceActive) {
  // ... insight + bet + claim
}
```

### Rollout 計画

| Day | rollout | 条件 |
|-----|---------|------|
| 0 | **0%** | デプロイ前にベースライン取得 |
| 1 | **10%** | デプロイ。少数ユーザーで発火を確認 |
| 3 | 10% | 中間チェック: エラーなし + insight 生成成功率 > 80% |
| 5 | **50%** | 中間 PASS の場合のみ拡大 |
| 7 | 50% | フル評価: 5 KPI + 人間評価 |
| 10 | **100%** or **0%** | PASS → 100% / FAIL → 0%（即時ロールバック） |

### Kill Switch

```bash
# 即時無効化（Vercel CLI or Dashboard）
vercel env rm ALTER_THIN_SLICE_ENABLED
# または
vercel env add ALTER_THIN_SLICE_ENABLED false
```

全ユーザーで即座に thin-slice が不発火になる。
既存フローは 100% 無改変なので、kill switch 後は完全に元の挙動に戻る。

### Analytics 記録

thin-slice が OFF のユーザーにも `thin_slice.enabled: false` を記録し、A/B 比較を可能にする:

  turn_budget_reason: turnValue.reason,

## 6. 既存フローから何を残すか

**答え: 全部残す。**

```
[既存フロー — 100% 維持]                         [thin-slice 追加]

                                                 ← feature flag check
                                                 ← reconstructThinSliceState() [state復元]
reaction(P1-C) → correction → greeting →
direct → ambiguity → mode decision
                                                 ← assessTurnValue() [差し込みA]
prompt build (既存テンプレート)
P1-C リアクション注入
                                                 ← if (enabled && elevated/critical):
                                                    generateInsight() + bet + claim [差し込みB]
→ LLM → validate → retry → dedup →
fallback → quality check → creepiness →
                                                 ← evaluateBetOutcome() [差し込みC]
MI lint → analytics                              ← metadata 拡張（thin_slice フィールド）
```

### Feature Flag OFF 時の影響

**完全にゼロ。**
- `isThinSliceEnabled()` が false → assessTurnValue() は実行するが insight/bet/claim は全スキップ
- analytics に `thin_slice.enabled: false` が記録されるだけ
- LLM呼び出し・プロンプト・バリデーション・全既存処理に一切の変更なし

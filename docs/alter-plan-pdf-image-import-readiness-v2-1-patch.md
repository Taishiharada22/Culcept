## §2. Phase 1 の対象再定義（最重要補正）

### 2.1 対象 IN（Phase 1）

**「定期的に配布される表形式の予定ドキュメント」**:
- 会社シフト表（航空/看護/コンビニ/物流/コールセンター/工場 etc.）
- 学校時間割
- 部活・サークル予定表
- 保育園・幼稚園の月予定表
- PTA当番表
- フィットネスクラスSchedule

→ **共通特性**: 表構造あり / 本人行特定可能 / 毎月（または毎週）同形式 / 略号や記号が業務ローカル / 複数人が同一ドキュメント

### 2.2 対象 OUT（Phase 1）

- 自由文 PDF（イベントチラシ / 案内状 / 招待状）
- 単票（病院予約票 / ホテル予約確認）
- スクリーンショット（メール / SNS / web ページ）

→ **Phase 3 以降**で同じ pipeline 上に乗せる。

### 2.3 これにより変わること

- **VLM プロンプト**: 「表形式と仮定して構造抽出」固定で精度跳ね上がり
- **テンプレート学習**が最初から主役（Phase 2 を Phase 1 末から開始）
- **本人行 local crop** が必須（複数人混在前提）
- **目標精度を厳しく設定可能**（個別指標で 95%+ 帯）

## §3. 私が GPT を超える追加観点（独自・⑦革新）

GPT 補正に加えて、私の v2 リサーチ資産から追加する 4 つ:

### 3.1 Golden dataset の 2 重価値

評価データに**正解 JSON + 各イベントの sourceRegion (bbox)** を付ければ:
- **eval data** として精度測定に使用（現在の用途）
- **future learning data** として、テンプレート学習の初期 corpus に使用（将来の用途）
- **prompt engineering の few-shot example** にも使用

→ 1 度作れば 3 つの用途に再利用。**evaluation を product feature data 化する** という構造的革新。

### 3.2 自動凡例 OCR を独立モジュール化

シフト表の凡例部分は表本体より構造が単純（「G = 9:00–17:45」形式の行）:
- 表本体 = dense layout、 cell-level 精度が課題（71-74% TEDS）
- 凡例 = 線形リスト、 100% 近い精度で取れる

→ **凡例抽出を別 module で先に走らせる**、辞書 default を提案。不完全項目（時間帯未記載 HREQ/AL/BD）だけユーザーに尋ねる。**全略号を聞くより摩擦半減**。

### 3.3 本人行 fingerprint アプローチ

GPT「本人行 local crop」を**どう実装するか**まで踏み込み:
- **初回**: ユーザーが画像上**1 クリック**で本人行を指定（PDF viewer 上で）
- 周辺特徴を fingerprint 化:
  - 上下何行目か（top-to-bottom index）
  - 名前文字列の hash
  - 行の bbox（高さ / 縦座標）
  - 行内の最頻略号（初期値）
- **2 回目以降**: fingerprint で auto-match → 直接 crop → VLM へ
- **mismatch**（人員追加で行が動いた等）: 「前回は X 行目でしたが、今月は Y 行目に見えます」確認

→ **「ユーザーに毎回指定させる」(GPT 案最低保証) より、「fingerprint で記憶」(我々の実装)** = 月次反復で UX が累積向上。

### 3.4 Diff モード（Phase 2 候補）

月例の同表を取り込む際、**「前回からの変更点」を強調**:
- 前月の取り込み結果と今月を diff
- 「先月は休みだった日が今月は夜勤に変わっています」
- 「7 連勤が含まれています」
- これはシフトワーカーが**最も気にする情報**

→ シフト表 vertical の真の差別化。Magic Calendar 等にはない。

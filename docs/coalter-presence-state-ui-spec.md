# CoAlter Presence State UI Spec

> v1.1 で定義された CoAlter の存在論・3 モード・S0-S8 を、Figma / コード実装者が迷わず作れる解像度に固定する文書。

## §0 メタ・責務境界

### 0.1 本書の位置づけ

本書は `coalter-core-ux-layered-presence.md`（以下 v1.1）の**実装解像度版**である。v1.1 で定義された存在論・3 モード・S0-S8 状態・発話パターンを、**UI 要素の配置・遷移・直交関係**まで固定する。

本書は v1.1 を前提とし、v1.1 で定義された用語・原則を複製せず参照する。v1.1 の定義と衝突する記述は本書側の誤りとして削除される。

### 0.2 CoAlter ドキュメント群の役割分担

| 文書 | 責務 | 主な問い |
|---|---|---|
| **v1.1**（`coalter-core-ux-layered-presence.md`） | **what / why** | CoAlter とは何か、なぜ通常モードが本体か、S0-S8 は何を意味するか |
| **本書**（`coalter-presence-state-ui-spec.md`） | **how（visual / interaction）** | どこに何が表示されるか、state × mode でどう変わるか、chip は何個まで出せるか |
| **発話テンプレ doc**（未起草） | **how（text）** | 実文面、語彙、禁止表現、tone の具体化 |

**責務違反の判定基準:**
- 本書が v1.1 の定義を再定義していたら削除
- 本書が具体文面を記述していたら削除（→ テンプレ doc へ）
- 本書が pixel 値・Tailwind クラス・実装コードを記述していたら削除

### 0.3 スコープ

**含む:**
- UI 要素の配置・優先順位・サイズカテゴリ
- state × mode の直交関係と表示切替
- chip / action 体系（種別・上限・禁止）
- 状態遷移時の UI アニメカテゴリ
- メインチャットとの境界条件

**含まない:**
- 実装コード（React / Tailwind / 型定義など）
- ピクセル単位の寸法・色コード
- 発話の実文面・語彙
- メインチャット側の内部構造（吹き出し・入力欄・履歴・timestamp）
- `ChatClient.tsx` 等の既存実装詳細への踏み込み

### 0.4 正本分離ルール（A1）

本書では**発話パターン × 状態**の管理を単一正本にせず、以下のルールで分離する。

| 問い | 正本 | 記述場所 |
|---|---|---|
| 発話パターン X はどの状態で出現可能か | **パターン側（Stage 1: existence gate）** | §7 各パターンの `state 許可リスト` |
| (state, mode) で発話パターン X は抑制されるか | **モード側（Stage 2: suppression gate）** | §4 マトリクスの `禁止 action` 欄 |
| 最終可否 | **two-stage gating** | Stage 1 AND Stage 2（§7.12 で正式定義） |

§7 と §4 は同じ種類の正本ではない。**§7 = positive allow**（存在許可）、**§4 = negative override**（抑制・優先度調整）。両者は dual authority ではなく、**異なるレイヤーの直列ゲート**として機能する。

**override の上書き範囲:**

§4 マトリクスの `override` が上書きできるのは、UI 側の**優先表示・抑制・密度**までである。§7 で定義された**発話パターン自体の意味定義・内部構造（ヘッダ／本文／chip 構成）**は上書きしない。つまり §4 は §7 を壊さない。

この分離が守られていない記述（§4 で pattern 内部構造を書く、§7 で mode 制約を再掲する、§4 で pattern の意味定義を再定義する等）は削除対象。

### 0.5 発話トーンの扱い（A2）

本書では**トーンカテゴリタグのみ**扱い、具体文面・語彙・語尾は一切記述しない。

**本書で扱うカテゴリタグ:**
- `calm` — 落ち着いた、短く
- `neutral` — 中立、事実寄り
- `urgent` — 緊急、介入価値が閾値を超えた時のみ
- `retreat` — 退出時、短く締める

これらのカテゴリは UI 側の**配色・密度・アニメ選択**に影響するため本書で管理する。実文面・語彙・語尾は発話テンプレ doc に委譲。

### 0.6 メインチャット非介入ルール

CoAlter はメインチャット（2 人の会話）に常駐しない。本書では以下の境界条件のみ記述する。

**記述する（§2 で詳細化）:**
- z-index（CoAlter layer > メインチャット）
- focus 競合時の優先順位
- scroll 連動の有無
- action 実行後の handoff 境界

**記述しない:**
- メインチャットの吹き出し内部レイアウト
- 入力欄の内部デザイン
- 履歴スクロール構造
- タイムスタンプ / 既読表示 / message status
- reaction / thread reply 等の機能

### 0.7 v1.1 との参照ポリシー

本書は v1.1 の §番号を直接引用し、定義を複製しない。例:

- ❌ 「通常モードは本体である（本書 §X.Y）」← 定義を本書で再主張
- ✅ 「通常モード = 本体（v1.1 §2.3）を前提とし、本書では UI 上の差を固定する」

v1.1 に存在しない概念を本書で新造する場合は、§0.2 の責務境界の範囲内（visual / interaction）であることを必ず確認する。

### 0.8 改訂管理

- v0.1（2026-04-24）: 初版起草開始（§0-§2）

## §1 UI 全体原則

### 1.1 CoAlter は「上部レイヤー」である

v1.1 §3 の通り、CoAlter はメインチャット画面の**上部レイヤー**として存在する。

**原則:**
- 主役は常にメインチャット（2 人の会話）
- CoAlter の発話は上部レイヤーで完結し、メインチャットの吹き出し列に混入しない
- 1 ターン 1 発話（v1.1 §11.4 連投禁止と整合）
- 退出（S8）は常に可能。UI 上でも「いったん戻る」が常設

### 1.2 UI 要素の 3 層分類（v1.1 §9 準拠）

全 UI 要素は以下 3 層のいずれかに分類する。層を増やさない。

| 層 | 定義 | 出現条件 | 例（v1.1 §9 より） |
|---|---|---|---|
| **常設** | S0-S8 全状態で常に表示 | 無条件 | CoAlter シンボル / ステータス / モード切替 / @coalter 導線 |
| **状態依存** | 特定の state でのみ表示 | state による | 発話本文カード（S2/S5/S7）/ 応答チップ（S2/S5）/ 提案導線（S6）等 |
| **チップ** | アクション受付 UI | state と pattern による | 近い / 少し違う / 続けて / たいしに聞く / 提案を聞く / いったん戻る |

新しい要素を追加する場合は、この 3 層のいずれに属するかを明示する。どれにも属さない要素は追加禁止。

### 1.3 状態優先 vs モード優先の原則（A3）

CoAlter の UI 挙動は、通常時と緊急時で優先軸が切り替わる。

**原則:**

> **通常時はモード優先。S2 / S5 では、介入価値が閾値を超える場合に限って状態優先となり、必要ならモード昇格が起こる。**

**意味:**

| 状況 | 優先軸 | UI 挙動 |
|---|---|---|
| 通常動作中（S0-S1, S3-S4, S6-S8） | **モード優先** | 現在のモード（通常 / Daily / Travel）が UI 密度・chip 構成・トーンを決める |
| S2（入口発話）時 | モード優先が基本。ただし介入価値が閾値を超えれば**状態優先に切替可能** | 状態優先時は urgent トーン、chip 数削減、告知なしで発話 |
| S5（橋渡し中）時 | モード優先が基本。ただし介入価値が閾値を超えれば**状態優先に切替可能**。この時**必要ならモード昇格が起こる** | 状態優先時は urgent トーン、通常→Daily/Travel への自動昇格が起こり得る |

**重要:**
- S2 / S5 であっても、介入価値が閾値以下なら通常通りモード優先で振る舞う
- 「S2 / S5 = 必ず状態優先・必ず昇格」ではない
- S2 / S5 は**状態優先が起こりうる代表点**として本書が扱う
- **状態優先は「介入の必要性」を上書きするが、常設要素（§1.2 の常設層）の骨格までは破壊しない**
  - CoAlter シンボル / ステータス / モード切替 / 入力導線 といった常設要素は状態優先切替時も維持される
  - 変わるのは発話トーン・chip 数・アニメ・UI 密度であり、上部レイヤーの骨格そのものではない

**具体化の場所:**
- §4 マトリクス: (S2, 通常) / (S5, 通常) セルに「状態優先切替の条件」を記載
- §6 モード切替: 昇格の閾値判定と UI 表現
- §8 介入 UI: 状態優先切替時の視覚表現

### 1.4 UI 密度の段階

本書では UI 全体の「密度」を以下 3 段階で扱う。§4 マトリクスと §5 状態別 UI で参照される。

| 密度 | 定義 | 出現する状態 |
|---|---|---|
| **single-line** | 1 行のみ表示。CoAlter シンボル + ステータスのみ、カードなし | S0, S8 |
| **compact-card** | 短い発話カード（2-4 行）+ 1-2 チップ | S1, S2, S3, S4, S6 |
| **expanded-card** | 発話カード本文（2-6 行）+ 3 個以上のチップ + 補助導線 | S5, S7 |

### 1.5 アニメカテゴリ

遷移時の UI 変化は以下 4 カテゴリで管理する。具体的な duration / easing は §9 で保留論点として扱う。

| カテゴリ | 用途 | 例 |
|---|---|---|
| **fade** | 要素の出入り（控えめ） | S0 → S1 のチップ出現 |
| **slide-down** | 上部レイヤーの展開 | S1 → S2 のカード展開 |
| **pulse** | 注意喚起（緊急時のみ） | S2/S5 で状態優先切替時 |
| **none** | 遷移なし（即時切替） | S6 内での chip 切替 |

### 1.6 連投抑制の構造的担保

v1.1 §3.2 の通り、上部レイヤーは**一度に 1 発話単位**。UI 側も以下で連投を構造的に抑制する。

- 発話本文カードは同時に 1 枚しか表示しない
- S2 / S5 / S7 のいずれかがアクティブな間、次の発話は表示しない
- S8 退出を経ないと S0 → S1 → S2 の再起動はかからない（v1.1 §8.6 準拠）
- 最短再起動間隔 = 5 分（v1.1 §8.6、`recentProposalWithin5Min` と整合）

## §2 マウント位置と画面境界

### 2.1 マウント位置

CoAlter レイヤーは**ペア chat 画面**の上部に常駐する。

**正本モック:** v1.1 §3.1 の ASCII モック（行 163-200）を一次情報源とする。本書では**複製せず参照のみ**。

**位置関係（v1.1 §3.1 より要点）:**
1. チャットヘッダー（既存、変更なし）
2. **CoAlter 上部レイヤー** ← 本書の対象
3. メインチャット（2 人の吹き出し列、既存）
4. 返答導線ヒント（「@coalter への返答は @coalter で送信」）
5. 入力欄（既存）

**起動導線:**
- 既存「CoAlter ボタン」（`ChatClient.tsx:1898-1908`）が手動起動の起点（v1.1 §3.4）
- ボタン起動後の CoAlter の居場所 = 本書で扱う上部レイヤー
- ボタンと上部レイヤーの接続ロジックは v1.1 §12 Stage 4 で別判断、本書では**表示側のみ**扱う

### 2.2 マウント範囲（対象画面）

CoAlter 上部レイヤーが出現するのは v1.1 §3 で規定された**ペア chat 画面**のみ。

- ✅ ペア chat 画面（2 人の共有コンテキスト）
- ❌ 個人の Home / 個人の Alter 空間
- ❌ Rendezvous マッチング画面（ペア成立前）
- ❌ 設定画面・プロフィール画面

他画面への展開は本書のスコープ外。将来展開時は v1.1 側の更新が先行する。

### 2.3 z-index

CoAlter 上部レイヤーは、メインチャットより上位の z-index を持つ。

| 要素 | z-index 階層 | 説明 |
|---|---|---|
| Modal / Overlay（既存 UI の規則） | 最上位 | 設定 modal, 画像拡大等。CoAlter レイヤーも**これより下** |
| **CoAlter 上部レイヤー** | モーダル直下 | 常にメインチャットより上 |
| メインチャット（吹き出し列） | 中位 | 既存 |
| チャットヘッダー | CoAlter と同等または直下 | 既存 UI との整合性で調整 |

**原則:**
- CoAlter がモーダルを表示しない（モーダル化禁止 — v1.1 §11.2 主役を奪わない）
- 既存モーダル表示中は CoAlter レイヤーは背後に下がる（フォーカス保全）

### 2.4 focus（フォーカス競合）

**原則:** ユーザーの注意は**原則メインチャット**に向けられる。CoAlter は脇から話す。

| ケース | 挙動 |
|---|---|
| 通常時（S0, S8 等） | focus はメインチャット入力欄に | 
| 発話中（S2, S5, S7） | focus はメインチャットに留まる。CoAlter のチップは tap で応答可能だが、入力欄フォーカスを奪わない |
| 状態優先切替時（S2/S5 + 閾値超え） | 視覚的な注意喚起（pulse アニメ）のみ。**focus trap は作らない** |
| チップ tap 時 | 一時的に CoAlter にフォーカス → 応答後すぐメインチャットに戻す |

**禁止:**
- CoAlter がメインチャットの入力欄から focus を奪う UI
- focus trap（キーボード操作で CoAlter レイヤー内に閉じ込める）
- 全画面オーバーレイ（CoAlter がメインチャットを覆う）

### 2.5 scroll（スクロール連動）

**原則:** CoAlter 上部レイヤーは**pinned**（スクロールしても常に上部に残る）。ただし高さは状態で変わる。

| 状態 | 上部レイヤーの高さ | scroll 挙動 |
|---|---|---|
| S0 / S8 | single-line（最小） | メインチャットが広く見える |
| S1 / S3 / S4 / S6 | compact-card（中） | メインチャットは残り領域でスクロール |
| S2 / S5 / S7 | expanded-card（大） | メインチャット領域は圧縮されるが、スクロール位置は維持 |

**メインチャット側のスクロール:**
- CoAlter 状態変化（S1 → S2 等）で上部レイヤー高さが変わっても、**メインチャットの scroll 位置は維持**
- 新着メッセージへの auto-scroll は CoAlter 状態と独立（既存挙動を尊重）
- CoAlter の展開でメインチャットの最下段が隠れることを避けるため、メインチャット側に上部レイヤー高さ分の safe-area padding を確保（実装詳細は §9 で保留）

### 2.6 入力欄との競合境界

**原則:**
- 入力欄は**単一**（メインチャット／CoAlter で別々に持たない）
- CoAlter 宛の routing は**明示トリガー**または**状態依存トリガー**で行う
- **具体的な routing 記法は本書では固定しない**。将来の interaction refinement フェーズに委ねる

**本書で固定する事項:**

| 項目 | 固定内容 |
|---|---|
| 入力欄の個数 | 1 個（専用欄追加禁止） |
| routing の原則 | 明示トリガー or 状態依存トリガー |
| CoAlter 宛入力中の視覚差別化 | 必要（形式は §9 保留論点） |
| 送信後の routing 決定 | トリガーの有無・種類により分岐（記法は未固定） |

**本書で固定しない事項（将来の interaction refinement で判断）:**
- 明示トリガーの記法（`@coalter` prefix / mention / chip 起点 / その他のいずれに寄せるか）
- 状態依存トリガーの発動条件と mixing ルール
- prefix 補助 UI（候補サジェスト・自動補完など）

**v1.1 §3.1 モックとの関係:**
v1.1 §3.1 ASCII モックには「@coalter で話しかける」「@coalter で相談」という表現がある。これは**現時点の暫定表現**であり、本書の正本として固定しない。最終的な routing 記法は interaction refinement フェーズで確定する。本書では当該表現を**例示**として扱い、設計上の制約にしない。

**禁止:**
- CoAlter 専用の入力欄を別途追加（入力欄は単一原則）
- CoAlter 宛入力中に視覚差別化が一切ないレイアウト（ユーザーが「今どちらに向けて書いているか」を判別できない状態）

### 2.7 action 実行後の handoff 境界

ユーザーが CoAlter のチップ / 導線を tap した後、どこで処理が完結しどこでメインチャットに戻るかを明確にする。

| action | 実行後の挙動 | メインチャットへの transfer |
|---|---|---|
| 応答チップ tap（「近い」「少し違う」等） | CoAlter 内で state 遷移（S5 → S6 等） | **しない** |
| 片側フォーカス（「たいしに聞く」等） | CoAlter 発話が片側向けに更新 | **しない**（メインチャット側の 2 人の発話は独立） |
| 「提案を聞く」tap | S6 → S7 遷移、提案が上部に表示 | **しない** |
| 「いったん戻る」tap | S8 退出、上部レイヤー最小化 | **しない** |
| **「この提案をチャットに共有」tap（あれば）** | 提案がメインチャットの吹き出しとして transfer | **する**（明示的転送のみ） |

**原則:**
- CoAlter 内で完結する action は**メインチャットに混入しない**
- メインチャットに転送するのは**ユーザーが明示的に転送を選択した時のみ**
- 転送された提案は CoAlter の吹き出しとしてではなく、ユーザー（たいし or みさき）の発話として記録される（「〜さんが共有した提案」のような形式。詳細はテンプレ doc）

## §3 UI 要素カタログ

### 3.1 カタログの位置づけと読み方

本章は v1.1 §9 で列挙された UI 要素を**実装解像度に深掘り**する。v1.1 §9 に存在する要素はそのまま引き継ぎ、本書で意味や名前を変えない。v1.1 §9 に存在しない要素は **§3.6 拡張要素** として明示区分する。

**各要素を以下 5 属性で記述する:**

| 属性 | 取りうる値 |
|---|---|
| **層**（§1.2） | 常設 / 状態依存 / チップ |
| **位置** | 上部レイヤー左 / 中央 / 右 / カード上部 / カード本文 / カード下部 / カード下部左 / カード下部右 / 入力欄内 |
| **サイズカテゴリ**（§1.4） | single-line / compact-card / expanded-card / inline（要素単位の最小単位） |
| **出現条件** | 常時 / 状態依存（S コードを列挙）/ 緊急時のみ |
| **action** | tap / chip tap / 読むのみ / なし |

### 3.2 常設要素（v1.1 §9.1 準拠）

| 要素 | 位置 | サイズ | 出現条件 | action | 備考 |
|---|---|---|---|---|---|
| **CoAlter シンボル** | 上部レイヤー左 | inline（小） | 常時 | tap（詳細表示は §9 保留） | v1.1 §9.1: 「2 人のアイコン融合、紫・青グラデの球体」 |
| **ステータス** | シンボル右隣 | inline（1 行） | 常時 | なし | 表示内容の集合: 見守り中 / 理解更新中 / 発話中 / 返答待ち / 提案準備中（v1.1 §9.1） |
| **モード切替** | 上部レイヤー中央 | inline（横並び 3 チップ） | 常時 | chip tap（3 択） | [通常] [Daily] [Travel]、現在選択は強調（v1.1 §9.1） |

### 3.4 チップ体系（v1.1 §9.3 準拠）

本節は**チップの列挙と共通原則**のみ扱う。状態別の最大数・抑制ルール等は本節では固定せず、§4 および §7 を正本とする（§0.4 正本分離ルールの延長）。

**標準チップ一覧（v1.1 §9.3 準拠、9 種）:**

| チップ文言（機能） | 用途 |
|---|---|
| 近い | §4.3 Confirm 肯定 |
| 少し違う | §4.3 Confirm 否定（修正招待） |
| 続けて | 発話継続承認 |
| たいしに聞く | §4.4 Focus Side A 側起動（動的に表示名適応） |
| みさきに聞く | §4.4 Focus Side B 側起動（動的に表示名適応） |
| 提案を聞く | S6 → S7 承認 |
| もう少し整理する | S6 での継続意思 |
| 今はここまでにする | S6 → S8 退出 |
| いったん戻る | S5 / S6 / S7 → S8 退出 |

> **注記**: 本章では各チップが「どの状態で出現するか」の正本化はしない。出現可否は **§7（pattern → state 許可）** と **§4（state × mode による抑制／override）** の AND 判定で決まる（§0.4 準拠）。

**チップ配置のグローバル原則（本章で固定する事項）:**

- **overflow UI 禁止** — 「その他」ボタン・ドロップダウン展開などで隠さず出さない
- **隠さず出さない** — 入りきらない chip は表示しない（隠して畳むのではなく、そもそも出さない）
- **低密度優先** — 同時表示は最小限。多段展開・折り畳みで密度を上げる設計禁止
- **視認性最優先** — 1 行に収まる数まで。可読性を損なう詰め込みをしない
- **動的チップのベース** — ユーザー表示名ベースで生成（v1.1 §9.4）、A/B の表示名を使用

**状態別・パターン別の数値ルールの正本（本章では定めない）:**

| 何を決めるか | 正本 | 本書での章 |
|---|---|---|
| (state, mode) 単位の最大チップ数 | マトリクス | **§4** |
| pattern 内部構造に由来する上限（compact / expanded 等） | パターン仕様 | **§7** |
| 最終可否 | AND 判定 | `§7 許可 AND §4 抑制なし`（§0.4） |

**チップの文言方針（本書では固定しない）:**

- 具体文面・語彙・語尾は発話テンプレ doc に委譲（§0.5 準拠）
- 本書では「何が押せるか（機能）」のみを固定する

## §4 状態 × モード 優先順位マトリクス【中核】

本章は本書の**中核**である。v1.1 §2（3 モード）と v1.1 §8（S0-S8）の**直交関係**を UI 上で固定する。

### 4.1 読み方

マトリクスは **行 = 状態（S0-S8 の 9 行）、列 = モード（通常 / Daily / Travel の 3 列）**。

**縮約ルール:**
- **通常モード列 = 基線**（各セルで 6 属性を full 記述）
- **Daily / Travel 列 = 差分のみ**、以下の記号で表記:

| 記号 | 意味 |
|---|---|
| `= 通常` | 通常モードと完全一致。差分なし |
| `+ 追加: ...` | 通常モードに加えて新規要素・action が加わる |
| `→ override: ...` | 通常モードの一部が別のルールに置き換わる |

**差分が無い (state, mode) セルは `= 通常` と記載。空欄は残さない。**

### 4.2 セル 6 属性の定義

通常モード列の各セルは以下 6 属性を full 記述する。Daily / Travel 列では差分のみ記載。

| 属性 | 定義 | 本書内の参照 |
|---|---|---|
| **1. 表示面** | 出現する UI 要素（常設／状態依存／チップ）の集合 | §3 要素カタログ |
| **2. 許可 action** | 当該 (state, mode) で受け付ける action（tap / chip tap / モード切替等） | §3 要素の action 属性 |
| **3. 禁止 action** | 当該セルで**明示的に禁止**される action（発話連投・複数パターン同時発話等） | v1.1 §11 の禁止事項と整合 |
| **4. 発話トーンカテゴリ** | calm / neutral / urgent / retreat / N/A（発話なし） | §0.5 準拠、具体文面はテンプレ doc |
| **5. 昇格／降格可能性** | 次に遷移可能な状態・モード | v1.1 §8.3 基本フローと整合 |
| **6. UI 密度** | single-line / compact-card / expanded-card | §1.4 準拠 |

**override の範囲:**
§0.4 準拠。`override` が上書きできるのは上記 6 属性のうち**表示面・許可 action・UI 密度**まで。pattern の意味定義（§7 正本）は上書きしない。

| 2. 許可 action | 承認 tap、閉じる導線 tap、「この提案をチャットに共有」tap（§2.7 明示的 handoff） | = 通常 | = 通常 |
| 3. 禁止 action | 複数提案同時表示（F-1 と F-2 同時禁止、1 件のみ）、追加発話、新規 chip 追加 | = 通常 | = 通常 |
| 4. 発話トーン | neutral（提案本文のトーン） | = 通常 | = 通常 |
| 5. 昇格／降格 | → S8（承認／不承認どちらも S8 退出） | = 通常 | = 通常 |
| 6. UI 密度 | expanded-card | = 通常 | = 通常 |

### 4.4 状態優先切替の例外（§1.3 A3 の具体化）

S2 / S5 では、介入価値が閾値を超える場合に限って**状態優先**に切り替わり、必要ならモード昇格が起こる。§1.3 の原則を本章で具体化する。

**状態優先の発動条件（本書では閾値の数値は固定しない）:**
- 強いすれ違い / 攻撃性 / 感情のヒートアップ（v1.1 §8.4）
- 関係保護の必要性が通常より高い
- CoAlter 側の介入価値スコアが閾値以上（閾値ロジックは実装詳細、§9 保留論点）

**状態優先切替時の UI 変化:**

| 属性 | 通常時（モード優先） | 状態優先切替時 |
|---|---|---|
| 発話トーン | calm | **urgent**（トーンカテゴリ切替、具体文面はテンプレ doc） |
| UI 密度 | S2 = compact-card / S5 = expanded-card | **expanded-card**（いずれも展開） |
| アニメ | fade / slide-down | **pulse** 1 回（注意喚起、§1.5） |
| chip 数 | S2 = 最大 2 / S5 = 最大 3 | **削減**（最優先の 1-2 個のみ） |
| 告知 | モード昇格時は告知チップ（§4.3.3 / §4.3.6 の「Daily スコープ告知」等） | **事前告知（予告発話・前置き）なしで即時発話可。ただし視覚的状態変化は必須** — 上記 pulse / urgent トーン / expanded 密度 / chip 数削減 のいずれか（複数で構成される）を**即時視覚シグナル**として提示する |

**視覚キューの必須性（明示ルール）:**
- 「事前告知なし」 ≠ 「視覚変化なし」
- 即時発話する場合でも、**無告知かつ無視覚変化**は禁止（ユーザーの驚き・支配感を避けるため）
- 最低限、pulse アニメ 1 回 + トーン変化または密度変化のいずれかは発動する
- これは UX の納得感と緊急介入の即時性を両立する線

**状態優先切替時のモード昇格（S5 のみ）:**
- S5 で状態優先切替が発生し、かつ関係保護に**長期的構造化**が必要と判定された場合、通常 → Daily/Travel への自動昇格が起こる
- 昇格は自動だが、**既存の常設要素（§1.3 後半）は維持**され、UI の骨格は崩れない
- 昇格判定ロジックは実装詳細、§9 保留論点

**状態優先が発動しない場合:**
- S2 / S5 であっても、介入価値が閾値以下なら通常通りモード優先で振る舞う（§1.3 重要ブロック）
- 「S2 / S5 = 必ず urgent・必ず昇格」ではない

本章は (state, mode) セルごとの UI 規則を確定する。以下は後続章の責務:

## §5 状態別 UI 詳細

### 5.1 本章の責務と非責務

**本章で扱う:**
- 各状態の**見え方**（ASCII レイアウトでの要素配置）
- **相対配置**（要素間の上下・左右関係）
- **折りたたみ方**（畳み状態 / 展開状態 / 中間状態）
- **前後関係**（遷移時に何が残り、何が消え、何が新規出現するか）

**本章で扱わない（§4 / §7 正本）:**
- 許可 action / 禁止 action → **§4 マトリクス**
- 発話トーン → **§4 マトリクス**（カテゴリのみ）／**テンプレ doc**（具体文面）
- 昇格／降格条件 → **§4.3 各セル / §6 モード切替**
- 発話パターンの内部構造 → **§7 発話サーフェス仕様**

**二重正本の禁止:**
§5 で上記「本章で扱わない」事項を再記述したら、その記述は削除対象（§0.4 正本分離ルールの延長）。

### 5.2 ASCII レイアウトの共通規約

**使用記号:**
- `╔═╗ ║ ╚═╝` — CoAlter 上部レイヤーの外枠（v1.1 §3.1 モックと同じ）
- `┌─┐ │ └─┘` — 発話本文カード・提案カードの枠
- `[ text ]` — チップ / ボタン
- `● ○ 🔵` — アイコン・ステータスドット（v1.1 §3.1 と同じ）
- `···` — 折りたたまれた要素の省略
- `〜` — 文面の省略（具体文面はテンプレ doc）

**省略ルール:**
- 表示名は `たいし` / `みさき` をプレースホルダとして使用（v1.1 §3.1 と同じ）
- 発話本文・チップ文面は簡略化（具体は §7 / テンプレ doc）
- 入力導線ヒントの具体記法は記載しない（§2.6 未固定）

```
╚══════════════════════════════════╝
```

**折りたたみ:** 退出メッセージを 1 行表示後、ステータスのみの single-line に収束。発話本文カード・チップ類は完全消失。

**固定アンカー:**
- 退出メッセージは**1 行のみ**、ステータス位置に統合
- 数秒後 single-line に収束、カード類完全消失
- 高さは single-line に戻る（S0 と同じ骨格）
- 警告色・叱責的アイコンへの変化禁止（§6.8 UI 禁止表現準拠）

**前後関係:**
- ← S5 / S6 / S7（任意状態から閉じる導線）
- → S0（5 分再起動禁止解除後、自動）

### 5.12 遷移時のレイアウト継承

状態遷移時にどの要素が**残り**、**消え**、**新規出現**するかを一覧化する。

| 遷移 | 残る | 消える | 新規出現 | アニメ |
|---|---|---|---|---|
| S0 → S1 | 常設要素 | — | status chip | fade |
| S1 → S2 | 常設要素、status chip（fade out） | status chip | 発話本文カード、応答チップ | slide-down |
| S2 → S3 | 常設、発話本文カード、応答チップ | — | — | ステータスのみ変化 |
| S3 → S4 | 常設 | 応答チップ（fade out） | — | 発話本文カード薄表示 |
| S4 → S5 | 常設 | — | 発話本文カード（更新後）、応答チップ、閉じる導線 | slide-down |
| S5 → S6 | 常設、発話本文カード（薄表示） | 応答チップ、片側フォーカス導線 | 提案導線 3 ボタン | fade |
| S6 → S7 | 常設 | 発話本文カード（薄表示）、提案導線 | 提案カード（expanded）、承認チップ、共有 tap | slide-down |
| S7 → S8 | 常設 | 提案カード、承認チップ、共有 tap | 退出メッセージ（1 行） | fade |
| S* → S8（閉じる導線） | 常設 | 当該状態の全要素 | 退出メッセージ（1 行） | fade |
| S8 → S0 | 常設 | 退出メッセージ | — | fade（最小化） |

**共通原則:**
- 常設要素（§3.2）は**全遷移で維持**（状態優先切替時も含む、§1.3 後半）
- アニメカテゴリ（fade / slide-down / pulse / none）は §1.5 準拠
- 状態優先切替（§4.4）が発生する遷移では **pulse 1 回が必須**視覚キューとして追加される

## §6 モード切替と昇格／降格 UI

### 6.1 本章の責務

**本章で扱う:**
- モード切替の 3 形態（手動 / 自動昇格 / 自然退出）
- §4.4 状態優先切替を**時間軸 × UI イベント**として展開
- **拒否の 3 分類**を分離定義（信頼・再介入条件の崩れを防ぐため）
- 通常モードへの復帰導線

**本章で扱わない（他章正本）:**
- モードの存在論的意味（v1.1 §2）
- 状態別 UI 密度・要素（§4 / §5）
- 発話文面（テンプレ doc）

### 6.2 モード切替の 3 形態

モード切替は発生元の違いで 3 形態に分かれる。混同厳禁。

| 形態 | 発生元 | 発動タイミング | 承認 |
|---|---|---|---|
| **手動切替** | ユーザー操作 | モード切替チップ tap | 不要（即時反映） |
| **自動昇格** | CoAlter 内部判定 | S5 状態優先切替時、かつ長期構造化必要と判定 | ユーザー拒否可（§6.6.1） |
| **自然退出** | プラン完成 | Daily/Travel でプラン出力完了後 | 不要（自動で通常モードへ） |

### 6.3 手動切替の UI フロー

ユーザーが任意のタイミングでモード切替チップを tap。即時反映。

**フロー:**
1. ユーザーが `[通常] [Daily] [Travel]` のいずれかを tap
2. active チップがアニメで切替（fade、§1.5）
3. ステータスが新モードに応じて更新
4. 共有メモリ surface（§3.6）の文脈スコープが新モードのルールで再参照される（v1.1 §10.2 準拠、詳細は §8）

**禁止:**
- 切替時にモーダル確認を出す（§2.3 モーダル化禁止と整合）
- 切替を完了する前に状態遷移を発火する

### 6.4 自動昇格の UI フロー（S5 状態優先切替時）

S5 で介入価値が閾値を超え、かつ**長期的構造化が必要**と判定された場合のみ発動（§4.4 準拠）。

**フロー:**
1. S5 で介入価値閾値超え判定（閾値数値は §9 保留）
2. 長期構造化必要性の判定（ロジックは §9 保留）
3. **視覚キュー必須**（§4.4）:
   - pulse アニメ 1 回（枠強調）
   - トーン urgent（具体文面はテンプレ doc）
   - 密度 expanded-card
4. モード切替チップの active が通常 → Daily/Travel に切替（アニメ fade）
5. 発話本文カードが新モードの文脈ヒント付きで継続（例: `◇ 今日のスケジュール見ながら 〜`）

**発動条件の非固定:**
- 介入価値閾値・構造化必要性判定のロジックは本書では固定しない（§9 保留）

**ユーザー拒否可:**
§6.6.1 のモード昇格拒否が可能。詳細は §6.6.1 で。

### 6.5 通常モードへの復帰

Daily/Travel モードから通常モードに戻る経路は 2 つ。

**6.5.1 自然退出（自動）:**
- 発動: プラン完成後（Daily 1 日プラン出力完了 / Travel Plan Brief 出力完了）
- 遷移: Daily/Travel → 通常（即時、fade アニメ）
- 文脈継承: プラン結果が共有メモリ surface に格納（v1.1 §10.3 準拠）
- ユーザー操作不要

**6.5.2 手動復帰（ユーザー操作）:**
- 発動: モード切替チップで `[通常]` tap
- 遷移: 即時（§6.3 手動切替と同じフロー）
- 未完成プランの扱い: 共有メモリ surface に中間状態として格納（詳細は §8）

### 6.6 拒否の 3 分類（分離定義）

「拒否」を 1 種類で扱うと、信頼・再介入条件が崩れる。以下 3 種類に分離する。

#### 6.6.1 モード昇格の拒否

**対象:** §6.4 の自動昇格に対する拒否

**発動条件:** 自動昇格直後にユーザーが元モードに戻す

**UI:**
- モード切替チップで元モード tap
- または、昇格直後に補助表示される「通常に戻す」チップ tap（具体表示条件は §9 保留）

**介入感度への影響:**
- CoAlter 側の自動昇格閾値ロジックを**厳格化方向に調整**（将来の学習として）
- 当該セッション内では自動昇格の再試行を抑制

**再介入条件:**
- **同一セッション内の自動昇格再試行は禁止**（当該セッション終了まで）
- 次のセッションでの自動昇格は通常通り閾値判定で判断

#### 6.6.2 個別提案の拒否

**対象:** S7 で表示された個別提案に対する拒否

**発動条件:** S7 で承認チップを押さず、「閉じる」導線 or 時間経過で S8 に退出

**UI:**
- S7 の閉じる導線 tap
- あるいは承認なしでの S8 遷移（タイムアウトは §9 保留）

**介入感度への影響:**
- 当該提案の内容を共有メモリ surface に「提示済み・未採択」として記録
- CoAlter は同じ内容の提案を**短期的には再提示しない**（短期の定義は §9 保留）

**再介入条件:**
- **別の signal trigger で再度 S1-S2 起動可能**（通常のフロー）
- 同じ提案内容の再提示は、ユーザー側で「別の提案を聞く」意思表示があった場合のみ許可

#### 6.6.3 介入そのものの後退要求

**対象:** CoAlter の介入そのものに対する拒否（「しばらく黙って」等）

**発動条件:**
- ユーザーの明示的な後退要求（入力テキストの signal 検出 or 専用 chip、詳細は §9 保留）
- 例: 「今日は声をかけないで」「しばらく見ていて」「黙って」等

**UI:**
- CoAlter が即座に S8 退出
- 退出メッセージは retreat トーンで明示（「わかった、しばらく見てるね」等、具体文面はテンプレ doc）
- 長めの cooldown 期間中は上部レイヤー最小化（single-line 維持）

**介入感度への影響:**
- **最も大きい**。後退要求は**介入制御の強いシグナルとして永続的に記録**（共有メモリ surface、人格評価・信頼評価ではない）
- 今後の signal 検出閾値を厳格化（CoAlter が「控えめ」方向に学習）

**再介入条件:**
- **指定期間経過まで S0 → S1 自動遷移を完全停止**（指定期間の数値は §9 保留、例: 24 時間）
- 期間中も**ユーザー明示呼び出し（モード切替 tap / @coalter 相当の明示トリガー）には応答可**
- 期間経過後は通常の signal 検出フローに復帰

### 6.7 再介入条件サマリ

拒否種類別に、再介入可否と cooldown を一覧化する。

| 拒否種類 | セッション内再試行 | 次セッション以降 | 明示呼び出し応答 |
|---|---|---|---|
| モード昇格の拒否（§6.6.1） | **禁止**（同一セッション） | 通常通り閾値判定 | 可（モード切替 tap は常時） |
| 個別提案の拒否（§6.6.2） | 同内容は抑制、別 trigger は可 | 通常通り | 可 |
| 介入後退要求（§6.6.3） | **完全停止**（指定期間） | 指定期間経過後、通常復帰 | 可（期間中も） |

**原則:**
- 3 種類の拒否は**それぞれ独立の cooldown・信頼影響**を持つ
- 一括化は禁止（「拒否されたから何もしない」は信頼の過剰自己抑制）
- 明示呼び出し（ユーザーからの能動的な接触）は**常にどの拒否状態でも応答可**

### 6.8 拒否の非判定性

CoAlter は拒否を**罰として扱わない**。

- 拒否されても CoAlter は「ダメだった」と自己評価しない
- 介入感度への影響は「閾値調整」「抑制期間」の**機械的処理**のみ（信頼・人格評価ではない）
- ユーザー側の「断る権利」を UI 設計で保証する（拒否導線は常に視認性ある位置に配置）

これは v1.1 §11.1「裁判官にならない」の UI 実装面への展開。

**UI 禁止表現（非判定性の視覚ルール）:**

理念だけでは「文言は優しいが見た目が圧になる」事故が起きる。以下は視覚チャネルでの非判定性を担保するための禁止項目。§6.6.1-§6.6.3 のいずれの拒否に対しても全適用される。

- **警告色への変化禁止** — 拒否後にステータス・枠・背景を赤・オレンジ等の警告系に変えない
- **叱責的アイコン禁止** — ⚠️ / ❌ / 😟 / 🚫 等、罰・失望・失敗を示唆するアイコンを拒否後に表示しない
- **拒否カウンタの累積表示禁止** — 「〇回断られた」「これ以上は…」等の累積可視化 UI を出さない
- **「また」系の再接触演出禁止** — 「またか」「しつこく」等、累積責任を匂わせる文面・トーン・アニメを使わない
- **cooldown カウントダウン禁止** — 「あと N 時間で復帰」等の残時間カウントダウン UI を出さない（ユーザーに「時間切れで戻ってくる」と感じさせない）

これらの禁止は、CoAlter が「関係中心の共有 Alter」であり続けるための UI 側の最低限の担保である。

## §7 発話サーフェス仕様

### 7.1 本章の責務と非責務

**本章で扱う:**
- 各発話パターンの**意味定義（不変核）** — 何をする pattern か、主語、出力最小単位、同居補助
- **surface 構造** — ヘッダ / 本文 / chip の構成
- **pattern → state 許可リスト**（§0.4 正本分離の§7 側正本）
- **合成規則**（F-1 / F-2 の共存ルール）
- **非同居規則**（禁止される組み合わせ）

**本章で扱わない（他章正本）:**
- mode 制約（Daily/Travel での抑制・override） → **§4 正本**
- chip 数上限の最終決定 → **§4 正本**
- UI 配置・レイアウト・アニメ → **§5 正本**
- 具体文面・語彙・語尾 → **テンプレ doc**

**最終可否の判定（two-stage gating）:**

§7 と §4 は同じ種類の正本ではない。**§7 = positive allow**（存在許可）/ **§4 = negative override**（抑制・優先度調整）。

1. **Stage 1 — §7 existence gate**: pattern × state が §7.12 で ✓ か（存在可能性）
2. **Stage 2 — §4 suppression gate**: §4.3 mode 別マトリクスで抑制・優先度低下されていないか（mode 内での実発火可否）

最終可否 = Stage 1 AND Stage 2（§0.4 正本分離準拠）

### 7.2 不変核の定義フォーマット

各 pattern について以下 4 項目を**最小固定**する。ここが曖昧だと、後で全パターンが「少し違う supportive card」に潰れる。

| 項目 | 定義 |
|---|---|
| **目的** | 何をする pattern か（1 文） |
| **注意の主語** | 2 人 / 片側 / 関係 / 外界 のいずれか |
| **出力の最小単位** | 1 行介入 / 翻訳 / 提案 / 承認ゲート 等 |
| **同居してよい補助要素** | 補助 chip / 1 行配慮 / 閉じる導線 等 |

### 7.3 Pattern A — 入口発話（Entry）

**不変核:**

| 項目 | 内容 |
|---|---|
| 目的 | 介入前の「入ってよいか」の短い打診（v1.1 §4.1） |
| 注意の主語 | **2 人の関係**（両者向けの入口確認） |
| 出力の最小単位 | **1 文の打診**（断定せず、結論を言い切らない） |
| 同居補助要素 | 応答チップ（最大数は §4.3.3 正本） |

**誤読防止:** 提案化しない

**surface 構造:**
- ヘッダ: なし（最小構造）
- 本文: **1 文**（計画・分析・翻訳を含まない）
- chip: 応答チップのみ

**state 許可:** **S2 のみ**

**非同居規則（当パターン固有）:**
- 同パターン連投禁止（§1.6、v1.1 §11.4）
- B / C / D / E / F-1 / F-2 との**同時発話禁止**
- 長い前置き禁止（v1.1 §4.1 準拠）

### 7.4 Pattern B — 状況言語化（Frame）

**不変核:**

| 項目 | 内容 |
|---|---|
| 目的 | **通常モードの中核**。2 人が言葉にし損ねた現状を、責めずに短く外部化（v1.1 §4.2） |
| 注意の主語 | **関係**（A 側 / B 側 の視点を並列提示） |
| 出力の最小単位 | **2-4 行の並列表現**（A 側／B 側／まとめ） |
| 同居補助要素 | 応答チップ、閉じる導線 |

**誤読防止:** 評定しない

**surface 構造:**
- ヘッダ: なし（本文に統合）
- 本文: **2-4 行**（A 側視点 → B 側視点 → まとめ）
- chip: 応答チップ

**state 許可:** **S5 のみ**

**非同居規則（当パターン固有）:**
- F-1 / F-2 との同時発話禁止（v1.1 §11.3: 関係保護 → 提案の順序厳守）
- C との統合禁止（確認質問は別ターン）
- 感情ラベル貼り禁止 / 片側肩入れ禁止 / 本音代弁禁止（v1.1 §4.2 禁則）

### 7.5 Pattern C — 確認質問（Confirm）

**不変核:**

| 項目 | 内容 |
|---|---|
| 目的 | CoAlter の読みが合っているか、**2 人に修正招待**（v1.1 §4.3） |
| 注意の主語 | **2 人**（両者に確認を投げる） |
| 出力の最小単位 | **1-2 文の問い**（例: 「この理解で近い？」） |
| 同居補助要素 | 応答チップ（近い / 少し違う / 続けて） |

**誤読防止:** 尋問化しない

**surface 構造:**
- ヘッダ: なし
- 本文: **1-2 行の問い**
- chip: 応答チップ 3 個（近い / 少し違う / 続けて）

**state 許可:** **S2 / S5**（入口での確認 or 橋渡し中の確認）

**非同居規則（当パターン固有）:**
- F-1 / F-2 との同時発話禁止
- D との同時発話禁止（確認と片側フォーカスは別ターン）
- B との統合禁止（B が出た同一ターンで C を重ねない）

### 7.6 Pattern D — 片側フォーカス（Focus Side）

**不変核:**

| 項目 | 内容 |
|---|---|
| 目的 | 片方に**静かに視線を向ける**（判定せず、聞く間を作る。v1.1 §4.4） |
| 注意の主語 | **片側**（A または B、動的に決定） |
| 出力の最小単位 | **1-2 文の静かな問い**（片側向け、判定しない） |
| 同居補助要素 | 片側フォーカス導線（「たいしに聞く」等）、応答チップ |

**誤読防止:** 代弁しない

**surface 構造:**
- ヘッダ: なし
- 本文: **1-2 行**（片側向け、もう片側を責めない）
- chip: 片側フォーカス導線（2 個：A 側 / B 側）+ 応答チップ（必要時）

**state 許可:** **S5 のみ**

**非同居規則（当パターン固有）:**
- F-1 / F-2 との同時発話禁止
- 裁判官化禁止（「どちらが正しい」判定の文面禁止、v1.1 §11.1）
- Travel モードでは**既定優先度低下**（§4.3.6）、ただし関係シグナル検出時は再昇格
- もう片側への責め文禁止（片側への焦点は、もう片側の不在を責めない形で表現）

### 7.7 Pattern E — 橋渡し・翻訳発話（Bridge）

**不変核:**

| 項目 | 内容 |
|---|---|
| 目的 | 相互の意図・背景を**翻訳して橋を架ける**（関係保護。v1.1 §4.5） |
| 注意の主語 | **関係**（両者の橋渡し、片側の代弁ではない） |
| 出力の最小単位 | **1-3 行の翻訳**（A の意図を B に、B の意図を A に） |
| 同居補助要素 | 応答チップ、閉じる導線 |

**誤読防止:** 裁定しない

**surface 構造:**
- ヘッダ: なし
- 本文: **1-3 行**（翻訳、「〜かも」「〜に見える」で仮説として提示）
- chip: 応答チップ

**state 許可:** **S5 のみ**

**非同居規則（当パターン固有）:**
- F-1 / F-2 との**同時発話禁止**（v1.1 §11.3: 関係保護が先、提案は S7）
- 片側本音の代弁禁止（v1.1 §4.2 禁則準拠、仮説としての翻訳に留める）
- B と同時発話禁止（言語化と翻訳は別ターン）

### 7.8 Pattern F-1 — 関係提案

**不変核:**

| 項目 | 内容 |
|---|---|
| 目的 | **関係内部向け**の軽提案（2 人の合意形成・相互理解・ルール化） |
| 注意の主語 | **関係**（2 人の内部構造に関する提案） |
| 出力の最小単位 | **1 件の軽提案**（1-3 行）+ **承認ゲート 1 個** |
| 同居補助要素 | 承認チップ、閉じる導線、明示共有 tap |

**誤読防止:** 生活効率最適化に寄せない

**surface 構造:**
- ヘッダ: 意味ラベル（「提案:」等、具体文面はテンプレ doc）
- 本文: 提案 1 件（1-3 行）
- chip: 承認チップ 1 個 + 閉じる導線

**state 許可:** **S7 のみ**

**非同居規則（当パターン固有）:**
- **F-2 との独立カード化禁止**（§7.10 合成規則準拠）
- 複数の F-1 同時表示禁止（提案は 1 件のみ、v1.1 §8.2 S7 定義）
- B / C / D / E との同時発話禁止
- Daily/Travel モードでの扱いは §4.3.8 準拠（Daily: 関係ノイズ低時抑制可 / Travel: 副次同伴必須）

### 7.9 Pattern F-2 — 生活提案

**不変核:**

| 項目 | 内容 |
|---|---|
| 目的 | **外界向け**の軽提案（レストラン / プラン / スケジュール等、2 人が一緒に触れる外） |
| 注意の主語 | **外界**（2 人の外側にある対象） |
| 出力の最小単位 | **1 件の提案**（通常 = 単発、Daily = 1 日プラン、Travel = 複数日 Plan Brief）+ **承認ゲート 1 個** |
| 同居補助要素 | 承認チップ、閉じる導線、明示共有 tap、F-1 副次同伴（§7.10 合成規則） |

**誤読防止:** 関係評価に寄せない

**surface 構造:**
- ヘッダ: 意味ラベル（「今日のプラン:」「旅行案:」等、具体文面はテンプレ doc）
- 本文: 提案 1 件（expanded-card、通常は 1-3 行、Daily は時間軸付き、Travel は Brief 形式）
- chip: 承認チップ 1 個 + 閉じる導線 + 明示共有 tap

**state 許可:** **S7 のみ**

**非同居規則（当パターン固有）:**
- **F-1 との独立カード化禁止**（§7.10 合成規則準拠）
- 複数の F-2 同時表示禁止（提案は 1 件のみ）
- B / C / D / E との同時発話禁止
- Travel モードでは**承認ゲート厳しめ**（§4.3.8、確認 1 クッション）
- 関係保護（B / E）を経ずに出す禁止（v1.1 §11.3）

### 7.10 合成規則 — F-1 / F-2 の共存

F-1 と F-2 の共存は**新たなハイブリッド pattern を作らない**。pattern 数は 7 のまま保ち、共存はレイアウト規則として吸収する。

**合成ルール:**

1. **F-2 を主表示**とする（expanded-card、提案カードの主役）
2. **F-1 は提案カード内の最終行**として 1 行収容（独立カードを作らない）
3. **独立カード化は禁止**（F-1 と F-2 の同格二枚化を避ける）
4. F-1 副次同伴の発動条件:
   - Daily: 関係ノイズが高い場合のみ（§4.3.8 準拠）
   - Travel: 原則常時（複数日では疲労・主導権偏り・温度差が積むため）

**合成時の surface 構造:**

```
┌──────────────────────────┐
│ ◇ 今日のプラン:（F-2 ヘッダ）│
│ 〜（F-2 本文、1-3 行）    │
│ — 関係配慮 1 行（F-1 副次）— │  ← F-1 は最終行に 1 行のみ
└──────────────────────────┘
```

**禁止:**
- F-1 独立カード + F-2 独立カードの二枚表示
- F-1 を複数行にわたって挿入
- F-1 副次同伴と F-2 本体の順序逆転（F-2 が主、F-1 が副）

### 7.11 非同居規則 — 共通禁止組み合わせ

許可のみ書くと実装は「全部盛り」になる。以下は**全 pattern に共通する禁止組み合わせ**。

| 禁止組み合わせ | 理由 |
|---|---|
| **2 pattern 同時発話** | v1.1 §11.4 連投禁止、§1.6 構造的担保 |
| **片側フォーカス D + 複数日 Brief（F-2 Travel）を同格二枚化** | 2 種類の大きい surface が競合、認知負荷過大 |
| **大きい承認ゲートの複数同時表示** | S7 は提案 1 件原則、ゲート分散による判断曖昧化を防ぐ |
| **緊急介入時（§4.4 状態優先切替）の補助 chip 増量** | chip 数削減が §4.4 の必須要件 |
| **関係保護（B / E）前の提案（F-1 / F-2）** | v1.1 §11.3、提案は S7、関係保護は S5 |
| **裁判官化（D / E / F-1 で「どちらが正しい」判定）** | v1.1 §11.1、CoAlter は翻訳者 |
| **発話 + 提案を同一ターンで同時出し** | 提案は S7 のみ、他状態での提案混入禁止 |
| **同パターン連投** | §1.6、v1.1 §11.4 |

### 7.12 Pattern → State 許可マトリクス（Two-Stage Gating の §7 側正本）

本章は **positive allow**（存在許可）の正本。§4 は **negative override**（抑制・優先度調整）の正本。同じ "許可" を 2 箇所で定義しているのではなく、**異なる種類の権威**である。

**Two-stage gating:**

1. **Stage 1 — §7 existence gate**: この表で pattern × state が ✓ か（存在可能性）
2. **Stage 2 — §4 suppression gate**: §4.3 mode 別マトリクスで抑制・優先度低下されていないか（mode 内での実発火可否）

最終可否 = Stage 1 AND Stage 2。§7 は "どの pattern がどの state に存在可能か"、§4 は "存在可能な pattern を mode / 状態優先切替でどう抑制・調整するか" を定義する。

| Pattern | S0 | S1 | S2 | S3 | S4 | S5 | S6 | S7 | S8 |
|---|---|---|---|---|---|---|---|---|---|
| **A 入口発話** | — | — | ✓ | — | — | — | — | — | — |
| **B 状況言語化** | — | — | — | — | — | ✓ | — | — | — |
| **C 確認質問** | — | — | ✓ | — | — | ✓ | — | — | — |
| **D 片側フォーカス** | — | — | — | — | — | ✓ | — | — | — |
| **E 橋渡し** | — | — | — | — | — | ✓ | — | — | — |
| **F-1 関係提案** | — | — | — | — | — | — | — | ✓ | — |
| **F-2 生活提案** | — | — | — | — | — | — | — | ✓ | — |

**補足:**
- S0 / S1 / S3 / S4 / S6 / S8 は**発話パターンなし**（v1.1 §8.2 準拠）
- S1 の status chip は「発話パターン」ではなく「介入気配の UI」であり、本マトリクスの対象外
- S7 に F-1 / F-2 両方が ✓ だが、**同一ターンでは 1 つのみ発火**（§7.10 合成時は F-2 主 + F-1 副次で 1 つのカード扱い）
- D の S5 許可は**Travel モードで既定優先度低下**（§4.3.6、関係シグナル検出時は再昇格）

**状態ごとの発話パターン上限:**
- S2 = A / C の 1 つ（排他）
- S5 = B / C / D / E のいずれか 1 つ（v1.1 §8.2 S5 定義: 「§4.2 / §4.4 / §4.5 のいずれか 1 つ」）
- S7 = F-1 / F-2 のいずれか 1 つ（合成時は F-2 主 + F-1 副次 = 1 カード）

**state 内選択優先順位:**

複数 pattern が許可されている state で、どれを選ぶかの指針（two-stage gating 通過後の選択 fallback）。許可マトリクスとは別レイヤーの "選択ロジック" として固定する。

| State | 選択優先順 |
|---|---|
| **S2** | 1. **A default**<br>2. **C** — 安全な介入に必要な情報が欠けている場合のみ |
| **S5** | 1. **C** — 不確実性が介入の有効性を阻害している場合<br>2. **B** — 関係全体の可視化が先に必要な場合<br>3. **D** — 片側の揺れ・疲労が主である場合<br>4. **E** — 両者間の翻訳が必要な場合 |
| **S7** | 1. **F-2** — Daily / Travel での default<br>2. **F-1 standalone** — 通常モード S7 のみ<br>3. **F-1 副次** — Daily / Travel で §7.10 合成規則下のみ（独立カード化禁止） |

**論理的意図:**
- 「許可は分かったが複数候補がある時どれを選ぶか」で実装者が迷うのを防ぐ
- S2 の A default: 介入気配検出直後は情報欠落でない限り A（入口発話）で入る（C は確認偏重でユーザー負荷が増すため fallback 扱い）
- S5 の C 最優先: 不確実性下での発話は裁判官化のリスクを高めるため、確認を先行させる（v1.1 §11.1 準拠）
- S7 の F-2 default: Daily / Travel では生活文脈が主であり、関係提案単独（F-1 standalone）は通常モード限定
- F-1 副次は §7.10 合成規則下でのみ（独立 F-1 カードを Daily / Travel で出さない）

## §8 共有メモリと介入 UI

### 8.1 本章の責務と非責務

CoAlter は「関係の共有 Alter」として、**2 人の間に保持する共有メモリ** と、**緊急時の介入視覚層** の 2 つの大きい UI 要素を持つ。本章はこの 2 要素を視覚層として定義する。

**本章で扱う:**
- 共有メモリ surface の**表示モデル**（コンポーネント形態・アクセス導線・更新タイミング）
- 共有メモリ項目の**由来・確定度・可視性**を示す**ラベル体系**（§3.6.1 の具体化）
- 共有メモリに対する**可視性操作 UI**（表示 / 隠す / 明示共有 / 共有解除 / 再表示）
- **緊急介入視覚層**（§3.6.2 / §4.4 の UI 実装）
- memory surface と urgent layer の**同時出現時の優先順位**

**本章で扱わない（他正本へ）:**
- 共有メモリの DB スキーマ・保存構造 → **実装側**（本書範囲外）
- 共有メモリの推定アルゴリズム・抽出ロジック → **v1.1 / 実装側**
- 緊急介入の**発火トリガ条件・閾値** → **§4.4 / §9**
- 具体文面・語彙・ラベルの日本語表記 → **テンプレ doc**
- 配色 HEX・アニメ duration・タイポグラフィ → **§9 / デザインシステム**
- 既存 `ChatClient.tsx` 等の実装詳細への踏み込み（§0.3 境界）

**正本分離:**
- **§4.4** = 状態優先切替の論理（いつ urgent になるか） / **§8.5** = その視覚層の型定義
- **§5** = state 別レイアウト（各 state で surface がどう出るか） / **§8** = shared memory と urgent layer の 2 要素の定義
- **§6.8** = 非判定性（urgent でも警告色・叱責アイコン禁止） / **§8.5** はこれを継承

### 8.2 共有メモリ surface の表示モデル

#### 8.2.1 コンポーネント形態

共有メモリ surface は以下の**4 形態**を持つ。状況により切り替わるが、形態は固定される（実装者が新形態を追加しない）。

| 形態 | 定義 | 使用文脈 |
|---|---|---|
| **panel**（パネル） | home sheet 内の恒常表示ブロック | 平常時、memory が主役の時 |
| **drawer**（引き出し） | 画面端からスライドイン、展開で全項目閲覧可 | ユーザーが明示的に「何を知っているか見る」時 |
| **inline reference**（インライン参照） | 発話 surface 内の項目引用（1 行の埋め込み） | 発話が memory 項目に基づく時、参照元を示す |
| **badge**（バッジ） | home sheet 最小表示、件数のみの集約 | urgent layer 出現時、memory surface を縮退させた状態（§8.6） |

**禁止:**
- 4 形態以外の追加（modal / toast / popover 等は本書では採用しない）
- panel と drawer の同時表示（同一情報の二重表示禁止）
- inline reference の複数連続埋め込み（1 発話に 1 参照まで、認知負荷防止）

#### 8.2.2 アクセス導線

| 導線 | 説明 | 形態 |
|---|---|---|
| **恒常表示** | home sheet を開けば panel が目に入る | panel |
| **明示展開** | 「何を知っている？」系 tap で drawer | drawer |
| **発話経由** | 発話 surface に inline reference として項目引用 | inline reference |
| **縮退表示** | urgent 中は badge のみ（§8.6） | badge |

**原則:**
- 恒常表示の panel は**常設要素**（§3）扱い、state に依存しない
- drawer 展開は**ユーザー起動のみ**（CoAlter 側からの自動展開は禁止 — これをやると "覗かれた感" を生む）
- inline reference は CoAlter 側が発話時に必要最小限で付与

#### 8.2.3 更新タイミング

memory 項目が追加・更新・削除された時の UI 再描画タイミング。

| 変更種別 | 更新タイミング | 視覚キュー |
|---|---|---|
| **項目追加（明示共有）** | 即時 | 静かに fade-in（1 回） |
| **項目追加（CoAlter 推定）** | 遅延（次のユーザー操作時） | 「新しい読み」バッジ 1 個（確認後消える） |
| **項目更新** | 即時 | 項目のみフラッシュ（周囲は動かない） |
| **項目削除** | 即時 | fade-out（1 回） |
| **batch 更新（複数項目）** | 5 秒以内に batch、まとめて 1 回再描画 | 「N 件更新」チップ（tap で drawer 展開） |

**禁止:**
- 追加項目の**強い強調**（拡大アニメ・色転換・音など）— memory は静かに積む
- 連続フラッシュ（2 件以上が同時に光ると通知感が強くなる → batch 扱いに集約）
- 削除の完全サイレント（ユーザーが変化に気づけない → 最小限の視覚キュー必須）

### 8.3 共有メモリ項目の由来・確定度・可視性ラベル

共有メモリの最大リスクは **「いつの間にか関係の事実として固定される」** ことである。これを防ぐため、各項目に **3 軸のメタラベル** を UI 上で明示する。

#### 8.3.1 3 軸の独立定義

3 軸は相関するが **独立に決まる**。実装者は 3 軸を混同せず、それぞれ別レイヤーで表示する。

| 軸 | 値 | 意味 |
|---|---|---|
| **由来**（ソース種別） | `explicit_shared` | 両者または片側が CoAlter に明示的に共有した |
| | `inferred` | CoAlter が会話・文脈から推定した |
| | `transient_summary` | 直近の会話を一時要約したもの（短期保持） |
| **確定度**（信頼度） | `high` | 明示共有 + 双方承認、または長期確認済み |
| | `medium` | 一度は確認された、または複数回観測 |
| | `low` | 初回推定のみ、未検証 |
| **可視性**（スコープ） | `both_visible` | 両ユーザーに表示される |
| | `user_a_only` | 片側ユーザー A にのみ表示（B には内部のみ） |
| | `user_b_only` | 片側ユーザー B にのみ表示（A には内部のみ） |
| | `internal_only` | CoAlter 内部参照のみ、両者とも非表示（参照数のみカウント可） |

**相関例（独立性を保ちつつ典型的に共起するパターン）:**
- `explicit_shared` × `high` × `both_visible` — 両者が明示的に合意した事実
- `inferred` × `low` × `internal_only` — 初期推定、ユーザーに見せるには早い
- `transient_summary` × `medium` × `both_visible` — 直近会話のまとめ、時間経過で自動消滅

**禁止:**
- 3 軸のいずれかを **省略したまま項目を表示する**（3 軸がつかないラベルは memory 項目として扱わない）
- 由来 `inferred` を `explicit_shared` と同一の視覚で表示する（推定と事実の視覚混同禁止）
- 確定度 `low` を高確定度と同格に扱う表示（視覚的弱化が必須）

#### 8.3.2 視覚記号の型定義

ラベルの具体配色・アイコン・語尾はテンプレ doc / デザインシステム（§9）だが、**型**は本章で固定する。

| 軸 | 視覚型（主チャンネル） | 視覚型（補助チャンネル） | 位置 |
|---|---|---|---|
| **由来** | アイコン 1 個（形状で区別、色ではなく形） | — | 項目左端 |
| **確定度** | **線種・枠・補助記号**（`high` = 実線 / `medium` = 細線 or 軽い区切り / `low` = アウトライン or ghost + 補助ラベル） | 透明度・彩度（補助、主チャンネルを打ち消さない範囲で） | 項目全体 |
| **可視性** | ミニラベル（「あなただけに見えています」等の語彙型） | — | 項目右上 |

**型の論理的意図:**
- 由来を**形（アイコン）で区別** — 色盲配慮、かつ「種別」は色より形が直感的
- 確定度を**線種・枠で表現（主）+ 透明度で補強（副）** — 透明度単独依存を避ける:
  - 低透明度単独は「非活性 / disabled / 操作不可」に誤認されやすい
  - urgent / 縮退時に視認性が過度に低下する
  - 単一チャンネル依存はアクセシビリティ上の欠陥
  - → 線種・枠（主）で情報を担保、透明度（補助）で "静かさ" を追加する 2 チャンネル構成
- 可視性を**言葉で明示** — 片側限定は最も誤認が怖いため、アイコンではなく文言で明示

**禁止:**
- 確定度の**透明度単独表現**（主チャンネルを持たずに透明度だけで区別禁止）
- `low` 項目を通常項目から**線種でも透明度でも区別しない**表示（最低 1 チャンネルは必須）

#### 8.3.3 ラベル階層ルール

| 項目状態 | 由来表示 | 確定度表示 | 可視性表示 |
|---|---|---|---|
| `explicit_shared` × `high` × `both_visible` | ✓（必須） | 省略可（既定 = high） | 省略可（既定 = both） |
| `inferred`（全て） | ✓（必須） | ✓（必須） | 省略可（`both` 時のみ） |
| `transient_summary`（全て） | ✓（必須） | ✓（必須） | ✓（時間経過で消える旨含む） |
| いずれかが `_only` の可視性 | ✓（必須） | ✓（必須） | ✓（必須） |

**不変原則:**
- **由来は常に表示**（省略禁止）— これが本節の最大のガードレール
- 片側可視性（`_only`）は**必ず明示文言で示す** — ユーザーが「相手にも見えていると誤認」する事故を防ぐ

#### 8.3.4 有効組み合わせ制約

3 軸は**概念上は独立**だが、**UI 上の有効組み合わせには制約がある**。これがないと、`inferred × high × both_visible` のような組み合わせがそのまま「2 人の共有事実」として見えてしまい、CoAlter が関係の事実を勝手に捏造するリスクを生む。

**由来ごとの可視性昇格ルール:**

| 由来 | `both_visible` への到達経路 | 自動昇格の可否 |
|---|---|---|
| `explicit_shared` | 初期から `both_visible` 可 | ✓ 自動可 |
| `inferred` | **明示共有（§8.4 share 操作）または確認付き昇格（§7 Pattern C 経由の両者承認）を経由した場合のみ** | ✗ 自動昇格禁止 |
| `transient_summary` | 原則 `internal_only` または片側表示止まり。`both_visible` 昇格は**二段階確認**（明示共有 + 確定度 `medium` 以上への昇格）が必要 | ✗ 自動昇格禁止 |

**禁止される組み合わせ（自動的には到達できない状態）:**

| 組み合わせ | 禁止理由 |
|---|---|
| `inferred × high × both_visible` — 明示共有経由なし | 「CoAlter の推定」が「両者の共有事実」に見える |
| `transient_summary × high × both_visible` | 一時要約が永続的な共有事実に見える |
| `transient_summary × medium/high × both_visible` — 二段階確認経由なし | 同上の低強度版 |

**`both_visible` の意味固定（誤読防止）:**

`both_visible` = 「**両者に可視** である」という**可視性スコープ**の定義であり、**真偽・事実確定を意味しない**。UI 上も「共有事実」ではなく「両者可視」として表現する（具体文言はテンプレ doc）。

**論理的意図:**
- 3 軸独立を維持しつつ、**事実昇格には明示的な確認ステップが必要**というルールを UI 規則として固定
- `explicit_shared` 以外の由来は `both_visible` への自動遷移不可 → 推定・一時要約が勝手に共有事実化するのを構造的に防ぐ
- CoAlter の責務は「関係の読みを提示する」ことであり、「関係の事実を確定する」ことではない（v1.1 整合）

### 8.4 可視性操作 UI

共有 Alter において、**「共有する導線」と同じ重みで「引っ込める導線」が必要**である。片側が "見せたくない" と感じた時に後退できなければ、CoAlter は片側ユーザーにとって圧になる。

#### 8.4.1 4 つの操作

| 操作 | 定義 | UI 効果 |
|---|---|---|
| **表示**（閲覧） | 項目を通常通り見る（default） | 3 軸ラベルつきで表示 |
| **隠す**（mute） | 自分の画面からのみ非表示化（CoAlter 内部には残る） | 自分からは消える、相手には影響なし、CoAlter 参照は継続 |
| **明示共有**（share） | 片側可視性 → 両者可視性への昇格 | 可視性ラベルが `_only` → `both_visible` に変わる、相手にも現れる |
| **共有解除**（unshare） | 両者可視性 → 片側可視性または `internal_only` への降格 | 可視性ラベルが `both_visible` → `_only` / `internal_only` に変わる |

#### 8.4.1.1 操作の意味境界（誤読防止）

`隠す` が「削除」「忘却」と誤読されると、共有 Alter として最悪の事故が起きる（ユーザーは「隠したのに CoAlter が覚えている」と感じ、信頼が破壊される）。UI spec では**文面は扱わないが、操作の意味境界は固定する**。

| 操作 | 意味範囲 | 範囲外（誤読注意） |
|---|---|---|
| **隠す**（mute） | 自分の画面から外すだけ | ✗ CoAlter 内部から消えるわけではない<br>✗ 相手の画面には影響しない<br>✗ 発話参照・内部メモリは継続 |
| **共有解除**（unshare） | 相手側の可視性を落とす（`both_visible` → `_only` / `internal_only`） | ✗ 削除ではない<br>✗ 自分の画面からも消えるわけではない<br>✗ CoAlter 内部記憶の忘却ではない |
| **削除 / 忘却** | **本 §8 スコープ外**（v1.1 関係 OS レイヤー / 実装側の責務） | — |

**UI 規則:**
- 「隠す」ボタンの文言型 = **「自分の画面から外す」** の意味を示す語彙（具体語彙はテンプレ doc）
- 「共有解除」ボタンの文言型 = **「相手の可視範囲を下げる」** の意味を示す語彙（具体語彙はテンプレ doc）
- **「削除」「忘却」「消去」の語彙を mute / unshare 操作に使用禁止** — 範囲を越える誤読を生む
- 操作結果の UI 反応も、範囲に応じて変える（mute = 自分の画面のみ fade-out / unshare = 可視性ラベル変化、画面からは消えない）

**論理的意図:**
- CoAlter が共有 Alter である以上、"画面から外す" と "記憶から消す" を混同させない
- 真の削除 / 忘却が必要な場合（例: 誤って共有された機密事項）は、**§8 スコープ外**の別経路（実装側の削除導線、v1.1 の関係 OS レイヤー）で扱う
- UI spec の責務 = 意味境界を構造的に固定すること、文面そのものはテンプレ doc

**原則:**
- **隠す** ≠ **共有解除**: 隠すは自分の UI からの退出のみ、共有解除は可視性スコープの変更
- 隠した項目は **再表示 UI** で復活可能（永続削除ではない）
- 共有解除の方が影響が大きいため、**1 クッション確認** を挟む（即時実行禁止）

#### 8.4.2 後退導線の UI 配置

各 memory 項目には、**隠す / 共有解除** の後退導線が**常にアクセス可能**でなければならない。

| 配置 | 操作 | 発動 |
|---|---|---|
| 項目の右端 kebab アイコン | 全 4 操作メニュー | tap で展開、項目ごとに独立 |
| drawer 内の一括操作 | 選択 → 隠す / 解除 | 複数項目まとめて操作可 |
| 履歴からの revert | 直前の操作を戻す | 誤操作リカバリ |

**禁止:**
- 後退導線の**隠蔽**（設定画面の奥、複数階層下などへの配置禁止）
- 後退導線を**特定状態・モードで抑制**（どの state / mode でもアクセス可能）
- 後退操作に**警告文・心理的障壁文言**を挿入（「本当に解除しますか？" CoAlter が忘れます "」等は禁止 — §6.8 非判定性準拠）

#### 8.4.3 操作フィードバックのトーン

操作後の UI 反応は **§0.5 トーンカテゴリ** に準拠。

| 操作 | トーン | 理由 |
|---|---|---|
| 表示 | `neutral` | 観察のみ、反応不要 |
| 隠す | `calm` | 静かに引っ込む、強調しない |
| 明示共有 | `neutral` | 「広がった」を控えめに知らせる |
| 共有解除 | `retreat` | 短く締める、責めない |

**禁止:**
- `urgent` トーンの採用（どの操作も緊急ではない）
- 解除操作後の**追加挽留発話**（CoAlter から「本当にいいの？」は出さない）
- 明示共有時の**過度な祝祭感**（承認・完遂アニメの増量は避ける）

#### 8.4.4 片側可視性の範囲（§9 送り論点の明示）

以下は **§9 実装保留論点** に送る:
- 隠した項目が CoAlter 内部参照で発話にどこまで影響するか（完全除外 / 弱参照 / 通常参照のどれにするか）
- 共有解除後、相手側の履歴から **過去に見えていた事実** を消すか、保持するか
- 同項目を一方が `explicit_shared` に昇格、他方が `unshare` を試みた時の競合解決

### 8.5 緊急介入視覚層

#### 8.5.1 本節の責務

§4.4 で定義された**状態優先切替**（通常モード下でも S2 / S5 で介入価値閾値を超えた時、状態が優先される）の**視覚層の型**を定義する。

**トリガ条件・閾値・timeout 具体値** は §4.4 / §9 が正本。本節は視覚層のみ。

#### 8.5.2 視覚形態

| 形態 | 定義 | 使用 |
|---|---|---|
| **overlay banner** | home sheet 最上部に重ねる薄いバンド | §4.4 発動時の既定形態 |
| **dominant card** | 中央上部の展開カード（発話 surface 主役） | 発話 pattern が emergency 介入として発火した時 |
| **inline cue**（弱キュー） | home sheet 通常要素に重ねる小さな視覚変化（枠線の彩度） | 閾値接近時の予兆（まだ切替前） |

**禁止:**
- 全画面 modal（ユーザー操作を完全ブロックする UI は採用しない）
- 音声 alert / 強振動ハプティクス（§6.8 非判定性、CoAlter は叱責しない）
- 赤色 / 警告アイコン（§6.8 UI 禁止表現の継承）

#### 8.5.3 トーンと視覚言語

- トーン: **`urgent`**（§0.5 既定定義）
- 視覚言語: **彩度を高めず、透明度を下げる**（強く主張するが、威圧しない）
- アニメ: **1 回の fade-in + 最小 pulse**、継続 loop 禁止
- 文面: 本章では規定しない（テンプレ doc）

#### 8.5.4 解除条件

| 解除契機 | UI 反応 |
|---|---|
| 介入完了（発話成立 → 応答取得） | overlay banner fade-out、通常状態へ |
| ユーザー反応（dismiss tap） | fade-out + §6.8 非判定性維持（責めない） |
| timeout（具体値 §9） | 静かに fade-out、履歴にのみ残る |
| 上位優先切替（さらに強い urgent 発生） | 前の urgent layer は置換（重ね表示しない） |

**禁止:**
- dismiss 後の**追加挽留**（閉じた後に別の urgent を自動発火させない）
- timeout 後の**沈黙ペナルティ**（ユーザーに "無視した" とカウントしない、§6.8 継承）

#### 8.5.5 §6.8 非判定性の継承

urgent layer であっても、§6.8 UI 禁止表現 5 項目を継承する:
- 警告色（赤・オレンジの強彩度）禁止
- 叱責アイコン（✗ / ！の視覚強調）禁止
- 介入回数カウンタの可視化禁止
- cooldown カウントダウンの可視化禁止
- 「〇〇しましたね」式の追跡口調の文言禁止（文面はテンプレ doc だが、UI 側で文言欄の文字数上限を設けることで長い追跡文を構造的に排除）

### 8.6 memory surface と urgent layer の優先順位・同時出現規則

memory surface と urgent layer が**同時に前面化すると壊れる**。両者は直列ゲートで扱う。

#### 8.6.1 平常時・緊急時の優先順位

| 時間帯 | memory surface | urgent layer |
|---|---|---|
| **平常時**（S0-S3 / S4 / S6 / S8） | **主**（panel / drawer / inline reference） | **非表示**（存在を出さない） |
| **緊急時**（S2 / S5 で §4.4 発動時、urgent S7） | **降格または縮退**（背景化 or badge 化） | **主**（overlay banner / dominant card） |

**原則:**
- 平常時に urgent layer の弱キュー（inline cue）を恒常表示させない（ユーザーが "常に何か起きそう" と感じる UI は禁止）
- 緊急時に memory surface を平常時と同じサイズで残さない（注意の競合を作らない）

#### 8.6.2 降格 vs 縮退の使い分け

memory surface を緊急時にどう後退させるか、2 モードを使い分ける。

| モード | 定義 | 使用条件 |
|---|---|---|
| **降格**（demote） | 背景化（透明度低下、彩度低下、位置はそのまま） | urgent layer が overlay banner のみで、memory panel と重ならない場合 |
| **縮退**（compact） | サイズ縮小（panel → badge 化、件数のみ表示） | urgent layer が dominant card で memory panel 位置と競合する場合 |

**判断基準:**
- 空間的に競合するか → 競合: 縮退 / 非競合: 降格
- urgent の表示時間が短い（< 10 秒）→ 降格（復帰が早い）
- urgent の表示時間が長い（≥ 10 秒）→ 縮退（注意の省エネ）

**禁止:**
- **降格 + 縮退の同時適用**（透明度も下げて badge 化も、は過剰）
- urgent 解除後の memory surface の**強い復帰アニメ**（静かに元に戻る、祝祭感禁止）

#### 8.6.3 同時出現時の禁止組み合わせ

| 禁止組み合わせ | 理由 |
|---|---|
| memory drawer 展開中に urgent dominant card 発火 | 2 つの大型 UI の衝突 → drawer を先に縮退させてから urgent を出す |
| urgent 中に memory の batch 更新キュー表示 | 注意分散 → batch 更新は urgent 解除後に繰越 |
| 複数 urgent layer の重ね表示 | §8.5.4 上位優先切替で置換、重ねない |
| memory inline reference を urgent dominant card 内に埋め込む | 情報密度過多、urgent の主役性が薄れる |

#### 8.6.4 遷移アニメのトーン連続性

平常時 → 緊急時、緊急時 → 平常時の遷移で、トーンは連続的に変化する（断絶禁止）。

| 遷移 | トーン変化 | アニメ |
|---|---|---|
| 平常 → 緊急 | `neutral` → `urgent` | memory fade-to-background、urgent fade-in（同時、500ms 程度想定、具体値 §9） |
| 緊急 → 平常 | `urgent` → `neutral` | urgent fade-out、memory fade-to-foreground（同時） |
| 緊急継続中の urgent 入替 | `urgent` maintained | 前 urgent fade-out → 新 urgent fade-in（置換、重ねない） |

**禁止:**
- 瞬時切替（アニメなしの瞬間入替）
- トーンの飛び級（`neutral` → `urgent` 途中に `calm` を挟まない等の連続性違反は避ける、ただし逆方向は `urgent` → `retreat` → `neutral` も許容）

### 8.7 他章との境界と §9 送り論点

#### 8.7.1 他章との境界

## §9 実装保留論点

### 9.1 本章の責務と非責務

本章は **§0-§8 で送られた未決論点の保管庫** である。**意思決定本文ではない**。

**本章で扱う:**
- 未決論点の**保管**と**10 項目フォーマットでの記録**
- 各論点の**暫定 default / candidate の有無**の明示
- 各論点の **block level 分類**（実装前必須 / 後で可 / 実験待ち）
- **意思決定ルーティング**（primary owner として誰が推進するか: CEO / テンプレ doc 起草者 / デザインシステム / 実装 spike 担当、および consulted / approval 経路）

**本章で扱わない（他正本へ）:**
- **意思決定そのもの**（決めるのは primary owner: CEO / テンプレ doc 起草者 / デザインシステム / 実装 spike 担当）
- **正本の書き換え**（§0-§8 の記述を §9 で上書きしない）
- **具体文面・語彙・語尾** → テンプレ doc
- **配色 HEX・SVG パス・アニメ duration 具体値** → デザインシステム
- **検証データの取得と分析** → 実装 spike / データ計測

**§9 の位置づけ:**
- **未決 ≠ 破綻**: 論点が未決でも、UI spec の型と正本分離が固まっていれば実装は進められる
- **暫定 default**: 実装時の動作確保のための一時的な値。正本ではない。
- **block level**: 「CEO 判断まで固定禁止の論点」と「暫定 default で進行可の論点」を分類する属性

### 9.2 保留論点の記録フォーマット

各論点は**必ず**以下 10 項目で記録する。項目が埋まらない場合、その論点は §9 に記録する準備ができていない（他章でまず型定義する）。

| 項目 | 定義 |
|---|---|
| **論点名** | 論点を一意に識別する短い名前 |
| **所属テーマ** | §9.3 / §9.4 / §9.5 / §9.6 のいずれか 1 つ（重複禁止） |
| **現在の暫定扱い** | §0-§8 で既に書かれている扱い（型・原則・禁止事項など） |
| **未決要素の種類** | 数値 / 文言 / token / 可視性 / 実装検証待ち / 競合解決規則 など |
| **未決の理由** | なぜまだ決まっていないか（型だけ固定した / 検証が必要 / 別 doc 責務など） |
| **primary owner**（主決定者） | **1 人だけ**。その論点の意思決定を推進する主体（CEO / テンプレ doc 起草者 / デザインシステム / 実装 spike 担当のいずれか） |
| **consulted / approval**（協議・承認） | primary owner が推進する際に **協議する先** と、最終判断に **承認が必要な先**（例: 実装 spike が primary、CEO 承認必須） |
| **最終的な正本先** | 決定後にどの doc が正本になるか（UI spec / テンプレ doc / デザインシステム / 実装コード） |
| **block level** | `実装前必須` / `後で可` / `実験待ち` のいずれか |
| **暫定 default / candidate の有無** | **暫定 default あり**（値を記載、UI 挙動寄り）/ **candidate 候補集合**（A / B など、trust/privacy/visibility 影響値）/ **なし**（値未定） |

**属性としての「型」:**

「数値」「文言」「token」などは**章のテーマではなく、論点の属性**として `未決要素の種類` に記録する。これにより同じ型の論点（例: timeout）が複数テーマに跨る場合でも、各論点が 1 テーマに所属したまま識別できる。

**RACI 風の責任分解:**

`決定者` を単欄にすると、複数決定者が混在して責任が曖昧化する（§9.7 集計で 1 論点 > 1 決定者 になりやすい）。`primary owner` を**1 人に固定**し、`consulted / approval` を別欄に持たせることで、**主責任の明確化**と**レビュー・承認経路の保全**を両立する。

**block level の定義:**

| レベル | 意味 |
|---|---|
| **実装前必須** | CEO 判断 / 検証完了までは本番実装進行不可（ただし spike では仮値で動かせる） |
| **後で可** | 暫定 default で進行可能。正本確定は別タイミング。 |
| **実験待ち** | 検証データ取得後に判断。暫定 default または candidate 集合で動かし、データ取得を待つ。 |

**`block level` と `暫定 default / candidate` の整合性（原則 + 例外）:**

整合性は**原則**として扱い、**例外条件**を明示する。硬直的な矛盾判定は避ける。

| 組み合わせ | 原則 | 例外条件 |
|---|---|---|
| `実装前必須` × `暫定 default あり` | **本番固定値としては使えない** | **spike 限定の仮値**として置くのは可（本番注入禁止が守られる限り） |
| `実装前必須` × `候補値あり` | 本番固定不可だが、**CEO 判断時の候補集合**として記録可 | — |
| `実装前必須` × `なし` | 原則通り。CEO 判断まで固定禁止 | — |
| `後で可` × `暫定 default あり` | 原則通り | — |
| `後で可` × `暫定 default なし` | **許容**（動かない場合は block level 見直し） | **実装依存先**が既定 token 継承などで動く場合は**要注記** |
| `実験待ち` × `暫定 default あり` | 原則通り（比較基準として使用） | — |
| `実験待ち` × `暫定 default なし` | **非推奨**（比較起点が必要） | **候補集合（candidate A / B）で比較する場合は可** |

**暫定 default と candidate の使い分け:**

| 種別 | 使うべき論点 | 例 |
|---|---|---|
| **暫定 default**（単一仮値） | UI 挙動寄りの値、固定しても実害が限定的 | urgent timeout / animation duration / panel height |
| **candidate 候補集合**（A / B / ...） | trust / privacy / visibility / mode 昇格に効く値、恣意的 default が「真理っぽく見える」リスクが大きい | travel 判定閾値 / cooldown 期間 / 介入価値閾値 / transient_summary 減衰 |

**判断基準:**
- UI 挙動の「動き」を決める値 → 暫定 default
- ユーザー体験の「意味」を決める値 → candidate 集合で記録

**禁止:**
- 10 項目のいずれかを**省略して記録する**こと（省略する場合は §9 に記録せず、該当章で型から整備）
- 暫定 default / candidate を**正本として参照する**こと（暫定・候補は暫定・候補、正本は decided 後に他 doc）
- `primary owner` を**複数人に設定する**こと（主責任は 1 人、複数関与は consulted / approval で記録）
- trust / privacy / visibility 影響値を**単一 default で記録する**こと（candidate 集合で記録する）

### 10.3 decided 時の更新規則

§9 論点が **decided** された時、または §0-§8 の正本記述が修正された時の運用規則。

**§9 論点が decided された場合:**
1. §9 該当エントリを **削除せず**、`decided` マーク + 決定日 + 決定内容を追記（意思決定の履歴保全）
2. 該当章（§0-§8）に **正本記述** を追加。`candidate` 集合があった論点は、decided された値のみを正本に採用し、他の候補は §9 エントリ内に「不採用」として残す
3. §10.2 に **1 行** 追記（`v0.N: §X.Y decided — 〈内容要約〉`）
4. 版番号を **v0.N+1** に昇格

**§0-§8 の正本記述を修正する場合（型定義の改訂）:**
1. 該当章を修正し、§10.2 に **1 行** 追記（`v0.N: §X.Y 修正 — 〈修正要約〉`）
2. 版番号を **v0.N+1** に昇格
3. 破壊的変更（既存実装の作り直しが必要な変更）の場合は **v1.0** に昇格

**禁止:**
- §10 に新しい設計判断を書くこと（本章は履歴専用）
- 版番号昇格なしに §0-§9 の内容を書き換えること（履歴が追えなくなる）
- `decided` エントリを §9 から削除すること（履歴保全違反）

**本書独自版と v1.1 との整合:**
- v1.1 が改訂された場合、本書は §0.1 の「v1.1 を前提とする」規則に従い、**矛盾箇所を本書側から削除** する（本書側では正本を上書きしない）
- v1.1 改訂に伴う本書の修正も §10.2 に記録する（`v0.N: v1.1 改訂 vA.B に追随 — 〈修正箇所〉`）
